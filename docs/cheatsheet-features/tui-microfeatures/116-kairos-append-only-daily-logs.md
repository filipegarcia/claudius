# KAIROS Assistant-Mode Append-Only Daily Memory Logs

**Source:** Claude Code TUI — memory-system
**Status:** MISSING

## What it is
Long-lived "assistant" sessions get a different memory paradigm from coding sessions: instead of editing `MEMORY.md` live, the model appends timestamped bullets to a date-named log file and a separate nightly `/dream` skill distills the day's logs into topic files plus a refreshed `MEMORY.md`. A leak comment in `memdir/memdir.ts` spells out the rationale and the path shape: "Assistant sessions are effectively perpetual, so the agent writes memories append-only to a date-named log file rather than maintaining MEMORY.md as a live index. A separate nightly /dream skill distills logs into topic files + MEMORY.md. ... `const logPathPattern = join(memoryDir, 'logs', 'YYYY', 'MM', 'YYYY-MM-DD.md')`". The path is described as a *pattern* (`YYYY`/`MM`/`YYYY-MM-DD.md` literal placeholders) rather than a resolved string so the system prompt that carries it stays byte-identical across midnight rollover and the prompt-cache prefix survives the day boundary.

## Claudius today
Not surfaced in Claudius. The auto-memory subsystem is single-file/CRUD-only: `lib/server/auto-memory.ts` lays out `~/.claude/projects/<encoded>/memory/` as a flat directory of topic `*.md` files with frontmatter plus an idempotent `MEMORY.md` index (`autoMemoryDir` line 15, `writeMemoryFile` line 89, `appendMemoryIndex` line 300), and `app/api/memory/auto/route.ts` + `lib/client/useAutoMemory.ts` drive the `/memory` panel against that flat layout. There is no `logs/` subtree, no date-keyed file ever opened in write-append mode, and `isValidMemoryFilename` (`/^[\w.\-]+\.md$/`, line 64) actively *rejects* the `logs/YYYY/MM/YYYY-MM-DD.md` shape — slashes are non-`\w` and would be filtered out by the whitelist and the `assertWithin` barrier (`lib/server/safe-path.ts`) at line 108. Nothing in the repo distinguishes a "coding session" from an "assistant session" — `lib/server/session.ts`'s session model is one shape for every session — and there is no `/dream` slash command (`lib/shared/slash-commands.ts` has no entry, and `lib/server/session.ts` registers no `Stop`-hook distiller; the only auto-dream involvement is the `autoDreamEnabled` settings passthrough at `app/settings/page.tsx` line 496 that forwards to the SDK). The natural home for an analogue would be a new session-kind flag on the session row in `lib/server/session-manager.ts` plus an `appendAssistantLog(projectCwd, isoTs, bullet)` helper in `lib/server/auto-memory.ts` that opens `logs/${YYYY}/${MM}/${YYYY-MM-DD}.md` with `flag: "a"` (creating parents with `mkdir -p`), a relaxed whitelist for the `logs/` subtree, and a cron-style nightly distiller registered through the existing `lib/server/scheduler.ts` that runs a forked `query()` over yesterday's log and routes the output through `writeMemoryFile` / `patchMemoryFile`.

## Decision
MISSING. Per the leak in `memdir/memdir.ts`, Claude Code's assistant-mode paradigm is fundamentally different from its coding-mode paradigm — append-only date-keyed logs feeding a nightly `/dream` distillation, with the path expressed as a pattern so prompt-cache survives the midnight boundary — and Claudius has none of the four moving parts: no session-kind distinction, no `logs/YYYY/MM/YYYY-MM-DD.md` writer (the path shape is actively rejected by `isValidMemoryFilename` and `assertWithin` in `lib/server/auto-memory.ts`), no `/dream` slash command in `lib/shared/slash-commands.ts`, and no nightly distillation job hung off `lib/server/scheduler.ts` or a `Stop` hook in `lib/server/session.ts`. Backfilling is a substantial cross-cutting change (session model + storage layout + slash-command surface + scheduler entry); the cheaper near-term move is to treat this as TUI-only and revisit if/when Claudius grows a persistent "assistant" session kind distinct from per-task coding sessions.
