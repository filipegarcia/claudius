# Dream Task Background Subagent + Footer Pill

**Source:** Claude Code TUI — tasks-teammate
**Status:** MISSING

## What it is
When the auto-dream cooldown elapses, the TUI registers the forked memory-consolidation agent as a first-class `DreamTask` in the task registry rather than running it in the background unattributed. The minified launch site spells it out: `d("tengu_auto_dream_fired",{hours_since:Math.round(z),sessions_since:A.length,team_memory_enabled:J});let{taskRegistry:M}=q.toolUseContext,D=new AbortController,f=f$K(M,{sessionsReviewing:A.length,priorMtime:j,abortController:D})` (`tasks/DreamTask/DreamTask.ts`). The registered entry surfaces in the footer background-tasks pill with the label `dreaming`, appears in the Shift+Down dialog with phases `starting` → `updating` (flipped once the first Edit/Write lands), and carries a live turn count plus the set of files touched. Killing it from the dialog aborts the AbortController and rewinds the consolidation-lock mtime back to `priorMtime` so the next cooldown gate re-fires instead of treating the half-finished pass as a successful consolidation.

## Claudius today
Not surfaced in Claudius. The auto-dream feature is exposed only as a pure settings passthrough — `app/settings/page.tsx` line 496 renders the `autoDreamEnabled` `ToggleRow` ("Enable background memory consolidation (auto-dream). When set, overrides the server-side default."), persisted into `~/.claude/settings.json` for the SDK to read. The SDK does the actual dream pass inside its own process, and Claudius's task surface never learns about it: `components/panels/BackgroundTasksPanel.tsx` enumerates only the SDK-reported task kinds `local_bash` / `local_monitor` / `local_agent` / `local_workflow` (`PROCESS_TASK_TYPES` line 101, `taskIcon` lines 122-126), and a `local_dream` kind would have no host. There is no `tengu_auto_dream_fired` plumbing in `lib/server/session.ts`, no DreamTask row in `lib/client/task-status.ts`'s subagents partition, and no "consolidation-lock mtime" concept in `lib/server/auto-memory.ts` (the writer is per-file CRUD over `autoMemoryDir(projectCwd)` with no whole-directory consolidation pass). The natural seam would be a server-side dream forker in `lib/server/auto-memory.ts` that emits a synthetic `task_started` with `task_type: "local_dream"` through the SSE channel in `lib/server/session.ts`, with a matching icon and `dreaming` label in `BackgroundTasksPanel.tsx` and a phase/`priorMtime` payload carried alongside the existing `taskType` field at `lib/client/use-session.ts` line 2948.

## Decision
MISSING. The leak in `tasks/DreamTask/DreamTask.ts` describes an SDK-internal subsystem (forked consolidation agent, abort controller, mtime rollback) that Claudius does not run — its auto-dream involvement begins and ends at the `autoDreamEnabled` settings toggle in `app/settings/page.tsx`. Surfacing a "dreaming" pill plus phase / turn-count / files-touched / kill-with-rollback would require Claudius to either (a) host its own forked consolidation agent and lock file in `lib/server/auto-memory.ts` (broadcasting lifecycle events through `lib/server/session.ts`'s SSE channel into the `taskType` registry in `lib/client/use-session.ts`), or (b) wait for the SDK to expose `task_type: "local_dream"` task events through the same channel that already carries `local_bash` / `local_monitor` / `local_agent` / `local_workflow`. Until either path exists, treat this as a TUI-only contract: the leak documents the SDK's bookkeeping, not a Claudius-side gap to backfill.
