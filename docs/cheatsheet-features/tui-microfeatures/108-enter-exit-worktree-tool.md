# EnterWorktree / ExitWorktree Tools

**Source:** Claude Code TUI — tools
**Status:** PARTIAL

## What it is
A pair of discrete agent tools that, only when the user explicitly says "worktree", create an isolated checkout under `.claude/worktrees/<name>/` (delegating to `WorktreeCreate` / `WorktreeRemove` hooks for non-git repos) and switch the session's cwd into it mid-conversation. `tools/EnterWorktreeTool/prompt.ts` spells the contract out verbatim: "Creates an isolated worktree (via git or configured hooks) and switches the session into it ... Use ExitWorktree to leave the worktree mid-session (keep or remove). On session exit, if still in the worktree, the user will be prompted to keep or remove it". A 16-hit binary-grep across the CLI confirms `EnterWorktree`/`ExitWorktree` are first-class tool identifiers, not just hook names.

## Claudius today
Claudius treats the tools as side-effect *signals*, not user-facing surfaces. `lib/server/session.ts` (line 96-105) documents that "harness-level `EnterWorktree` moves the session into a git worktree WITHOUT firing the SDK's `CwdChanged` (or, in some builds, `WorktreeCreate`) hook" — so the PreToolUse hook at line 1561-1586 sniffs any Edit/Write/MultiEdit/NotebookEdit `file_path` for a `.claude/worktrees/<name>/` prefix via `worktreeRootFromPath` and calls `broadcastCwd(wt)` as the fallback signal. That feeds `components/chat/StatusLine.tsx` (line 138, 191-198), which renders the `data-testid="status-line-worktree"` badge using `worktreeBadge` from `lib/client/worktree.ts`. The comment block at line 1552-1557 also warns NOT to register a passive `WorktreeCreate`/`WorktreeRemove` hook because the SDK treats those as creation *extension points* — a no-op handler makes `EnterWorktree` fail outright with "hook succeeded but returned no worktree path". The `WorktreeCreate`/`WorktreeRemove` hook names are catalogued in `lib/shared/hook-events.ts` (line 29-30, 94-95) for the hook-config UI. The `/worktrees` slash command (`lib/shared/slash-commands.ts` line 161) plus `components/overlays/WorktreesOverlay.tsx` and `app/api/worktrees/route.ts` cover *listing and opening* existing worktrees in a fresh session — but there is no equivalent of the `ExitWorktree` mid-session move, no in-chat affordance to invoke the tool, and no keep-or-remove prompt anywhere in `lib/server/session.ts` or the overlays for when a session that entered a worktree is closing.

## Decision
PARTIAL. The status-line badge already reflects when `EnterWorktree` has fired (via the PreToolUse path heuristic in `lib/server/session.ts`), and `lib/shared/hook-events.ts` lists `WorktreeCreate`/`WorktreeRemove` so users can wire custom creation/teardown hooks for non-git roots. The visible gaps versus `tools/EnterWorktreeTool/prompt.ts` are: (1) no UI to invoke `ExitWorktree` mid-conversation — a chat-level action that closes the worktree branch and broadcasts `cwd_changed` back to the session root — and (2) no keep-or-remove confirmation when a session that fired `EnterWorktree` is closed (natural location: a session-end gate in `lib/server/session.ts` plus a small confirm dialog reusing the `WorktreesOverlay` styling). Both are low-priority for browser sessions (the user typically closes the tab, not the agent), but a chat action and a one-time prompt would close the parity gap.
