# Bridge Inbound Web Attachments Resolved to @path Refs

**Source:** Claude Code TUI — bridge-remote
**Status:** MISSING

## What it is
The cloud bridge accepts an `assistant`-side composer payload that carries `file_attachments` alongside the text, then materializes each one onto the local agent's filesystem before forwarding the turn. A leaked comment in `bridge/inboundAttachments.ts` spells out the contract: "Web composer uploads via cookie-authed /api/{org}/upload, sends file_uuid alongside the message. Here we fetch each via GET /api/oauth/files/{uuid}/content (oauth-authed, same store), write to ~/.claude/uploads/{sessionId}/, and return @path refs to prepend. Claude's Read tool takes it from there." The staged filenames follow `~/.claude/uploads/<sessionId>/<uuid-prefix>-<safe-name>` and the message is rewritten so quoted `@"<path>"` refs sit ahead of the user's text — letting the model's existing Read tool pick the file up without any bespoke `image` content block.

## Claudius today
Not surfaced in Claudius. The send path takes attachments inline as base64 `image` content blocks instead: `components/chat/PromptInput.tsx` (state `images: AttachedImage[]` line 195, `MAX_IMAGE_BYTES` line 92, the `[Image #N]` token stripper at line 104) collects pasted/dropped files, posts them with the prompt, and `lib/server/session.ts` line 2064 receives `images?: Array<{ data: string; mediaType: string; ordinal?: number }>` and weaves them into the SDK message as base64 blocks. The persistence side (`lib/server/asset-ingest.ts` `recordSendUses` / `backfillProject`) writes a content-addressed copy under the project DB for the Files page, but nothing stages anything under `~/.claude/uploads/<sessionId>/` or prepends `@"<path>"` refs to the prompt. The natural home for a TUI-style bridge would be a new `lib/server/inbound-attachments.ts` that, when the SDK actually exposes a file_attachments inbox (Claudius is a local agent, not a cloud bridge — so the cookie-authed upload half doesn't apply), saves to `~/.claude/uploads/` and rewrites the prompt before it reaches `session.ts`.

## Decision
MISSING. The leaked `bridge/inboundAttachments.ts` describes the cloud-bridge half of a two-process pipeline — cookie-authed `/api/{org}/upload` on the web side, oauth-authed `/api/oauth/files/{uuid}/content` fetch on the agent side — and Claudius runs the agent locally, so the upload half is irrelevant and the download half has nothing to download. The interesting carry-over is the *prompt rewrite*: prepending `@"<path>"` refs and letting the Read tool ingest attachments instead of stuffing base64 image blocks into the SDK message. That could replace `lib/server/session.ts` line 2064's image-block weaving with a `~/.claude/uploads/<sessionId>/` staging dir plus `@"<path>"` injection, which would also unblock non-image attachments (PDFs, CSVs) that the current `MAX_IMAGE_BYTES` / `AttachedImage` pipeline rejects. Worth filing as a follow-up; not blocking.
