# MEMORY.md Entrypoint Truncation with Warning Banner

**Source:** Claude Code TUI — memory-system
**Status:** MISSING

## What it is
On load, Claude Code caps the `MEMORY.md` index at both 200 lines and 25KB; when either cap fires it truncates at the last newline before the byte cap and appends an in-context warning so the model knows it's seeing a partial index. The leak in `memdir/memdir.ts` literally builds the banner — `` `\n\n> WARNING: ${ENTRYPOINT_NAME} is ${reason}. Only part of it was loaded. Keep index entries to one line under ~200 chars; move detail into topic files.` `` — and the binary carries the two `reason` strings (`index entries are too long` for the byte cap vs `N lines (limit: 200)` for the line cap) so the model can tell which limit it tripped and self-correct (keep each `- [name](file.md) — description` line under ~150 chars; offload anything longer into the topic file).

## Claudius today
Not surfaced in Claudius. The full `MEMORY.md` CRUD pipeline ships — `lib/server/auto-memory.ts` writes the index via `appendMemoryIndex` (line 300), `replaceMemoryIndexLine` (line 249), and `removeMemoryIndexLine` (line 278); `app/api/memory/auto/route.ts` exposes list/read/create/patch/delete; `lib/client/useAutoMemory.ts` drives the `/memory` panel — but nothing enforces or even *measures* the 200-line / 25KB caps. `appendMemoryIndex` happily appends an unbounded number of `- [name](file.md) — description` lines, and the read path in `auto-memory.ts:51` (`readMemoryFile`) returns the raw `MEMORY.md` body with no size accounting. There is no count badge in the panel and no write-time warning when a new line would push the index past either threshold. The closest sibling is the existing `memory-update` system-reminder in `lib/server/system-reminders.ts` line 28 — that nudges the model after an edit, but it's about freshness, not size. The natural home for a Claudius-side analogue is twofold: a server-side count in `app/api/memory/auto/route.ts` that surfaces `{ lines, bytes }` for `MEMORY.md`, and a panel-side warning chip in the `AutoMemorySection` that flips on once `lines >= 200` or `bytes >= 25_000` (with a hint pointing each over-long index entry into a topic file via the existing `EditMemoryForm`).

## Decision
MISSING. Per `memdir/memdir.ts`, Claude Code silently truncates `MEMORY.md` at the agent-runtime layer and tells the model *why* via an appended `> WARNING:` block — Claudius has the full CRUD surface for `MEMORY.md` (`lib/server/auto-memory.ts`, `app/api/memory/auto/route.ts`, `lib/client/useAutoMemory.ts`) but does not measure or warn on the 200-line / 25KB caps, so a power-user can grow the index past the point where the SDK will quietly drop tail entries at load. A follow-up could add a cheap `{ lines, bytes }` summary to the `GET /api/memory/auto` payload and render a "MEMORY.md is over the load cap" chip in `AutoMemorySection` with the same two reason strings the SDK uses, so the user sees the same signal the model would have seen.
