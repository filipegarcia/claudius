# Monitor-vs-Bash Distinct Completion Summaries

**Source:** Claude Code TUI — tasks-teammate
**Status:** MISSING

## What it is
The `local_monitor` kind (a streaming background script subscribed to via the Monitor tool, not a Bash command) gets its own terminal-notification phrasing distinct from the generic bash-collapse line — `Monitor X stream ended`, `Monitor X script failed (exit N)`, `Monitor X stopped` — and is dispatched with priority `next` instead of `later` so the exit lands before the next agent turn rather than batching with other deferred reminders. The leak in `tasks/LocalShellTask/LocalShellTask.tsx` (21 binary hits) spells out the contract: `"Monitor "`, `" stream ended"`, `" script failed"`, `" (exit "`, `" stopped"`, `" completed"`, `task-notification`, `next`.

## Claudius today
Not surfaced in Claudius. The kind-aware partitioning exists — `PROCESS_TASK_TYPES = new Set(["local_bash", "local_monitor"])` (`components/panels/BackgroundTasksPanel.tsx:101`), and `iconForType` switches `local_monitor` to the `Activity` pulse vs `local_bash`'s `Terminal` (lines 123-124) — but the `task_notification` reducer at `lib/client/use-session.ts:3004` reads only `status: "completed" | "failed" | "stopped"` plus `summary`, and writes whatever the SDK supplies straight into `TaskInfo`. There's no per-`taskType` rephrase branch (monitor vs bash both flow through the same `base.summary = t.summary` path at line 3031), and there's no notion of `next`-vs-`later` priority on the notification edge — the row simply settles in place. The row's terminal-status pill (`TERMINAL_STATUSES = new Set(["completed", "failed", "stopped", "killed"])` at `BackgroundTasksPanel.tsx:104`) is shared across both kinds. Exit codes aren't tracked on `TaskInfo` at all (`lib/client/types.ts`); the SDK's `usage` payload reconciled at line 3032 carries `total_tokens` / `tool_uses` / `duration_ms`, never an `exit_code`.

## Decision
MISSING. The TUI's monitor-specific copy and `next`-priority dispatch live entirely in `tasks/LocalShellTask/LocalShellTask.tsx` and have no Claudius analog — the SSE `task_notification` Claudius consumes is already collapsed by the SDK before it reaches the client. A lightweight follow-up that captures the *intent* without an SDK fork: at the reducer (`use-session.ts:3004`) branch on the resolved task's `taskType` and, when `local_monitor`, prefix the summary with the typed verbs (`stream ended` / `script failed (exit N)` / `stopped`) using a `summaryForProcessTask` helper alongside `lib/client/task-status.ts`; the `(exit N)` clause stays best-effort until the SDK exposes `exit_code` on the `task_notification` payload. The `next`-vs-`later` priority concept maps onto Claudius's notification feed only loosely — there's no batched-reminder queue to bypass — so it can stay out of scope until a deferred-reminder lane lands (see `lib/server/system-reminders.ts`).
