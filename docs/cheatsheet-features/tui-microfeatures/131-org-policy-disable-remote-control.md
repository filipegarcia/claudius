# Org-Managed disableRemoteControl Policy + Auto-Start Setting

**Source:** Claude Code TUI — bridge-remote
**Status:** NOT_APPLICABLE

## What it is
A managed-settings key `disableRemoteControl` that kills the entire Remote Control surface — the `/remote-control` slash command, the `claude.ai/code` pairing flow, the `--remote-control` / `--rc` CLI flags, the auto-start-on-launch path, and the in-session toggle — and replaces every entry point with the error `Remote Control is disabled by your organization's policy`. `bridge/bridgeEnabled.ts` (binary-confirmed, two hits) ships the contract verbatim: `"Disable Remote Control (claude.ai/code, ` + "`claude remote-control`" + `, `--remote-control`/`--rc`, auto-start, and the in-session toggle). Typically set in managed settings."` A sibling user-scoped setting, "Start Remote Control bridge automatically each session", auto-runs the bridge whenever the CLI starts; both surface in the settings UI but the managed policy always wins.

## Claudius today
Not surfaced in Claudius. The natural locus would be the same `external`-tagged platform commands in `lib/shared/slash-commands.ts` — `remote-control` (line 145, aliases `rc`, "Enable remote control from claude.ai."), `teleport` (line 144), `remote-env` (line 146), `mobile` (line 141), `web-setup` (line 137) — which Claudius already classifies as awareness-only because the bridge endpoint they delegate to is a claude.ai-hosted service, not a locally-launched SDK feature. The Settings catalog in `app/settings/page.tsx` likewise excludes the managed/enterprise key family by design (`allowManaged*Only`, `strictKnownMarketplaces`, `strictPluginOnlyCustomization`, …) — see the catalog comment around line 660 and the precedent decision at `docs/cheatsheet-features/config-settings/06-policy-fragments.md`. Claudius reads/writes only user/project/local `settings.json` via `lib/server/settings.ts`, never the system-deployed managed policy, so there is no in-app code path for the `Remote Control is disabled by your organization's policy` block to gate.

## Decision
Not applicable. The `disableRemoteControl` managed key in `bridge/bridgeEnabled.ts` is a claude.ai-hosted bridge concern — it lives in the same `--remote` / `remote-control` / `teleport` family that Claudius already classifies as `external` in `lib/shared/slash-commands.ts` and that prior notes (`docs/cheatsheet-features/tui-microfeatures/85-bridge-environment-resume.md`, `docs/cheatsheet-features/tui-microfeatures/86-bridge-mobile-push-notifications.md`, `docs/cheatsheet-features/tui-microfeatures/87-bridge-multi-session-capacity-meter.md`, `docs/cheatsheet-features/tui-microfeatures/88-bridge-outbound-only-mirror-mode.md`, `docs/cheatsheet-features/tui-microfeatures/89-bridge-qr-code-toggle.md`, `docs/cheatsheet-features/tui-microfeatures/91-bridge-spawn-mode-toggle.md`) have all deferred. The auto-start setting is the inverse of the same surface — no bridge to auto-start, no toggle to gate. Combined with the deliberate omission of the managed-policy key family from `app/settings/page.tsx` (see `docs/cheatsheet-features/config-settings/06-policy-fragments.md`), there is no Claudius surface to lock down. Deferred — would need the hosted bridge transport, not a UI gap.
