# Notify After Idle Timeout (Bell + Desktop Notification)

**Source:** Claude Code TUI — hooks-ux
**Status:** PARTIAL

## What it is
After ~6s with no user interaction past a turn boundary, the TUI fires a desktop notification (and a terminal-bell / notif-channel-specific bell) so a long-running turn that finishes while the user is tabbed away still gets a ping; a fresh interaction inside the window cancels the ding. The leak spells out the constant: `export const DEFAULT_INTERACTION_THRESHOLD_MS = 6000` in `hooks/useNotifyAfterTimeout.ts`. The contract is: turn ends → wait for the threshold → if last-interaction-at is still older than the window, ring; otherwise skip.

## Claudius today
The same shape lives in `lib/server/notification-bus.ts`: the bus stamps `lastUserInputAt` whenever `Session.sendInput()` calls `notificationBus.markUserInput(this.id)` (`session.ts` line 2097), then on the next `sdk` `result` event `mapEventToKind` (line 689) emits a `session_idle` kind titled `"Claude finished a turn"` (line 711), which `useNotifications.notify()` (`lib/client/useNotifications.ts` line 217) dispatches as an OS notification through either the Electron bridge or the browser `Notification` API. What's *missing* is the interaction-threshold debounce itself: Claudius shipped an `IDLE_NOTIFY_MIN_MS` gate that mirrored `DEFAULT_INTERACTION_THRESHOLD_MS` almost exactly ("notify only if `now - last >= 5s`") and then deliberately removed it (see the comment block at `lib/server/notification-bus.ts` line 696: *"We used to gate this on `now - last >= IDLE_NOTIFY_MIN_MS` (a 5s window after the user typed) ... a quick turn on a backgrounded session almost always landed inside the window, was suppressed, and never surfaced anywhere"*). The replacement is a client-side visibility gate in `useNotifications.notify()` (line 231: `visibleRef.current && sameSession && !isActionableKind(row.kind)`), which suppresses the OS toast only when the user is actively foregrounded on the originating tab — orthogonal to the "did the user touch anything in the last 6s" question the TUI is asking. There is also no terminal-bell channel because Claudius is browser/Electron, not a TTY.

## Decision
PARTIAL. Claudius ships the outcome the TUI feature is trying to deliver — a `session_idle` OS notification when a backgrounded turn finishes, suppressed when the user is already watching the session — but the specific *interaction-recency* mechanism cited in `hooks/useNotifyAfterTimeout.ts` (`DEFAULT_INTERACTION_THRESHOLD_MS = 6000`) is absent by choice: the equivalent `IDLE_NOTIFY_MIN_MS` window in `lib/server/notification-bus.ts` was deleted because too many real turns finished inside the window and got silently dropped. If we want strict parity with the leak, the natural shape is a fresh per-session "last interaction" stamp (composer keypress / scroll / pointer move, not just `sendInput`) consulted at notify-time in `useNotifications.notify()`, so a recently-active user skips the ding without re-introducing the server-side suppression bug.
