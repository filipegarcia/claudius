# Typed Memory Taxonomy (user/feedback/project/reference)

**Source:** Claude Code TUI — memory-system
**Status:** PARTIAL

## What it is
A leak in `memdir/memoryTypes.ts` pins down a closed four-type taxonomy that the agent sees as structured XML in its system prompt: `ow8=["user","feedback","project","reference"]` paired with `J45={user:"the user's role, expertise, or working preferences",feedback:"a correction or confirmation of how you should approach work. Confirmations ('yes, good call') are quieter than corrections — watch for them",project:"ongoing work, deadlines, or decisions not derivable from code or git history",reference:"where to find information in an external system (issue tracker, dashboard, channel)"}` (5 binary hits). Each type ships its own `<when_to_save>` / `<how_to_use>` / `<body_structure>` block — the latter literally instructs `Lead with the rule itself, then a **Why:** line (the reason the user gave — often a past incident or strong preference) and a **How to apply:** line (when/where this guidance kicks in). Knowing *why* lets you judge edge cases instead of blindly following the rule.` The taxonomy is closed *by exclusion* too: derivable info (code patterns, git history, `CLAUDE.md` content) is explicitly NOT a memory.

## Claudius today
The four-type enum is the source of truth in `lib/server/auto-memory.ts` line 70 (`export type MemoryType = "user" | "feedback" | "project" | "reference"`) and is enforced on write/patch at lines 99 and 181; `useAutoMemory` re-exports the same union (`lib/client/useAutoMemory.ts` lines 64 and 89) and the /memory panel offers the four `<option>` values verbatim in both forms (`app/[workspaceId]/memory/page.tsx` lines 442-444 and 619-621). The panel even echoes the leaked body convention as in-form hint copy — `TYPE_HINTS` (lines 369-374) renders `feedback: "Guidance about how to approach work. Use **Why:** + **How to apply:** lines."` and the same `Why:`/`How to apply:` couplet on `project`. What is NOT wired: the actual system-prompt injection. There is no `<when_to_save>` / `<how_to_use>` / `<body_structure>` block emitted from `lib/server/session.ts` or `lib/server/system-reminders.ts` — a grep across the repo for those tag names returns nothing — so the agent never sees the per-type guidance, the "Confirmations ('yes, good call') are quieter than corrections" nudge on `feedback`, or the *exclusion* clause that derivable info (code patterns, git history, `CLAUDE.md` content) is not a memory. The `TYPE_HINTS.project` string in particular drops the leak's "not derivable from code or git history" qualifier, so the human picking the type sees a looser definition than Claude Code's agent does.

## Decision
PARTIAL. Per `memdir/memoryTypes.ts`, the storage enum and the human-facing form copy already ship in Claudius (`lib/server/auto-memory.ts:70`, `app/[workspaceId]/memory/page.tsx:359-374`), but the load-bearing half of the feature — per-type `<when_to_save>` / `<how_to_use>` / `<body_structure>` XML stitched into the agent's system prompt, plus the explicit "code patterns, git history, `CLAUDE.md` content are NOT memories" exclusion — is not surfaced. A follow-up could mint a `memoryTaxonomyBlock(type)` helper next to `memoryUpdateReminderBody` in `lib/server/auto-memory.ts` returning the three nested tags, then inject the relevant block at the auto-memory loader site (alongside the in-progress `memoryAgeReminderBody` envisioned in `124-memory-staleness-reminder.md`) so the per-type contract reaches the agent rather than only the human filling the form.
