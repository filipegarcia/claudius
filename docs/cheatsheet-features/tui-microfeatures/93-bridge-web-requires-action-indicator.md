# Bridge Web 'requires_action' Indicator via reportState

**Source:** Claude Code TUI — bridge-remote
**Status:** NOT_APPLICABLE

## What it is
When the local CLI is paired with a claude.ai Remote Control bridge, the bridge calls `CCRClient.reportState` to push per-session lifecycle states up to the hosted backend, and `requires_action` is the state that flips the per-session badge on claude.ai-web to a "waiting for input" indicator while a permission prompt or other user-input gate is pending. The contract is spelled out in the comment on the call site in `bridge/replBridgeTransport.ts`: "PUT /worker state (v2 only; v1 is a no-op). `requires_action` tells the backend a permission prompt is pending — claude.ai shows the \"waiting for input\" indicator." The same channel publishes `reportMetadata` (session-level facts the web feed needs) and per-event `reportDelivery` (`processing` → `processed`) so the remote feed shows live ACK status as each event lands.

## Claudius today
Not surfaced in Claudius. The natural locus would be the same `external`-tagged platform commands in `lib/shared/slash-commands.ts` — `remote-control` (line 145), `teleport` (line 144), `remote-env` (line 146), `web-setup` (line 137) — which Claudius already classifies as awareness-only because they delegate to claude.ai's hosted bridge rather than the locally-launched SDK. Claudius has a perfectly good *local* "needs attention" pipeline — `components/chat/PermissionPrompt.tsx` raises the inline allow/deny card, the same event flips an unread count via `lib/client/useFaviconBadge.ts` (line 8: "Drives the dynamic favicon + document.title for the total unread count") and a desktop toast through `lib/client/useNotifications.ts` / `electron/ipc/notifications.ts` — but nothing publishes that "waiting for input" state *outward* to a remote claude.ai-web feed, because there is no `CCRClient`-style transport: Claudius's renderer talks to the local `SessionManager` (`lib/server/session.ts`) over same-origin SSE and nothing else listens.

## Decision
NOT_APPLICABLE. The `reportState("requires_action")` call and its companion `reportMetadata` / `reportDelivery` traffic in `bridge/replBridgeTransport.ts` are a claude.ai-hosted bridge concern — they live in the same `--remote` / `remote-control` / `teleport` family Claudius already classifies as `external` (see prior notes at `docs/cheatsheet-features/cli-flags/21-remote-flag.md`, `docs/cheatsheet-features/workflows-tips/26-web-session.md`, and the sibling bridge entries `docs/cheatsheet-features/tui-microfeatures/85-bridge-environment-resume.md`, `docs/cheatsheet-features/tui-microfeatures/88-bridge-outbound-only-mirror-mode.md`, `docs/cheatsheet-features/tui-microfeatures/91-bridge-spawn-mode-toggle.md`). Claudius is a local-first wrapper over the Agent SDK with no hosted /worker endpoint to PUT state into, so there is no Claudius surface for this indicator. Deferred — would need the hosted bridge transport, not a UI gap.
