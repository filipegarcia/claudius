# Org-Policy 'enabledPlugins: false' Force-Disables Plugins

**Source:** Claude Code TUI — plugins-output-styles
**Status:** NOT_APPLICABLE

## What it is
Managed-settings (the system-deployed `policySettings` layer) can set `enabledPlugins[id] = false` to force-disable a specific plugin or whole marketplace, and the CLI then refuses installs and enables at any user/project/local scope. `utils/plugins/pluginPolicy.ts` (binary-confirmed, four hits) spells out the contract: `"Policy-blocked plugins cannot be installed or enabled by the user at any scope. Used as the single source of truth for policy blocking across the install chokepoint, enable op, and UI filters."` The chokepoint logs `Skipping policy-blocked marketplace ...` during marketplace reconcile, `version-unsatisfied but policy-blocked; not force-including` when a forced-version manifest collides with the policy, and the `/plugin` picker filters the blocked ids out of the install/enable lists entirely.

## Claudius today
Not surfaced in Claudius, by the same precedent as every other managed-settings key. `lib/server/settings.ts` (line 6) only knows the three user-writable scopes — `export type SettingsScope = "user" | "project" | "local"` — and `pathFor`/`readSettings`/`writeSettings` (lines 55-95) never touch the system `policySettings` layer where `enabledPlugins: false` would land. `lib/server/plugins.ts` reads each scope's `enabledPlugins` map and writes user toggles through `setEnabled` (line 57), but the read-side at line 42 (`const ep = settings.enabledPlugins`) treats every entry as a user preference; it does not distinguish a policy-set `false` from an absent key, and `listAvailable` (line 109) walks the marketplaces cache without consulting any blocklist. The `/plugins` UI in `app/plugins/page.tsx` (line 74 `Object.entries(s.enabledPlugins)`, line 178 `enabledInScope={Boolean(active?.enabledPlugins?.[row.id])}`) and the read-only "Plugins" section in `app/settings/page.tsx` (line 579 `JSON.stringify(draft.enabledPlugins ?? {}, null, 2)`) both reflect just the user-scoped settings file, so there is no `Skipping policy-blocked marketplace` log path, no `version-unsatisfied but policy-blocked` decision, and no filter in front of the install picker. The Settings catalog at `app/settings/page.tsx` already excludes the managed-key family by design (`allowManaged*Only`, `strictKnownMarketplaces`, `strictPluginOnlyCustomization`, …) — see the catalog comment around line 660 and the precedent at `docs/cheatsheet-features/config-settings/06-policy-fragments.md`.

## Decision
NOT_APPLICABLE. The contract in `utils/plugins/pluginPolicy.ts` is a managed-settings concern that lives one tier above what Claudius reads — Claudius only touches user/project/local `settings.json` via `lib/server/settings.ts`, never the system-deployed `policySettings` directory where an admin would write `enabledPlugins[id] = false`. The same precedent that defers `docs/cheatsheet-features/config-settings/06-policy-fragments.md` and `docs/cheatsheet-features/tui-microfeatures/131-org-policy-disable-remote-control.md` applies here: a normal user running Claudius cannot edit managed policy from the browser, and the install/enable chokepoints in `lib/server/plugins.ts` plus the `/plugins` picker have no managed-layer to gate against. Deferred — would need Claudius to start reading the managed-settings location (and a UX decision about how to surface "blocked by your organization" in the picker), which is out of scope for the existing user-scope-only settings surface.
