Install

One curl, then you're chatting.

The script clones the repo into ~/claudius, runs bun install, and prints the next steps. Re-running it pulls the latest commit on main and refreshes deps. Requires git; Bun auto-installs if missing. 

curl -fsSL https://claudius.network/install | bash
macOS · Linux · View the script · Read the source
Options

Customize the install

Pass flags after -- to change the destination, branch, or auto-start the dev server. 

curl -fsSL https://claudius.network/install \
  | bash -s -- \
      --prefix=$HOME/code/claudius \
      --branch=main \
      --start
  • --prefix=DIR — install destination (default ~/claudius)
  • --branch=BRANCH — git branch (default main)
  • --no-install — clone only, skip bun install
  • --start — run bun run dev after install
Prefer to read it first?

Audit, then run

Piping curl into bash is fine when you trust the source — but if you'd rather see what runs, save it locally first. 

curl -fsSL https://claudius.network/install \
  -o setup.sh
less setup.sh
bash setup.sh --start    # runs bun run dev after

Or skip the script and clone it yourself: 

git clone https://github.com/filipegarcia/claudius.git
cd claudius
bun install
bun run dev      # http://localhost:3000

Keep it running

A daemon for the production build.

bun run dev is great while you're hacking. When you want Claudius up between shell sessions, bin/claudiusd wraps the production build with a small PID-file daemon — no PM2, no systemd, no Docker. Both modes auto-build the first time.

Foreground

Logs in your terminal

Best for "I want to see what it's doing." Streams logs to stdout, Ctrl-C stops it. 

make run    # http://127.0.0.1:3000
Detached

Background, with log file

Survives the parent shell exiting. Logs append to .claudius/logs/claudius.log. 

make up         # start
make logs       # tail -F the log
make status     # pid + last 10 lines
make down       # SIGTERM (SIGKILL after 10s)
make restart    # down + up
Defaults Binds to 127.0.0.1:3000 — Claudius drives the SDK with filesystem and tool access, so LAN exposure is opt-in. Override with env vars: 
PORT=8080 make up                  # different port, still loopback
HOST=0.0.0.0 PORT=8080 make up     # reachable from your LAN (careful)
Sandboxed Want the agent's Bash, edits, dev servers, and MCP calls boxed in? Run Claudius inside a Docker container, devcontainer, or VM — the SDK spawns claude as a local subprocess, so keep Claudius and Claude in the same box, mount your workspace, and expose just :3000 to your browser.

What's inside

Every Claude Code surface, on the web.

Claudius mirrors the TUI's session model and adds the views that only make sense with a real screen: cost dashboards, file trees, git operations, multi-workspace switching.

01 · chat

Full agent chat

Streaming responses, tool calls, permissions UX, todos banner, AskUserQuestion forms, interrupt — the lot.

02 · workspaces

Multiple workspaces

Switch project context without losing sessions, schedules, or MCP config. Each workspace has its own SQLite store.

03 · sub-agents

Sub-agents

Define specialized agents with their own system prompts, models, and tool sets. Launch them from chat.

04 · mcp + plugins

MCP & plugins

First-class Model Context Protocol support and a plugin system for extending the agent's tool surface.

05 · hooks

Lifecycle hooks

Run commands at any agent lifecycle event — pre-tool, post-tool, session start, stop. Configurable per workspace.

06 · schedule

Scheduled agents

Cron-driven agent runs. Wake up at 9am, check the deploy, file a ticket — without you having to be there.

07 · cost

Cost & usage

Per-session, per-model, per-day spend. See where the tokens go before the bill arrives.

08 · git + files

Git & file tree

In-app file browser and git operations: stage, diff, commit, with AI-drafted commit messages.

09 · local-first

Local-first storage

Sessions, schedules, assets — all persisted on your machine. No cloud lock-in, no SaaS account required.

One more thing…

Build your own features, right in the app.

The agent you’re chatting with has the whole Claudius codebase on the other end. Want a new page, a keyboard shortcut, a widget in the activity rail? Just ask. The code is local, the agent has the tools — Claudius edits itself.

you · in chat
claudius · ~90s later
opened preview · 4 files · ready to review

Your turn. Start your own customization — hit New customization, ask the agent for whatever you want, preview, publish.

Customize

Edit Claudius from inside Claudius.

Hit New customization and Claudius mirrors its own source into a private folder, opens a workspace pointing there, and auto-spawns a preview on a separate port. Chat with the agent like normal — it edits the mirror, not the running app. When you’re happy, hit Publish. If anything breaks, hit Revert — or run make claudius-revert from a terminal if the UI itself is broken.

Nine demos, one afternoon

Ship a SQL console. Or ship Clippy.

A DataGrip-style SQL console, a Jupyter notebook runner, a live DAG dashboard, a Docker monitor, a synthwave repaint, a DOOM-themed Cost page, a Minecraft parkour clip next to the thinking block, the Konami code, a paperclip mascot — all built by asking the agent. Each lives in its own mirror, toggle on, toggle off, independent of the others.

Snapshot-backed. Every publish takes byte-level snapshots of the files it’s about to overwrite, indexed in a plain JSON manifest under ~/.claude/.claudius/customizations/. If a bad publish breaks the UI, run make claudius-revert from a terminal — the CLI has zero runtime dependencies, so it works even when Claudius itself won’t boot.
