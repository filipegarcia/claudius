## Summary

`@anthropic-ai/claude-agent-sdk@0.3.208` bundles Claude Code CLI 2.1.208.
Automatic changelog extraction failed because the `v0.3.208` tag doesn't
exist yet on `anthropics/claude-agent-sdk-typescript` (only `v0.3.207` and
earlier are tagged) — so this analysis was reconstructed by hand: diffing
the packaged `.d.ts` files (`sdk.d.ts`, `sdk-tools.d.ts`, `bridge.d.ts`,
`agentSdkTypes.d.ts`, `browser-sdk.d.ts`) between the 0.3.207 npm tarball
and the installed 0.3.208 tree for the *shape* of the change, then pulling
`anthropics/claude-code`'s own `CHANGELOG.md` `## 2.1.208` section for
*intent* (45 bullets — CLI/TUI fixes, perf, and two new settings). Net
result: **zero Claudius code changes required.** Every public-API-surface
diff (5 hunks total) is additive/widening or doc-only, and every
call-site-shaped changelog bullet (context-window display, MCP-server
status classification, checkpoint/file-history pruning, background-agent
listing, `apiKeyHelper`/Bedrock error surfacing) resolves to Claudius
consuming a `Query`/SDK method whose *output* just got more correct
upstream, not a shape Claudius has to adapt to. The one genuinely new
settings key (`vimInsertModeRemaps`) is deliberately left out of the
settings catalog for the same reason its parent `editorMode` already is
(see below). Headline risk to flag: none in the diff itself — the one
thing worth a reviewer's eye is a **pre-existing, unrelated** finding
surfaced during the audit: `lib/server/claude-code-changelog.ts` is dead
code (zero importers) that's ~39 versions stale, see Risks/follow-ups.

- The full gate suite was run end-to-end on `sdk-update/0.3.208` to
  confirm the SDK bump + cc-parity change don't regress anything:
  - `bun run lint` — clean.
  - `bun run test` (vitest) — 1484/1484 passed.
  - `bun run build` — succeeds.
  - `bun run test:e2e` (Playwright chromium) — 156/157 non-skipped
    specs passed across two full runs; the same 2 specs failed both
    times: `notifications-multi-tab.spec.ts:199` and `:595`
    ("backgrounded session badge ... after a switched-away
    notification" / "... via the SSE-driven refresh"). Both fail on
    an `expect.poll` timeout (15s) waiting for `/api/sessions/open-tabs`
    to reflect a `page.reload()`, in a test file whose own comments
    already call out this exact risk ("can be slow when the dev server
    is under load from the rest of the suite" — see `f07b879`).
    **Confirmed pre-existing and unrelated to this PR:** both specs
    pass cleanly when run in isolation (no load), and a discriminator
    run of the full suite against unmodified `main` (via a scratch
    worktree, same machine) failed the *same two* specs plus two more
    (`sticky-user-message.spec.ts`, `workspace-route-memory.spec.ts`)
    that don't fail on this branch at all — i.e. `main` is flakier
    under full-suite load than this PR's branch, not the other way
    around. Neither failing spec touches `plan_usage`/`CostOverlay`/
    open-tabs logic this PR changed. Root cause is dev-server resource
    contention over a ~15min/165-test single-worker run on this
    machine, not a behavioral regression from the SDK bump.

## Risks / follow-ups

- **2 pre-existing e2e flakes in `notifications-multi-tab.spec.ts`
  (load-timeout, not this PR's fault) — see "Tests" above for the full
  writeup and the main-branch discriminator run that confirms it.**
  Worth a maintainer follow-up to either raise the poll timeout or
  split that file so it doesn't run this late in a long single-worker
  suite, but out of scope for an SDK-bump PR to fix unprompted.
- **Unrelated pre-existing finding, surfaced during this audit:**
  `lib/server/claude-code-changelog.ts` bundles a static snapshot of the
  upstream Claude Code `CHANGELOG.md` (`CLAUDE_CODE_CHANGELOG_SNAPSHOT`,
  `readBundledSdkVersions()`) with a comment implying it feeds
  `/release-notes`. It doesn't: `app/release-notes/page.tsx` links out to
  `docs.claude.com` instead and never imports this file — grepping the
  whole repo for `CLAUDE_CODE_CHANGELOG_SNAPSHOT` / `readBundledSdkVersions`
  turns up zero call sites outside the file itself. It's also ~39 CLI
  versions stale (last entry `2.1.169`, captured `2026-06-09`; current is
  `2.1.208`). Left untouched here — updating it wouldn't have any
  user-visible effect since nothing renders it, and fully backfilling 39
  versions of changelog text is out of scope for a single-version SDK
  bump. Worth a follow-up decision: wire it into `/release-notes` for
  real, or delete it as dead code.
- **`vimInsertModeRemaps` / `editorMode` are a judgment call, not a
  certainty.** The reasoning (Claudius's composer isn't the CLI readline,
  so vim-mode settings can't affect it) is sound for the *composer*, but
  if the CLI ever grows a code path where `editorMode`/`vimInsertModeRemaps`
  affect something SDK-driven beyond terminal keybinding (e.g. an
  interactive-elicitation text field), that assumption should be
  re-checked. Flagging so a reviewer can sanity-check the call rather
  than silently trusting it.
- The GitHub tag for `v0.3.208` doesn't exist yet on
  `anthropics/claude-agent-sdk-typescript` as of this run — if a real
  upstream changelog / release notes entry appears later and contradicts
  anything in "SDK changelog highlights" above (reconstructed from a raw
  `.d.ts` diff plus the bundled CLI's own changelog), prefer the upstream
  text over this file's reconstruction.
