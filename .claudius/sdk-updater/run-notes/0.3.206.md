## Summary

`@anthropic-ai/claude-agent-sdk` 0.3.205 → 0.3.206 is a one-line
release: "Added `command_lifecycle` frames to stream-json and SDK
sessions, reporting each uuid-stamped message's terminal state
(`queued`/`started`/`completed`/`cancelled`/`discarded`); zero-API
results no longer report stale `duration_api_ms`." Two independent
pieces: (1) a new wire-level message type, `command_lifecycle`, that
reports per-uuid terminal state for async-queued commands; (2) a bug
fix so a turn result with zero API calls stops carrying over a stale
`duration_api_ms` value.

`command_lifecycle` is **not** in the npm package's public surface —
confirmed via `npm pack`+`diff` of the 0.3.205 vs 0.3.206 `sdk.d.ts` /
`sdk-tools.d.ts` (zero mention) and a grep of the compiled `sdk.mjs`
wrapper (zero mention). It IS real at runtime, though: the actual
spawned Claude Code CLI binary
(`node_modules/@anthropic-ai/claude-agent-sdk-darwin-arm64/claude`)
contains the literal string and the emit sites
(`yield{type:"command_lifecycle",uuid:...,state:...}`), and the SDK
wrapper's ingestion loop only excludes a short list of known
control-plane types (`control_response`, `control_request`,
`control_cancel_request`, `keep_alive`, `transcript_mirror`, a couple
of `system` subtypes) before forwarding everything else through to
`query()`'s async iterator — so `command_lifecycle` frames do reach
`lib/server/session.ts`'s `for await (const message of this.query)`
loop today, just untyped. (First analysis pass here checked only the
JS wrapper and concluded the frame was unreachable; an adversarial
verify pass caught that this was checking the wrong artifact and
traced the real emit site in the native binary — see below.)

Claudius's own queue UI (`QueueIndicator`, `queued_messages` SQLite
table) is server-managed and doesn't depend on the SDK's internal
async-message queue — Claudius only ever hands `query()` one message
at a time via `sendInput()` when idle (`flushQueueIfIdle`), so there
is no user-facing gap `command_lifecycle` closes today. What we
*did* ship: since these frames now flow through the generic
`broadcast()` funnel and would sit in the 1000-event replay buffer
doing nothing useful, we excluded them from the replay buffer (same
treatment as the existing `queue:updated`/`holder_changed` snapshot
exclusions) so they don't crowd out real transcript history on
sessions that queue heavily. They're still forwarded to live
subscribers — cheap, and forward-compatible if a future release
gives them a UI. The `duration_api_ms` half needs no code change:
Claudius's cost-overlay accumulator already does
`(prev?.durationApiMs ?? 0) + (r.duration_api_ms ?? 0)`, so it
already treats an absent/zero value correctly — the upstream fix is
a pure accuracy improvement to the "API time" stat with nothing for
us to do.

**Headline risk to flag:** the buffer-exclusion fix is a small,
low-risk, well-tested change, but `command_lifecycle` itself remains
formally untyped by the SDK — if Anthropic later gives it a real
`SDKMessage` union member with a different shape, our exclusion check
(`(event.message as { type?: string })?.type === "command_lifecycle"`)
should keep matching, but the field names inside `state`/`uuid`
aren't type-checked anywhere and could drift silently. Also flagging:
this branch's top of `main` already carries unrelated `cc-parity`
pipeline commits (`docs(cc-parity): plan for 2.1.206`,
`site/releases.html`) — see Risks/follow-ups.

## SDK changelog highlights

- **`command_lifecycle` frames added to stream-json and SDK
  sessions** — uuid-stamped terminal-state reporting
  (`queued`/`started`/`completed`/`cancelled`/`discarded`) for
  async-queued commands. `[shipped — partial]`. Not exposed in the
  npm package's public `.d.ts` (`sdk.d.ts`/`sdk-tools.d.ts` diff
  between 0.3.205 and 0.3.206 has zero mention; `sdk.mjs` has zero
  string-literal mentions either), but verified present at runtime in
  the native CLI binary
  (`node_modules/@anthropic-ai/claude-agent-sdk-darwin-arm64/claude`,
  13 literal occurrences incl. the `yield{type:"command_lifecycle",...}`
  emit sites) and confirmed the SDK wrapper's message-ingestion loop
  passes it through to `query()`'s async iterator untouched (only a
  short known-control-type exclusion list is filtered before the
  catch-all forward). Shipped: excluded these frames from the
  session's 1000-event replay buffer in `lib/server/session.ts`
  (`shouldBufferEvent()`) since Claudius has no consumer for them yet
  and letting them sit in the FIFO buffer would evict real transcript
  history on queue-heavy sessions — see "Code changes". NOT shipped:
  no client-side UI consumes the frame's queued/started/completed/
  cancelled/discarded state. Audited whether one should exist:
  Claudius's queue UI (`QueueIndicator.tsx`) and its state
  (`lib/server/queued-messages-db.ts`, `Session.queueCache`) are
  entirely server-managed — Claudius holds queued prompts itself and
  only ever calls `sendInput()` on one message at a time when idle
  (`flushQueueIfIdle`), so it doesn't depend on or observe the SDK's
  *own* internal async-message queue in normal operation. There is no
  current UX gap this frame closes, and building a UI on top of an
  informally-typed, not-yet-`.d.ts`-documented field risks churn if
  the SDK later reshapes it. Logged as a backlog idea under
  Risks/follow-ups rather than spec'd out now.
- **Zero-API results no longer report stale `duration_api_ms`** —
  bug fix: a turn result with no API calls (e.g. a purely local/
  slash-command turn, or a turn cancelled before any API call)
  previously could carry over a stale nonzero `duration_api_ms` from
  a prior turn; now it correctly reports 0/absent. `[skipped —
  self-resolving upstream fix, no client change needed]`. Claudius's
  only consumer (`lib/client/use-session.ts:3217`,
  `durationApiMs: (prev?.durationApiMs ?? 0) + (r.duration_api_ms ?? 0)`,
  displayed as "API time" in `components/overlays/CostOverlay.tsx:83`)
  already treats an absent value as 0 via `?? 0` — the bug (if it ever
  manifested for us) was purely upstream inflating the reported value
  we then summed; now that it reports correctly, our existing
  accumulation is automatically more accurate with zero code changes.

## Code changes

- `lib/server/session.ts` — extracted the replay-buffer inclusion
  check out of `broadcast()` into a new exported pure function,
  `shouldBufferEvent(event: ServerEvent): boolean`, and added a third
  exclusion case (alongside the pre-existing `queue:updated` /
  `holder_changed` snapshot exclusions) for SDK `command_lifecycle`
  frames — new on the wire in 0.3.206, not consumed anywhere in
  Claudius, and worth keeping out of the 1000-event FIFO replay
  buffer so they don't evict real transcript history on sessions that
  queue commands heavily. Live subscribers still receive the frame
  (the exclusion only affects what a reload/reattach replays) —
  matches the existing "safe passthrough" behavior for any
  unrecognized SDK message type.
- `components/chat/UserMessage.tsx` — removed a duplicate peer-badge
  render block. Unrelated to the 0.3.206 changelog, but the e2e gate
  (`tests/e2e/sdk-update-0.3.205-peer-message.spec.ts`) failed with a
  Playwright strict-mode violation: two `data-testid="user-message-
  peer-badge"` elements matched, from two separate JSX blocks both
  gated on `message.peer` truthiness — one added by the 0.3.205 SDK
  update (the `peerLabel`/`name`-aware version, lines ~104-112) that
  never removed the older, now-redundant block (lines ~130-138) it
  was meant to replace. Confirmed pre-existing (reproduces identically
  on this branch with the 0.3.206 `session.ts` change reverted) and
  unrelated to this bump's diff; fixed per the standing instruction
  that the gate doesn't distinguish "yours" from "theirs." Removed the
  older block, kept the name-aware one.
- `tests/unit/should-buffer-event.test.ts` — new vitest spec pinning
  down `shouldBufferEvent()`: excludes `queue:updated`,
  `holder_changed`, and `command_lifecycle` (across all five reported
  states); keeps ordinary `sdk` messages (`assistant`/`user`/`result`/
  `system`), a defensive case for `sdk` messages with no
  `message.type`, and non-`sdk` event types other than the two
  existing exclusions.

## Risks / follow-ups

- **First analysis pass got this wrong before an adversarial verify
  pass caught it** — worth a reviewer's attention as a process note,
  not a code risk: checking only the JS SDK wrapper (`sdk.mjs`) for
  "command_lifecycle" and finding nothing led to an initial (wrong)
  conclusion that the frame was unreachable from Claudius. It's
  actually emitted by the native CLI binary the wrapper spawns and
  passed through untouched. If a future SDK-update run does a
  similar ".d.ts diff only" analysis without checking the actual
  spawned binary, it could reach the same wrong conclusion for a
  different changelog item.
- **`command_lifecycle` remains informally typed.** We match on
  `message.type === "command_lifecycle"` with no schema validation
  on `uuid`/`state`. If the SDK later formalizes this as a typed
  `SDKMessage` union member, worth revisiting `shouldBufferEvent()`
  to import the real type instead of the loose cast.
- **Product opportunity, not a ship-blocker for this release:**
  Claudius currently infers its own queue-drain timing via idle
  polling (`Session.flushQueueIfIdle`) rather than consuming
  authoritative per-uuid lifecycle state from the SDK. Now that
  `command_lifecycle` frames are confirmed flowing at runtime, a
  future release could use them for scenarios where Claudius doesn't
  track state itself — e.g. cron-triggered or auto-resume-injected
  async messages the SDK enqueues internally, which the interrupt
  receipt's `still_queued` doc comment calls out as sometimes
  containing "uuids the client never sent." Worth a look if that gap
  ever causes a user-visible bug.
- **Branch contamination from the sibling `cc-parity` pipeline.**
  `git log main..HEAD` on `sdk-update/0.3.206` includes
