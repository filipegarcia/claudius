## Summary

SDK 0.3.176 ships two behavioral bug-fixes with no public-API changes: (1) turn
`result` messages that were previously dropped when multiple turns completed
simultaneously during a background-agent or workflow run are now delivered
reliably, and (2) background-agent, remote-agent, and MCP-task state is now
properly restored by the SDK when resuming a session. Neither fix requires
Claudius code changes. Fix 1 directly unblocks a latent stuck-session / stuck
queue-drain scenario in Claudius — `turnInFlight` would never flip back to
`false` if the SDK dropped a `result`, leaving the session status permanently
`"running"`. Fix 2 is handled independently by Claudius's own SQLite-backed task
persistence (`persistTask` → `saveSessionTask` → `sendTaskSnapshot`), so no
change is needed there either. Headline risk: none, this is a pure bug-fix
release. The `enable1mContext` / `context-1m` beta gating and fable-advisor
incompatibility logic in `session.ts` are unchanged and remain correct.

## SDK changelog highlights

- `[skipped — SDK-internal fix, no Claudius workaround needed]` Fixed turn
  `result` messages being dropped when multiple turns complete while a
  background agent or workflow is running. Claudius's `consume()` loop in
  `session.ts` relies on every `result` message to flip `turnInFlight = false`,
  drain the queued-message queue, and fire turn-end reminders. There is no
  client-side workaround for dropped events — Claudius directly benefits from
  this fix. The existing synthetic `result` at line ~6496 is an abort/crash
  backstop (SDK iterator drains without any `result`), not a compensating path
  for this scenario; it remains correct and unchanged.

- `[skipped — Claudius workaround already in place, SDK fix is additive]` Fixed
  background agent, remote agent, and MCP task state not being restored when
  resuming a session via the SDK. Claudius has its own task-persistence layer
  (`captureTaskState` → `persistTask` → `saveSessionTask` in `session.ts`;
  `listSessionTasks` → `sendTaskSnapshot` on subscribe) that is entirely
  independent of the SDK's internal restore. Task state for completed and
  in-flight subagents is flushed to SQLite incrementally and replayed from the
  DB on every reconnect or server restart. The SDK fix makes the SDK's own state
  more consistent but does not affect Claudius's display path.

## Risks / follow-ups

- Fix 1 (dropped `result` messages) surfaces a latent production risk: before
  0.3.176, any session with a concurrent background-agent or workflow turn could
  get stuck with `turnInFlight = true` indefinitely, never draining the queued-
  message queue and never firing turn-end reminders. This was silently swallowed
  in production. The fix arrives automatically with the SDK bump; no Claudius
  code change needed, but worth monitoring if any previously stuck sessions
  surface unexpected behavior on next resume.
- Fix 2 (task state on resume) makes the SDK's internal state consistent with
  what Claudius already shows via its SQLite-backed snapshot path. The two paths
  are now aligned; no known conflict, but the SQLite path remains load-bearing
  for the `task_snapshot` SSE events that are Claudius-specific and never
  written to the SDK's JSONL.
