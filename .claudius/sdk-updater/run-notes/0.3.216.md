## Summary

0.3.216 is a small, three-item release: a new `skippedLinks` count on
`rewindFiles()` results, a `tool_result_meta` sidecar the CLI is
described as adding to user messages, and two new latency-correlation
fields (`user_message_uuid`, `request_sent_wall_ms`) on the success
result message. Only the first has an actual public TypeScript surface
in the installed 0.3.216 package (`RewindFilesResult.skippedLinks` in
`sdk.d.ts`) and a Claudius call site (`RewindFilesButton.tsx`, backing
the "Restore files" affordance on user messages) — that one is
`[shipped]`: the button now surfaces a warning when the safety guard
refused to restore some files. The `tool_result_meta` item has no
corresponding field anywhere in the installed package (verified by
grep across `sdk.d.ts`, `sdk-tools.d.ts`, `bridge.d.ts`,
`agentSdkTypes.d.ts`, and the `sdk.mjs`/`browser-sdk.js` bundles — zero
hits), so it's `[skipped]`. The latency-correlation fields ride through
opaquely on the existing `SDKMessage`-typed "sdk" broadcast passthrough
with no Claudius call site that needs them, so that's `[type-only]`.
Headline risk: the `skippedLinks` warning copy is my best-effort
guess at what's actionable for a user to do next ("restore it by hand")
since the SDK doesn't tell us *which* files were skipped — only a
count — so a reviewer should sanity-check that framing.

## SDK changelog highlights

- **Added optional `skippedLinks` count to `rewindFiles` responses for
  paths the rewind safety guards refused to restore or delete** —
  `[shipped]`. `RewindFilesResult.skippedLinks` is declared in
  `sdk.d.ts` (only populated on a real, non-dryRun rewind). Claudius
  has a concrete call site: `Query.rewindFiles()` is proxied through
  `Session.rewindFiles()` (`lib/server/session.ts`) and
  `POST /api/sessions/[id]/rewind`, and consumed by
  `RewindFilesButton.tsx`'s "Restore files" affordance on user
  messages. Wired the field through the client `RewindResult` type and
  added a warning banner in the post-restore dialog when
  `skippedLinks > 0`.
- **Added `tool_result_meta` sidecar to user messages
  (`non_execution_kind`, `user_feedback`)** — `[skipped — no public SDK
  export in the installed 0.3.216 package]`. Grepped `sdk.d.ts`,
  `sdk-tools.d.ts`, `bridge.d.ts`, `agentSdkTypes.d.ts`, `sdk.mjs`, and
  `browser-sdk.js` for `tool_result_meta`, `non_execution_kind`, and
  `user_feedback` — zero occurrences anywhere. Read the full declared
  shape of `SDKUserMessage` and `SDKUserMessageReplay` in `sdk.d.ts`
  (both list every field verbatim: `type`, `message`,
  `parent_tool_use_id`, `isSynthetic`, `tool_use_result`, `priority`,
  `origin`, `shouldQuery`, `timestamp`, `uuid`, `session_id`,
  `subagent_type`, `task_description`) — no sidecar field present. This
  reads as a CLI/wire-protocol-side change ahead of its SDK type
  surface; there is nothing for Claudius to type against or consume
  yet. Not a model-identity item, so the higher skip bar in the run
  prompt doesn't apply — this is a plain "doesn't exist in the package
  we're on" skip.
- **Added optional `user_message_uuid` and `request_sent_wall_ms`
  fields to the success result message for cross-host request-latency
  correlation** — `[type-only]`. Both fields are declared on
  `SDKResultSuccess` in `sdk.d.ts`. Claudius never redeclares that
  shape locally — the "sdk" broadcast event
  (`lib/shared/events.ts` `ServerEvent`) types its payload as the SDK's
  own `SDKMessage` union and `lib/server/session.ts` forwards the raw
  `result` message object verbatim (`this.broadcast({ type: "sdk",
  message })`, session.ts:6763) — so the new fields already flow
  end-to-end through the existing type and existing passthrough with
  zero code change. There's no cross-host correlation concept in
  Claudius (single browser session per SDK query) for a UI or server
  feature to hang off of, so no behaviour was added.

## Code changes

- `components/chat/RewindFilesButton.tsx` — added `skippedLinks` to
  the local `RewindResult` type; `RewindDone` now renders an amber
  warning banner (`data-testid="rewind-skipped-links-warning"`) when
  `skippedLinks > 0`, explaining that a symlink/hard-link/non-regular
  file was refused by the safety guard. Also added
  `data-testid="restore-files-button"` to the trigger button so the
  new e2e spec (and future ones) can target it without a CSS selector.
- `tests/unit/rewind-route.test.ts` — new case asserting the route
  forwards `skippedLinks` through unchanged on a real (non-dryRun)
  rewind, pinning the passthrough contract at the API boundary.
- `tests/e2e/sdk-update-0.3.216-rewind-skipped-links.spec.ts` — new
  Playwright spec (see "## New UI surfaces").

No changes were needed in `lib/server/session.ts` or
`app/api/sessions/[id]/rewind/route.ts` — both already forward the
SDK's `RewindFilesResult` object verbatim (`unknown`-typed on the
server side, spread into the JSON response), so the new field rides
through without any server-side code change.

## New UI surfaces

- **"Restore files" post-rewind skipped-links warning** — when a real
  (non-dryRun) file restore reports `skippedLinks > 0`, the "Files
  restored" confirmation dialog now shows an amber warning banner
  under the restored-file-count line, explaining that some tracked
  files were left untouched by the safety guard. Screenshot:
  `docs/sdk-updates/0.3.216/rewind-skipped-links.png`. Captured by
  `tests/e2e/sdk-update-0.3.216-rewind-skipped-links.spec.ts`. Context:
  a live chat session on `/`, hovering a user message bubble to reveal
  the "Restore files" affordance, clicking through the preview dialog
  to confirm, with the mocked API response reporting 2 skipped files —
  the shot shows the confirmation dialog open over the chat transcript
  (tab strip / message list visible behind the modal overlay).

## Risks / follow-ups

- The `skippedLinks` warning copy tells the user a count but not
  *which* files were skipped (the SDK only returns a count, not a
  list) — worth revisiting if a future SDK release adds the file list,
  at which point the banner should name the paths instead of just the
  count.
- `tool_result_meta` (`non_execution_kind` / `user_feedback`) is
  described in the 0.3.216 changelog but genuinely absent from every
  `.d.ts` and bundled `.mjs`/`.js` file in the installed package —
  flagging in case this is a doc/changelog-ahead-of-release timing
  issue and the field lands in a patch release; worth re-checking next
  SDK bump before assuming it's still absent.
- `user_message_uuid` / `request_sent_wall_ms` pass through opaquely
  today. If Claudius ever adds cross-host session correlation (e.g. a
  companion mobile client), these are the fields to key off of — no
  action needed now.
