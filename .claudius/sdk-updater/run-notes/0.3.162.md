## Summary

0.3.162 brings two changes: (1) refusal events now carry `stop_reason: "refusal"` (and `stop_details`) on the assistant message and in session transcripts, enabling SDK consumers to detect content refusals programmatically without text-matching; (2) native-build SDK sessions now default to fast embedded Bash `find`/`grep` for search rather than always registering the dedicated Grep/Glob tools, matching the behaviour of the interactive CLI. Neither change required code modifications in Claudius. The `BetaStopReason` type in `@anthropic-ai/sdk` already included `'refusal'` before this release, and `SDKResultMessage.stop_reason` is typed as `string | null`, so the new runtime value is absorbed by existing types without any update. The embedded-search default is transparent to Claudius because `session.ts` does not special-case Grep/Glob in its `canUseTool` handler, and `lib/server/updater/apply.ts` already explicitly lists `Grep` and `Glob` in `allowedTools` — exactly the pattern the changelog prescribes to preserve dedicated tools. No breaking changes, no gate failures.

No Claudius code currently inspects `stop_reason` values, so no behavioural change is needed. Noted as a follow-up in Risks: Claudius could surface a special "refusal" indicator in the chat UI to take advantage of this.

This is transparent: `canUseTool` does not special-case Grep or Glob, so all search paths (embedded or dedicated) go through the same permission flow. `lib/server/updater/apply.ts` already explicitly lists `Grep` and `Glob` in its `allowedTools` array, which is exactly the mechanism the changelog prescribes to preserve dedicated tools when needed. No code change is required.

## Risks / follow-ups

- **Refusal UI indicator (potential enhancement):** Now that `stop_reason: "refusal"` is populated at runtime, Claudius could detect refusals in `use-session.ts` and render a distinct indicator (e.g. a yellow "Refusal" badge on the result message) instead of treating them identically to a normal `end_turn`. This was intentionally deferred — no existing code breaks without it — but would be a meaningful UX improvement for users who encounter content-policy limits.
- **`stop_details` field:** The changelog mentions `stop_details` as a companion to `stop_reason: "refusal"`, but this field does not appear in any public typed surface in `sdk.d.ts` or `@anthropic-ai/sdk`. If a future SDK version exposes it in the typed interface, Claudius should evaluate whether to display it alongside the refusal indicator.
- **Embedded search and hooks:** Users who have configured `PreToolUse`/`PostToolUse` hooks targeting the `"Grep"` or `"Glob"` tool names will find those hooks no longer fire for the main Claudius session after this bump (the agent will call Bash instead). This is a user-configuration concern, not a Claudius code issue, but worth noting in release communications.
