## Summary

SDK 0.3.211 → 0.3.214 is a small release: one CLI-argv fix, one
subagent-model-visibility addition (0.3.212), a Claude Code parity
marker with no itemized API surface (0.3.213), and six discrete
additions in 0.3.214 (stricter `set_permission_mode` validation +
`'manual'` alias, a `scheduled-trigger` subkind on task-notification
origin, `'max'` landing in `applyFlagSettings`'s TS type, an
`aborted: true` flag on interrupt-truncated assistant messages, two
new `tool_progress` fields for subagent rate-limit-retry visibility,
and plugin manifest `version` on `system/init`/`reload_plugins`). We
shipped the three genuinely user-visible additions end-to-end:
truncated-message "Interrupted" badges, a per-subagent "retrying"
indicator in the Tools rail, and plugin version numbers in the
Plugins page. We also did a small type-only cleanup of the
`effortLevel` cast in `session.ts` now that the SDK's own type
accepts `'max'` directly. Everything else is inert metadata Claudius
doesn't currently branch on (task-notification subkind, SessionStart
`fork` source) or SDK-internal/undocumented (the CLI argv fix, the
Agent-tool resolved-model field, which has no exported TS shape to
bind against). Headline risk: the new `aborted` / `subagentRetry`
fields are additive and defensive (optional, default-false), so
behavior for existing sessions is unchanged; the main thing worth a
second look is the `subagent_retry` UI treatment, which has no
existing e2e coverage of a *live* rate-limited subagent (mocked via
fixture injection instead, per the "no chromium-live needed" bar).

## SDK changelog highlights

- 0.3.214: `set_permission_mode` now rejects unrecognized permission
  modes with an error instead of silently adopting them; `'manual'`
  accepted as an alias for `'default'` at every ingress —
  `[skipped — audited]`. `app/api/sessions/[id]/mode/route.ts`
  already validates `body.mode` against a fixed 6-value `MODES`
  allowlist (`default`, `acceptEdits`, `auto`, `dontAsk`,
  `bypassPermissions`, `plan`) before ever calling
  `session.setPermissionMode()` — `'manual'` and any other stray
  value are already rejected at our own boundary with a 400, and
  `Session.setPermissionMode()` (`lib/server/session.ts:3860`) wraps
  the SDK call in `.catch(() => {})`, so even a hypothetical
  SDK-level rejection is swallowed exactly like the old silent-adopt
  behavior. No call site passes an unvalidated string through.
- 0.3.214: `SDKMessageOrigin`'s `task-notification` member gains
  optional `subkind: 'scheduled-trigger'` —
  `[skipped — no consuming code path]`. Claudius only reads
  `origin.kind === 'peer'` today (`lib/client/use-session.ts:505-528`,
  `extractPeerOrigin`, for cross-session sender badges); nothing
  branches on `kind === 'task-notification'` or would read a
  `subkind`. Claudius's own scheduler (`lib/server/scheduler.ts`)
  tracks "why did this fire" through its own routine/cron rows, not
  by inspecting SDK message origin. Genuinely inert metadata today —
  flagged as a follow-up feature idea below, not a bug.
- 0.3.214: `applyFlagSettings({effortLevel})` now accepts `'max'` in
  its TS type (runtime already supported it) — `[type-only]`,
  shipped as a small cleanup. `lib/server/session.ts` `setEffort()`
  previously cast `level` to `"low" | "medium" | "high" | "xhigh"`
  (dropping `"max"` from the literal union) with a comment explaining
  the SDK's older type gap. The 0.3.214 `applyFlagSettings` signature
  now maps `effortLevel` to `EffortLevel | null` (which includes
  `'max'`) independent of the narrower `Settings['effortLevel']`
  shape, so the cast/comment is stale — removed, `level` now passes
  straight through.
- 0.3.214: assistant messages truncated by `interrupt()` now carry
  `aborted: true` — `[shipped]`. New: `DisplayMessage.aborted`,
  extracted in the `use-session.ts` assistant-message branch and
  threaded through `upsertAssistantSplit` (sticky across splits,
  mirroring `opusHighDemand`), rendered as a small "Interrupted"
  badge in `AssistantMessage.tsx`. Previously a Stop-clicked message
  rendered identically to a normally completed one, silently cutting
  off mid-word.
- 0.3.214: `tool_progress` messages gain optional `subagent_type` and
  `subagent_retry` (`{agent_id, attempt, max_retries, retry_delay_ms,
  error_status, error_category}`) so clients can show a subagent
  waiting out an API rate-limit retry — `[shipped]`. New:
  `ToolProgressInfo.subagentType` / `.subagentRetry`, extracted in
  the `tool_progress` reducer branch, rendered as an amber "retrying"
  badge on the matching Task card in `TaskBlock.tsx` (moved here from
  an initial `BackgroundTasksPanel` placement — `tool_progress`'s
  `parent_tool_use_id` points at the *outer* Task/Agent tool_use, not
  the subagent's own, so the lookup has to scan for a match via the
  new `findSubagentRetry()` helper in `lib/client/task-status.ts`,
  threaded down through `MessageList` → `AssistantMessage` →
  `TaskBlock` as a `toolProgress`/`progress` prop). Claudius already
  has a *global* `apiRetry` indicator (`lib/client/api-retry.ts` +
  `SpinnerTip`) for the main thread, but it can't represent one of
  several *parallel* subagents being individually rate-limited — this
  closes that gap.
- 0.3.214: `system/init`'s `plugins` entries and the `reload_plugins`
  response now include each plugin's manifest `version` —
  `[shipped]`. `app/api/plugins/route.ts` already forwards the
  `reload_plugins` response's `plugins` array verbatim as `installed`
  (`version` was arriving and being silently dropped); added
  `version?: string` to `InstalledPlugin`
  (`lib/client/usePlugins.ts`) and rendered it next to the plugin
  name in `PluginRow` (`app/plugins/page.tsx`).
- 0.3.214: SessionStart hooks now report source `"fork"` instead of
  `"resume"` when the session begins as a fork —
  `[skipped — no behavior to fix]` + trivial cosmetic fix shipped.
  Claudius never intercepts/executes SessionStart hooks itself (that
  runs entirely in the CLI/SDK) and has no `source === 'resume'`
  branch anywhere to mis-categorize. The only local trace is a
  display-only `matcherHint` string in `lib/shared/hook-events.ts`
  (`"startup | resume | clear | compact"`, used as placeholder copy
  in the `/hooks` editor) that was already missing `'fork'` even
  before this SDK release — updated it to list all five sources.
- 0.3.213: "Updated to parity with Claude Code v2.1.213" —
  `[skipped — parity marker, no itemized API change]`. No concrete
  SDK export changed in this line item beyond what's separately
  itemized in 0.3.212/0.3.214 above; CC-side TUI parity work has its
  own pipeline (`cc-parity/*`) and is out of scope here.
- 0.3.212: Fixed dash-leading `resumeSessionAt`/`sessionId` being
  passed to the CLI as separate argv tokens; both now use
  equals-form (`--flag=value`) —
  `[skipped — SDK-internal CLI subprocess invocation]`. Claudius
  calls the SDK's `query()`/`Query` methods; it never constructs the
  CLI's argv itself, so this fix is entirely inside the SDK's own
  subprocess-spawn code with no Claudius call site.
- 0.3.212: Agent tool output now includes the resolved model when a
  mid-turn model swap changed the subagent's model —
  `[skipped — no exported/typed shape to bind against]`. The
  completed shape is referenced in `sdk.d.ts` only as a docstring
  name (`AgentToolCompletedOutput`) on an intentionally
  `unknown`-typed `tool_use_result` field — it is not an exported
  type we can check field names against. Claudius does have
  precedent for hand-defining a local shape against an *observed*
  runtime payload (`BashToolUseResult` in `use-session.ts`), but
  doing that here would mean guessing an unconfirmed field name
  (e.g. `model`) rather than binding against a checked contract, for
  a low-visibility informational field (no functional behavior
  depends on it). Flagged as a follow-up below rather than shipped
  speculatively.

## Code changes

- `lib/server/session.ts` — `setEffort()`: drop the stale
  `"low"|"medium"|"high"|"xhigh"` cast/comment now that
  `applyFlagSettings`'s type accepts `'max'` directly (type-only
  cleanup, 0.3.214 `applyFlagSettings` item).
- `lib/client/types.ts` — add `DisplayMessage.aborted?: boolean`;
  add `subagentType?: string` / `subagentRetry?: {...}` to
  `ToolProgressInfo` (0.3.214 `aborted` + `tool_progress` items).
- `lib/client/use-session.ts` — extract `aborted` in the assistant
  SSE branch and thread it through `upsertAssistantSplit` (sticky
  merge); extract `subagent_type`/`subagent_retry` in the
  `tool_progress` branch.
- `components/chat/AssistantMessage.tsx` — render an "Interrupted"
  badge when `message.aborted`.
- `lib/client/task-status.ts` — add `findSubagentRetry()`: scans the
  `tool_progress` map for an entry whose `parentToolUseId` matches a
  Task's outer `toolUseId`, since `subagent_retry` is never keyed by
  the outer tool_use directly.
- `components/chat/TaskBlock.tsx` — accept a `progress` prop, call
  `findSubagentRetry()`, render an amber "retrying N/M" badge on the
  Task card while running.
- `components/chat/AssistantMessage.tsx` / `MessageList.tsx` /
  `ChatSurface.tsx` — thread `session.toolProgress` down as a
  `toolProgress` prop so `TaskBlock` can reach it.
- `lib/client/usePlugins.ts` — add `version?: string` to
  `InstalledPlugin`.
- `app/plugins/page.tsx` — render the plugin version next to its
  name in `PluginRow`.
- `lib/shared/hook-events.ts` — fix the stale `SessionStart`
  `matcherHint` copy to include `fork`.

## Risks / follow-ups

- The `subkind: 'scheduled-trigger'` field on task-notification
  origin is inert today — a nice follow-up would be threading it
  into Claudius's scheduler UI so a fired routine's chat entry can
  say "fired by schedule X" instead of a generic notification.
- The Agent/Task tool's resolved-model field (0.3.212) has no
  exported TS shape yet; revisit once the SDK exports a concrete
  `AgentToolCompletedOutput` type (or once we can confirm the wire
  field name from a live trace) rather than guessing at JSON keys.
- `subagent_retry`'s UI treatment has no live-agent e2e coverage
  (fixture-injected only) — acceptable per the run's "no
  chromium-live unless truly needed" guidance, but a real
  rate-limited subagent run would be a stronger regression check.
