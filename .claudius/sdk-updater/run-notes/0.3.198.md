## Summary

0.3.198 is a small patch release: one diagnostic warning (`canUseTool`
shadowed by `bypassPermissions`/bare `allowedTools`), one MCP
control-plane field not exposed in the public TS surface
(`request_timeout_ms` on `mcp_set_servers`), one upstream ingestion
bug fix (`isSynthetic` mapped to `isMeta` on transcript replay), and
one CLI-internal `/workflows` TUI bug (dropped earliest agents in the
progress tree). After auditing every call site in `lib/server/` and
`lib/client/` against each item, none require a Claudius code change
— this PR is the dependency bump plus this run-notes file. No
model-identity, alias, context-window, or pricing items appear in
this changelog, so the elevated audit bar for those doesn't apply
this release (checked anyway — see below). Headline risk to flag for
review: bypassPermissions sessions (`lib/server/session.ts`) will now
emit a Node `process.emitWarning` (`CLAUDE_SDK_CAN_USE_TOOL_SHADOWED`)
on every turn, which is new console noise but not a behavior change —
see Risks/follow-ups.

## SDK changelog highlights

- **Added a runtime warning when `canUseTool` is configured alongside
  `allowedTools` or `bypassPermissions`, which shadow the callback.**
  `[skipped — diagnostic-only, no call-site change]`. Confirmed in
  `node_modules/@anthropic-ai/claude-agent-sdk/sdk.mjs` (`WO`/`SW`
  helpers, warning code `CLAUDE_SDK_CAN_USE_TOOL_SHADOWED`): fires via
  `process.emitWarning` only when `canUseTool` is truthy AND
  (`permissionMode === 'bypassPermissions'` OR `allowedTools` has a
  bare/unparenthesized entry). The shadowing *behavior* itself
  (`canUseTool` not being invoked) predates 0.3.198 — only the warning
  is new, so this is not a behavior change to absorb. Audited every
  Claudius call site that builds SDK `Options`: `lib/server/session.ts`
  (`canUseTool` always set, `permissionMode` can be `bypassPermissions`
  per user selection — this is the one call site that will emit the
  new warning), `lib/server/session-recap.ts` (`canUseTool` set,
  `permissionMode: "default"` — no trigger), `lib/server/updater/apply.ts`,
  `lib/server/commit-message.ts`, `lib/server/customization-description.ts`
  (all `bypassPermissions`/bare `allowedTools`/`tools: []`, but none set
  `canUseTool` — no trigger). No test listens on `process.on('warning')`
  and no test asserts clean stderr, so nothing goes red. See
  Risks/follow-ups for the log-noise note.
- **Added per-server `request_timeout_ms` option to `mcp_set_servers`
  control request.** `[skipped — not exposed in the public SDK TS
  surface we consume]`. `request_timeout_ms` does not appear anywhere
  in `node_modules/@anthropic-ai/claude-agent-sdk/` (`sdk.d.ts`,
  `sdk.mjs`, `bridge.mjs`, `browser-sdk.js`, `sdk-tools.d.ts` — zero
  hits, including camelCase forms). The reason this is unreachable
  isn't the missing grep hit by itself — it's that Claudius's only
  path to this control request is the typed `Query.setMcpServers(
  servers: Record<string, McpServerConfig>): Promise<McpSetServersResult>`
  (`sdk.d.ts` ~2460), and every `McpServerConfig` variant
  (`McpStdioServerConfig`, `McpSSEServerConfig`, `McpHttpServerConfig`,
  `McpSdkServerConfigWithInstance`) exposes only the pre-existing
  `timeout` field, not `request_timeout_ms`. Claudius's only caller
  (`Session.setMcpServers` in `lib/server/session.ts`, invoked from
  `app/api/sessions/[id]/mcp-dynamic/route.ts`) forwards the client's
  servers object through that same typed method — there is no field to
  plumb through even if we wanted to.
- **Fixed `SDKUserMessage.isSynthetic` not being mapped to `isMeta` on
  ingestion, which could cause synthetic messages to be treated as
  real user messages.** `[skipped — upstream-only fix, inherited for
  free]`. This is a bug in how the SDK/CLI replays persisted JSONL
  history (resume/pagination), not in the live message stream.
  Claudius's `lib/client/sdk-message-filters.ts`
  (`isSdkInternalEnvelope`) already reads `msg.isMeta` on both the live
  and disk-replay paths to hide transcript-only plumbing from the chat
  UI; once Claudius runs against 0.3.198, replayed synthetic messages
  that previously slipped through `isMeta`-based filtering (because
  the mapping was buggy) will now be caught automatically — no
  call-site change needed. The separate direct `isSynthetic` read in
  `lib/client/use-session.ts` (~line 2629) is on the *live* path and
  reads the field directly (never relied on the `isMeta` mapping), so
  it's unaffected either way.
- **Fixed workflow progress events silently dropping earliest agents
  from the list while the phase counter remained correct.**
  `[skipped — CLI-internal, no consumable event]`. This describes the
  Claude Code CLI's own interactive `/workflows` terminal progress
  tree, which tracks a live per-agent list. Claudius does not render or
  subscribe to any such list: `components/chat/WorkflowBlock.tsx`
  explicitly documents (line ~20-23) that "the SDK does not expose the
  per-agent breakdown," so the aggregate task is the only live progress
  signal available. Claudius's workflow UI only consumes the aggregate
  `local_workflow` task fields (status, summary, totalTokens, toolUses,
  durationMs) via `task_started`/`task_updated`/`task_progress`/
  `task_notification` system events (`lib/client/use-session.ts`
  ~3285-3400) — no `agents: [...]` array exists on any of those event
  types for this bug to have corrupted from our side. Independently
  verified: the only `agents` field anywhere near these message types
  in `sdk.d.ts` is `SDKInitMessage.agents` (a session-init list of
  available agent-type names), unrelated to per-workflow-phase
  progress tracking.
- No model-name/alias/default-model/context-window/pricing-tier items
  appear in this changelog, so the elevated model-identity audit bar
  doesn't apply this release — noting explicitly per the run
  instructions rather than silently skipping the check.

## Risks / follow-ups

- `lib/server/session.ts` always sets `canUseTool` and forwards
  `permissionMode` (including `bypassPermissions`, a user-selectable
  mode). Starting with 0.3.198, every turn run in `bypassPermissions`
  mode will emit a Node `process.emitWarning` with code
  `CLAUDE_SDK_CAN_USE_TOOL_SHADOWED` to the server process's stderr.
  This is harmless (no exception, no behavior change — the shadowing
  itself predates this release) but adds recurring log noise. Not
  fixed in this PR since it's cosmetic; if server logs get noisy,
  consider a targeted `process.on('warning', ...)` filter that drops
  only this warning code.
- Confirmed but out of scope: in `bypassPermissions` mode,
  `canUseTool` is never invoked at all (per the SDK's own shadowing
  behavior), which means the `ExitPlanMode`/`AskUserQuestion`
  interception logic in `Session.canUseTool`
  (`lib/server/session.ts` ~2415-2510) cannot fire for a session
  running in that mode. This is pre-existing behavior (not introduced
  by 0.3.198) and `ExitPlanMode` is mutually exclusive with
  `bypassPermissions` anyway (different `permissionMode` values), but
  `AskUserQuestion` could in principle be called by the model during a
  `bypassPermissions` turn and would not get our custom UI. Worth a
  follow-up audit of whether any Claudius flow actually runs
  `bypassPermissions` on a path where the model can call
  `AskUserQuestion` (customization/commit-message/updater paths use
  `tools: []`/`allowedTools` allow-lists that likely exclude it, but
  this wasn't exhaustively verified for every flow).
