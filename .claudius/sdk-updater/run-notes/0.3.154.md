## Summary

0.3.153 added `SDKThinkingTokensMessage` to the public `SDKMessage` union — a
new system event (`subtype: "thinking_tokens"`) emitted during the
redacted-thinking phase that carries a live estimate of how many tokens the
model has spent thinking so far. Without handling it, the message would
fall through to Claudius's catch-all `system/<subtype>` pill renderer and
appear as a cryptic `system/thinking_tokens` row in the chat. We suppress it
from the chat and wire the estimated token count through to the existing
thinking-spinner row in the Activity rail, so users see a live token counter
while the model thinks. 0.3.154 fixed an internal SDK bug where stdio MCP
servers were unnecessarily restarted on every reconcile pass; Claudius
benefits passively without any code change.

## Code changes

- `lib/client/sdk-message-filters.ts` — Added `"thinking_tokens"` to
  `SUPPRESSED_SYSTEM_SUBTYPES` so the new system message is never rendered as
  a `system/thinking_tokens` chat pill.

- `lib/client/types.ts` — Added `estimatedThinkingTokens?: number` to
  `ToolHistoryEntry` to carry the live token estimate on the synthetic thinking
  spinner row.

- `lib/client/use-session.ts` — Added a `thinking_tokens` handler in the SDK
  system message dispatch block: updates the most-recent open thinking row's
  `estimatedThinkingTokens` field with the incoming `estimated_tokens` value.
  Clears the estimate on `message_stop` so it doesn't bleed into the next turn.

- `components/panels/BackgroundTasksPanel.tsx` — Displays
  `estimatedThinkingTokens` (in k-tokens format) on active thinking rows,
  alongside the existing elapsed-time readout.

## Risks / follow-ups

- `SDKThinkingTokensMessage` carries no `message_id` or `parent_tool_use_id`,
  so the token estimate is attached to the most-recent open thinking row by
  heuristic. In practice there is at most one thinking block in flight, but
  a future SDK version that emits concurrent thinking blocks would need a more
  precise correlation key.
- `message_stop` clears `estimatedThinkingTokens` on thinking rows, which is
  correct for the live path. For the transcript-replay path, the estimate is
  never present (thinking_tokens messages are transient and not stored in the
  JSONL), so replayed thinking rows always have `estimatedThinkingTokens:
  undefined` — this is the expected behaviour.
