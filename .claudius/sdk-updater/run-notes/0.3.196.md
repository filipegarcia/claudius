## Summary

The 0.3.196 release adds a `prompt_id` field to `BaseHookInput` (the base type
extended by every hook payload) so that hook event handlers can correlate events
with the originating user prompt and with OpenTelemetry `prompt.id` spans. It
also fixes a control-protocol deduplication bug where the internal resolved
tool-use-ID cache was bounded at 1000 entries; entries dropped from that cache
could lead to duplicate `tool_result` deliveries in very long-running sessions.
Neither change requires any Claudius code modification: the `prompt_id` field is
absorbed transparently by the existing typed hook inputs, and the dedup fix is
entirely within the SDK binary with no corresponding workaround in the Claudius
client or server. All four gates (lint, vitest, build, e2e) passed on the
migrated tree without code changes.

## SDK changelog highlights

- `[type-only]` **`prompt_id` added to `BaseHookInput`** — A new optional
  `prompt_id?: string` field on every hook payload correlates hook events to a
  user-prompt grain and to OpenTelemetry `prompt.id` attributes. Claudius's
  inline hook handlers in `lib/server/session.ts` receive `PreToolUseHookInput`,
  `PostToolUseHookInput`, and `CwdChangedHookInput` — all extending
  `BaseHookInput` — but none of those handlers inspect or forward the field. The
  new field is available to external hook scripts (shell commands, HTTP handlers)
  via the JSON payload the SDK delivers; that delivery is handled entirely inside
  the SDK. No Claudius code reads or needs to read `prompt_id`, so no behavioural
  change is required. `[type-only]` rather than `[skipped]` because the type
  definition (`BaseHookInput`) is a public export that Claudius imports (via the
  three hook-input subtypes), and the new field is now in scope on every hook
  cast in `session.ts`.

- `[skipped — SDK-internal bug fix]` **Control-protocol dedup cap fixed** —
  The SDK's internal map of resolved tool-use IDs was capped at 1000 entries;
  after overflow, dropped IDs could cause duplicate `tool_result` deliveries in
  very long sessions. Fixed inside the SDK binary with no change to the public
  API surface. Audited `lib/client/use-session.ts`, `lib/client/merge-replay-messages.ts`,
  and `lib/server/thinking-replay-recovery.ts` for client-side tool-result
  dedup workarounds that might have been added in response to this bug — found
  none related to the control-protocol dedup issue. Existing client-side dedup
  in `use-session.ts` (message-UUID and cost dedup) is orthogonal to the
  protocol-layer fix; no simplification is needed.

## Code changes

- No code changes required. The `prompt_id` addition is absorbed transparently
  by the existing type imports (`PreToolUseHookInput`, `PostToolUseHookInput`,
  `CwdChangedHookInput` all extend `BaseHookInput`), so no casts or logic
  anywhere in `lib/server/session.ts` need updating. The deduplication fix
  required no Claudius workaround to exist in the first place; the client-side
  dedup in `use-session.ts` addresses a different concern (replay-merge
  idempotency) and is unaffected.
