## Summary

The 0.3.208 → 0.3.210 changelog has two entries: 0.3.210 adds
`timedOutAfterMs` to the SDK's `BashOutput` shape (set when a
foreground command hits its timeout and is auto-backgrounded), and
0.3.209 is a Claude Code v2.1.209 parity bump with no new public API
of its own — that parity work already landed on `main` via a
separate, already-merged pair of PRs (`feat(cc-parity): claude-code
2.1.207 → 2.1.209` #145 and `chore(deps): bump claude-agent-sdk
0.3.207 → 0.3.209` #146), so there's nothing left to do for it here.
For `timedOutAfterMs`: Claudius already tracked background bash
shells, but only ones explicitly launched with
`Bash(run_in_background: true)` — a command that ran in the
foreground and got auto-backgrounded after hitting its timeout was
completely invisible to the UI, since nothing at launch time
indicated it would ever go to background. This PR wires the new
field through: the client now reads the SDK message's structured
`tool_use_result` sibling field (previously never consumed anywhere
in the codebase) to detect this case, creates/updates the
background-bash entry even when it wasn't launched in the
background, and the Activity rail's "Running" list now shows a
"timed out" badge with the timeout duration. Headline risk: the new
detection path relies on `tool_use_result.backgroundTaskId` /
`timedOutAfterMs` being present on the SDK message, which is
untyped (`unknown`) upstream — if a future SDK version renames or
drops these fields silently, the feature degrades to its pre-0.3.210
behavior (no crash, just lost visibility) rather than erroring
loudly. See "## Risks / follow-ups".

## Code changes

- `lib/client/types.ts` — `BackgroundBash` gained `timedOutAfterMs?: number`,
  documenting the new auto-backgrounded-on-timeout case alongside the
  existing `run_in_background`-launch case.
- `lib/client/use-session.ts` — added `BashToolUseResult` type and the
  pure `applyBashAutoBackground(prev, args)` helper (exported for unit
  testing), and wired it into the `user`/tool_result reducer branch: for
  any Bash tool_result, it reads `msg.tool_use_result` (the SDK's raw
  per-tool `BashOutput`, a sibling of `message.content` that Claudius
  never read before this) and creates or merges a `backgroundBashes`
  entry when `backgroundTaskId` and/or `timedOutAfterMs` is present —
  regardless of whether the command was originally launched with
  `run_in_background`.
- `components/panels/widgets/BackgroundBashes.tsx` — renders an amber
  "timed out" badge (`data-testid="bash-timeout-badge"`) with a
  tooltip citing the timeout in seconds, and tints the row amber
  instead of the default green/running tone, when
  `timedOutAfterMs` is set and the shell hasn't been killed.

## Risks / follow-ups

- `tool_use_result` is typed `unknown` upstream (`sdk.d.ts`'s
  `SDKUserMessage.tool_use_result?: unknown`) and Claudius narrows it
  with an unchecked cast in `applyBashAutoBackground`'s call site. If
  a future SDK version renames `backgroundTaskId` / `timedOutAfterMs`
  or nests them differently, this degrades silently (the badge simply
  stops appearing) rather than throwing — there's no runtime
  validation. Low severity (visibility-only feature, not
  correctness-critical) but worth a comment-anchored grep next SDK
  bump if `BashOutput`'s shape changes again.
  Alternative rejected: adding a full runtime schema check (e.g. zod)
  for `tool_use_result` felt like overkill for a single optional
  visibility field, given the pattern isn't used elsewhere in the
  codebase for this message field.
  - Update: the same badge/tone logic was NOT extended to
    `backgroundedByUser` (manual Ctrl+B backgrounding) or
    `backgroundCwdHint`, both pre-existing sibling fields on
    `BashOutput` that Claudius also doesn't surface today. Those are
    out of scope for this changelog entry (neither changed in
    0.3.209/0.3.210) but are a natural follow-up if a future release
    wants full backgrounding-reason parity in the UI.
