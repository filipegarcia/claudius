## Summary

0.3.217→0.3.218 is a small release: a Remote Control reconnect fix, two
result-message accuracy fixes (`api_error_status`, `modelUsage.canonicalModel`/
`provider`), a `SkillToolOutput.background` flag, and — the one with real
teeth for Claudius — 0.3.217 lowered the default subagent nesting depth from
5 to 1 and added a default 20-way cap on concurrently-running subagents. That
silently changes behaviour for every existing Claudius session the moment
this bump lands: any workflow that relied on a subagent spawning its own
nested subagent (depth > 1) now gets blocked with no Claudius-side signal
that anything changed. We ship a `Limits` settings extension (mirroring the
existing CC 2.1.212 `maxWebSearches`/`maxSubagents` precedent) that lets a
user override both `CLAUDE_CODE_MAX_SUBAGENT_SPAWN_DEPTH` and
`CLAUDE_CODE_MAX_CONCURRENT_SUBAGENTS` per-workspace via `query()`'s `env`
option. Everything else audited to a real call site and found nothing to
change — see below for the audit trail. Headline risk: the depth-cap default
change itself (now silently in effect for every session that doesn't opt
into an override) is worth a reviewer's attention.

## SDK changelog highlights

- 0.3.218 — `SkillToolOutput` now reports `background: true` when a forked
  skill was dispatched as a detached background agent.
  `[skipped — no public type, no existing Skill call-site]`. Neither
  `sdk.d.ts` nor `sdk-tools.d.ts` export a `SkillToolOutput`/`SkillInput`
  type (grepped both files; the only `Skill`-adjacent exports are
  `SlashCommand` and the unrelated `ProposeSkillsInput`/`Output`). Claudius
  has no special-casing for the `Skill` tool anywhere in
  `lib/client/use-session.ts` or `components/chat/ChatSurface.tsx` — Skill
  tool_use/tool_result blocks render through the fully generic tool-call UI,
  same as any third-party tool. Unlike the `Workflow`-tool background-ack
  precedent (`lib/client/use-session.ts` ~2884), whose exact ack shape
  Claudius itself controls, the Skill tool's fork-to-background output shape
  is CLI-internal and undocumented in the public SDK types, so there is
  nothing to safely bind against.
- 0.3.218 — Fixed the result event's `api_error_status` reporting null for
  rate-limit/overloaded errors delivered mid-stream; now reports 429/529.
  `[skipped — already consumed generically, zero Claudius diff]`. Per
  `sdk.d.ts`, `api_error_status` exists only on `SDKResultSuccess` (line
  4275), not on `SDKResultError` (4239–4257) — so this fix only improves the
  "fallback-rescued success" path, which is exactly the case
  `lib/server/opus-overload-detector.ts`'s `isOverloadSignal()` already
  reads via its own text-matching on `errors[]` for the *failed* path and
  deliberately skips counting on the *success* path (see its doc comment,
  lines 17–23) — no change needed there. `app/api/models/probe/route.ts:131`
  already reads `e.api_error_status` generically
  (`` `api_error_${e.api_error_status}` ``); it now gets real 429/529 values
  instead of always falling through to `e.subtype`. Both call sites get more
  accurate data automatically from the SDK bump with zero Claudius code
  changes required.
- 0.3.218 — Added `canonicalModel` and `provider` to each `modelUsage` entry
  in result messages for billing/`costUSD` rate-table lookups.
  `[skipped — audited, unused surface]`. Model-identity/pricing item, so
  audited per the higher bar: `lib/client/types.ts:515` stores
  `modelUsage?: Record<string, unknown>` — deliberately opaque, no field is
  ever destructured from it (confirmed via repo-wide grep: no
  `.modelUsage.` field access outside the passthrough store). Live cost
  estimate reconciliation (`lib/client/use-session.ts` ~3357) uses the
  result message's own authoritative `total_cost_usd`, never a per-model
  breakdown. The on-disk cost report (`lib/server/cost-aggregate.ts`) reads
  `message.model`/`message.usage` straight off JSONL assistant-message rows
  and resolves pricing via `litellm-pricing.ts`'s `priceForModel()`, which
  has its own alias/family-fallback heuristic (exact match → strip
  provider prefix → family substring match) — it never touches
  `result.modelUsage`. `canonicalModel`/`provider` simply have no consumer
  to wire into; nothing in `litellm-pricing.ts`'s long-context tier logic
  changes either. Not `app/api/models/route.ts` fable-row material — this is
  a billing-attribution field on usage records, not a model list entry.
- 0.3.217 — Subagents no longer spawn nested subagents by default (depth cap
  lowered from 5 to 1); `CLAUDE_CODE_MAX_SUBAGENT_SPAWN_DEPTH` overrides.
  `[shipped]`. New `Limits.maxSubagentDepth` field, `env`-passthrough in
  `lib/server/session.ts`, and a `LimitsPanel` field — see Code changes.
- 0.3.217 — Added a cap on concurrently-running subagents (default 20,
  `CLAUDE_CODE_MAX_CONCURRENT_SUBAGENTS`). `[shipped]`. Same mechanism as
  above — `Limits.maxConcurrentSubagents`.
- 0.3.217 — Fixed Remote Control sessions not re-sending pending permission
  prompts to clients that connect after the prompt appeared.
  `[skipped — no Claudius call-site]`. "Remote Control" is the CLI's own
  feature for external (e.g. mobile) clients attaching directly to a running
  `claude` process; grepped the repo for `Remote Control`/`RemoteControl` —
  zero hits. Claudius talks to the SDK's `query()` iterator in-process from
  `lib/server/session.ts` and fans permission prompts out over its own SSE
  layer (`pendingPermissions` map + `broadcast`), which already re-sends
  pending prompts to newly-subscribing clients independent of this CLI
  feature. Nothing to change.

## Code changes

- `lib/server/limits-store.ts` — added `maxSubagentDepth?: number` and
  `maxConcurrentSubagents?: number` to the `Limits` type (CC 0.3.217 parity,
  same "0/undefined = use the CLI default" convention as the existing USD/
  tool-budget caps).
- `app/api/limits/route.ts` — validate + persist the two new fields on PUT.
- `lib/shared/tool-budget.ts` — added `buildSubagentEnvOverrides(limits)`,
  a pure function mapping the two new `Limits` fields to
  `CLAUDE_CODE_MAX_SUBAGENT_SPAWN_DEPTH`/`CLAUDE_CODE_MAX_CONCURRENT_SUBAGENTS`
  env vars, kept dependency-free for unit testing (mirrors
  `checkToolBudget`'s existing pattern in the same file).
- `lib/server/session.ts` — `start()` now merges
  `buildSubagentEnvOverrides(this.toolBudgetLimits)` into the `env` passed to
  `query()`, both when an account-switcher `envOverride` is active (merged in)
  and when it isn't (materializes `{ ...process.env, ... }` only when an
  override is actually configured, so the zero-config path stays
  byte-identical to before).
- `components/cost/LimitsPanel.tsx` — two new `Field`s ("Max subagent nesting
  depth", "Max concurrent subagents") in the same section as the existing
  tool-budget caps, with `data-testid`s for e2e.

## Risks / follow-ups

- The depth-cap default drop (5→1) takes effect for every Claudius session
  the moment this SDK bump ships, with no override configured. Any workflow
  relying on a subagent spawning a *nested* subagent (depth > 1) —
  Claudius's own `Workflow` tool's `agent()` calls are one level and
  unaffected, but a general-purpose subagent that itself calls `Task` would
  now be silently blocked — will regress with no in-app signal beyond
  whatever the CLI itself surfaces to the model. We did not add a banner/
  toast for this because there's no reliable event to hang it off (the
  block happens inside the SDK's own subagent scheduler, not something
  `canUseTool` sees) — flagging for reviewer awareness rather than shipping
  a speculative detector.
- `maxConcurrentSubagents`/`maxSubagentDepth` are one-way env overrides (set
  once at session start); changing them in Settings mid-session requires a
  fresh session to take effect, same as the existing `maxWebSearches`/
  `maxSubagents` caps' `/clear`-to-reset convention — not called out
  separately in the UI copy beyond the existing panel's session-scoped
  framing.
