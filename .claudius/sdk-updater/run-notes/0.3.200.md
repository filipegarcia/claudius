## Summary

SDK 0.3.200 is a small, mostly-internal release: it adds `'manual'` as an
accepted alias for `'default'` in the CLI's `permissions.defaultMode`
settings field, fixes `onSetPermissionMode` not firing for SDK-hosted
Remote Control sessions, and hardens `set_model` so an unrecognized model
string is rejected *before* it latches instead of possibly landing in a
half-applied state. The first two items have no Claudius call site (audited
below). The third does touch us: Claudius's `/model <id>` chat command lets
a user type an arbitrary, unvalidated model string, and it fired an
unconditional "Model → `<id>`" success toast the instant the request went
out — before the server round-trip could confirm it. That toast was always
slightly wrong for a bad id (the pre-existing `ModelSwitchNoticePanel`
would follow up seconds later with "Couldn't switch to `<id>`", so the user
saw two contradictory messages back to back), but 0.3.200's stricter,
more-reliable rejection makes that contradiction fire far more
consistently than before. Fixed by changing the toast copy from a claim of
success to a claim of an attempt (`Switching model → <id>`), so it no
longer contradicts the rejection notice when one follows. No other code
changes were required — the server-side rejection plumbing
(`session.ts#setModel` → 409 → `ModelSwitchNoticePanel`) already existed
and needed no changes to correctly benefit from the new guarantee.
Headline risk to flag for review: this is a copy-only UX fix, not new
functionality — reviewers should confirm the reworded toast still reads
naturally for the success case, which is still the overwhelming majority
of `/model` usage.

## SDK changelog highlights

- **Added `'manual'` as an accepted alias for the `'default'` permission
  mode in SDK inputs** — `[skipped — audited, no value-comparison call
  site]`. Traced to `Options.permissions.defaultMode` (`sdk.d.ts:4746`,
  doc comment: "'manual' is accepted as an alias for 'default'"). This is
  the CLI's `~/.claude/settings.json` `permissions.defaultMode` field, a
  different knob from the per-session `Options.permissionMode` that
  Claudius's mode selector drives. Audited every Claudius surface that
  touches `permissions.defaultMode`:
  - `lib/server/settings.ts` — `PermissionRules.defaultMode` is typed
    `string` and round-tripped opaquely through `readSettings` /
    `updatePermissions`; nothing compares it against a specific literal.
  - `lib/shared/tips.ts` — the only *read* of the field
    (`Boolean(_?.permissions?.defaultMode)`) is a plain truthiness check
    for the "make Plan Mode sticky" spinner tip; `'manual'` is just as
    truthy as `'default'`, so behavior is unchanged either way.
  - `app/api/settings/permissions/route.ts` — forwards whatever `patch`
    body it's given; Claudius itself never sends `defaultMode: 'manual'`
    or `defaultMode: 'default'` from any UI (see next point), so this is
    pure passthrough.
  - `components/workspaces/WorkspaceForm.tsx` (workspace "Permission
    mode" dropdown) and `components/chat/ModeSelector.tsx` (in-session
    mode cycle) — both only ever emit the canonical `PermissionMode`
    values (`default`, `acceptEdits`, `plan`, `bypassPermissions`,
    `dontAsk`, `auto`) into `Options.permissionMode` / `setPermissionMode`,
    never into `permissions.defaultMode`, and never the string `'manual'`.
  Since Claudius never writes or string-compares this field, the SDK
  normalizing a user's hand-edited `settings.json` is transparent to us.
- **Fixed `onSetPermissionMode` callback not firing for SDK-hosted Remote
  Control sessions** — `[skipped — Remote Control not implemented in
  Claudius]`. Claudius has no remote/teleport concept; the `/remote-control`
  slash command explicitly short-circuits with "Remote control lets
  claude.ai drive a terminal Claude Code session — not exposed by
  Claudius" (`components/chat/ChatSurface.tsx`). No SDK-hosted Remote
  Control session is ever created from this codebase, so the fixed
  callback path is unreachable here.
- **Fixed `set_model` control request accepting unrecognized model
  strings; invalid models are now rejected before latching** —
  `[type-only]` for the server plumbing, `[shipped]` for a small client
  UX fix it motivates. `Query.setModel(model?: string): Promise<void>`
  (`sdk.d.ts:2270`) is unchanged in shape; the fix is a runtime guarantee
  (reject-before-latch) that `lib/server/session.ts#setModel` already
  assumed — it only sets `this.model` / persists / broadcasts *after* a
  successful `await this.query.setModel(model)`, and already surfaces a
  thrown rejection as `{ ok: false, error }` → HTTP 409 → the pre-existing
  `ModelSwitchNoticePanel`. The one place this newly-more-reliable
  rejection was under-served: `ChatSurface.tsx`'s `/model <id>` handler
  fired an unconditional "Model → `<id>`" toast before the POST resolved,
  which reads as a false confirmation right before the rejection notice
  contradicts it. Fixed (see Code changes).

## Code changes

- `components/chat/ChatSurface.tsx` — the `/model` slash-command handler's
  immediate toast now reads `Switching model → <id>` instead of
  `Model → <id>`, so it no longer claims success before the SDK confirms
  the switch. The async `session.setModel(...)` call itself is unchanged
  (still fire-and-forget; the existing `ModelSwitchNoticePanel` /
  `modelSwitchNotice` state machinery in `lib/client/use-session.ts`
  already reverts the optimistic pill and surfaces the SDK's rejection
  reason on failure — that path needed no changes).
- `tests/unit/model-picker-route.test.ts` — added `POST` coverage pinning
  the reject-before-latch contract: a `query.setModel` rejection surfaces
  as HTTP 409 with the SDK's error message and the server-authoritative
  model, and a success surfaces as HTTP 200 with the new model. This
  contract already existed in `route.ts`; it simply had no test before
  this release exercised it more.

## New UI surfaces

- No new UI surfaces this release. `ModelSwitchNoticePanel` (the
  "Couldn't switch to `<id>`" banner) is pre-existing — this release
  didn't add it, it just makes it reachable more often via `/model
  <invalid-id>` now that the SDK rejects bad ids more reliably. The only
  visual change is the wording of the immediate chat-command toast
  (`Switching model → <id>` vs. the old `Model → <id>`), which is a copy
  tweak on an existing toast, not a new element. It's still covered by a
  screenshot for reviewer verification: `docs/sdk-updates/0.3.200/model-switch-rejected.png`,
  captured by `tests/e2e/sdk-update-0.3.200-model-switch-rejected.spec.ts`.
  The shot is taken mid-chat on an active workspace session after sending
  `/model definitely-not-a-real-model`, showing the composer, the
  transient "Switching model → …" toast area, and the amber
  `ModelSwitchNoticePanel` banner with its "Couldn't switch to
  definitely-not-a-real-model" headline and SDK error text, all in the
  surrounding chat chrome.

## Risks / follow-ups

- The `/model` toast fix is copy-only and deliberately conservative (see
  "The single most common failure mode" guidance to avoid speculative
  API changes) — it does not make the client aware of the actual
  success/failure outcome at the call site, it just stops the immediate
  toast from asserting an outcome it doesn't know yet. If a future
  release wants the toast to reflect the real result, `setModel`'s
  client-hook return type (`lib/client/types.ts`) would need to grow an
  `{ ok, error }` shape, which ripples into `ModelPicker`'s `onPickModel`
  prop — deliberately out of scope here.
- None identified beyond the above.
