## Summary

SDK 0.3.163 delivers two SDK-internal bug fixes and one type-level
addition. The `stop_task` control request now returns success instead
of throwing when the target task is already gone (`not_found` or
`not_running`), making stale-chip pruning reliable for host clients;
Claudius's try/catch wrapper already handles the success path
correctly, so no route or session code changes are needed, though the
chip-pruning opportunity (discussed below) was consciously deferred.
The `setMcpServers` bug fix enables hosts to add builtin MCP servers
(e.g. `claude-in-chrome`) via `setMcpServers` even when the CLI was
launched without them — Claudius's call site is unchanged. Finally,
`StopHookSpecificOutput` and `SubagentStopHookSpecificOutput` now
declare `additionalContext?: string`, enabling non-error feedback on
Stop/SubagentStop hooks that continues the turn; Claudius does not
register programmatic Stop hooks and user-configured ones work
transparently via the SDK, so this is type-only. No Claudius code
changes were required; all four gates remain green.

## Risks / follow-ups

- **Optimistic stale-chip pruning (deferred)**: The stop_task SDK fix
  was specifically motivated by enabling clients to prune stale task
  chips reliably. Claudius's `BackgroundTasksPanel.stopTask` is
  currently fire-and-forget (`.catch(() => {})`), so chips are only
  removed when a `task_notification` SSE event arrives. For stale tasks
  (where no `task_notification` will come), chips persist until the
  next reconciliation opportunity. Wiring stop-task API success to an
  optimistic task-state update would require passing a callback or
  refactoring task state into a shared store, which carries race-
  condition risk with simultaneous `task_notification` events. Deferred
  to a focused follow-up; the SDK change makes this safe to implement
  whenever the team is ready.

- **Stop/SubagentStop additionalContext — programmatic hook rejected**:
  The new `additionalContext` on Stop hooks could theoretically drain
  Claudius's pending mid-turn reminder queue at turn end (reminders
  that arrived between the last tool call and the stop). This was
  rejected because a Stop hook returning `additionalContext` causes the
  turn to continue automatically, which risks an infinite loop if a
  reminder is queued every turn. The safer path (draining at the next
  user input, which already happens via `takePendingReminders`) is
  retained. A future reviewer might consider implementing the Stop hook
  with a one-shot guard if the loop risk can be bounded.
