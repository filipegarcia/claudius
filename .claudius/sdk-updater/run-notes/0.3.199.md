## Summary

0.3.199 is a small patch release with three items, none of which are
breaking changes and none of which touch model identity, default
models, or pricing tiers. `CanUseTool`'s options gained a `requestId`
field and its return type widened to allow `null` (for out-of-band
permission responses); `sandbox.credentials` gained a `mode: "mask"`
+ `injectHosts` option for masking specific env vars into sandboxed
commands; and a `blocked` field was added to an internal
`workflow_agent` progress event that never reaches the SDK's public
`query()` message stream. After auditing every call site (`canUseTool`
in `lib/server/session.ts`, the `Options.sandbox` block, and the
Workflow-tool's task-progress plumbing), none of the three require
Claudius code changes — the type changes are either transparently
absorbed or apply to a sandbox sub-feature Claudius doesn't expose any
UI for today. Headline risk to flag for review: this is a
zero-code-change PR (dependency bump only); the reviewer should
confirm they agree none of the three items warrant new surface area
rather than assume the audit missed something.

## SDK changelog highlights

- **`requestId` on `canUseTool` options + `null` return to suppress the
  automatic control response** — `[type-only]`. `CanUseTool`'s
  callback signature (`lib/server/session.ts` imports the SDK's
  `CanUseTool` type directly) now exposes `ctx.requestId` (the SDK's
  internal control-request envelope id) and allows the callback to
  return `null` when the consumer has already written a
  `control_response` out-of-band (e.g. a separate process posting a
  signed HTTP response that echoes `requestId`, bypassing the SDK's
  own transport). Claudius's `canUseTool` (`session.ts:2415`) always
  resolves its `Promise<PermissionResult>` in-process — there is no
  separate worker/transport, no signed out-of-band HTTP path, and no
  reconnect logic keyed on the SDK's internal request id (Claudius
  mints and tracks its own `requestId` via `randomUUID()` for its
  `pendingPermissions` map, broadcast events, and the notifications
  DB — see `drainPendingDecisions`). The new field/return-type widen
  through the imported type with zero edits (a function returning
  `Promise<PermissionResult>` already satisfies
  `Promise<PermissionResult | null>`), which is exactly what
  `[type-only]` means — nothing to wire, nothing broke.
- **`blocked` field on `workflow_agent` progress events (auto-mode
  safety classifier)** — `[skipped — no public SDK export]`. The
  string `workflow_agent` appears nowhere in any shipped `.d.ts`
  (`sdk.d.ts`, `sdk-tools.d.ts`, `bridge.d.ts`, `browser-sdk.d.ts`,
  `agentSdkTypes.d.ts`) and zero times in the compiled `sdk.mjs` —
  it never reaches `query()`'s `SDKMessage` union, which is the only
  channel Claudius consumes SDK events through. (That grep doesn't
  cover the bundled `claude` CLI binary the SDK shells out to, so the
  event may exist on some internal wire between the CLI and its own
  TUI — but that's moot for us either way, since Claudius has no
  consuming surface for it regardless of where it originates.)
  Consistent with this, `components/chat/WorkflowBlock.tsx` already
  documents that "the SDK does not expose the per-agent breakdown" —
  Claudius only ever sees one aggregate `local_workflow` task
  (`task_started`/`task_progress`/`task_updated`, summed
  tokens/tools/duration) per `Workflow` tool call, never a per-agent
  stream. There is no existing per-agent progress surface for a
  `blocked` flag to attach to.
- **`mode: "mask"` + per-credential `injectHosts` on
  `sandbox.credentials`** — `[skipped — no existing sandbox-config
  surface to extend]`. This is additive to `Options.sandbox`
  (`SandboxCredentialsConfig`, part of the same `SandboxSettings`
  type Claudius already imports for `Options.sandbox`), so it is
  reachable in principle. Audited the one call site
  (`lib/server/session.ts` ~line 2038): Claudius sets exactly three
  sandbox keys — `enabled`, `autoAllowBashIfSandboxed`,
  `failIfUnavailable` — gated by a single `sandboxEnabled` boolean on
  the session/workspace form. There is no UI or session option for
  `sandbox.network`, `sandbox.filesystem`, or `sandbox.credentials`
  at all today; this release only adds one field to a config
  sub-object we don't expose any of. Building a credentials-masking
  UI (pick env vars, pick a mask mode, pick allowed egress hosts) from
  scratch is a real feature, not a narrow follow-through on existing
  wiring, so it's out of scope for a patch-release absorption pass —
  flagged below as a follow-up if the team wants sandbox credential
  masking as a feature.

## Risks / follow-ups

- If Claudius ever grows a distributed/multi-process permission-
  response path (e.g. a mobile-push-driven decision handled by a
  different process than the one running `query()`), revisit the new
  `ctx.requestId` / `null`-return `canUseTool` support — it's the
  SDK's sanctioned mechanism for exactly that, and right now Claudius
  would have to invent its own.
- Sandbox credential masking (`sandbox.credentials.envVars[].mode:
  "mask"` + `injectHosts`) is a real, if narrow, feature request:
  today Claudius's sandbox toggle is all-or-nothing with no way to
  protect specific env vars (e.g. cloud credentials) from sandboxed
  commands while still letting them reach an allowed egress host in
  masked form. Worth a scoped follow-up (workspace-settings UI +
  server wiring) if sandboxing sees more use, but out of scope for
  this absorption-only patch bump.
- The `workflow_agent`/`blocked` audit relied on grepping the shipped
  `.d.ts` files and the compiled `sdk.mjs`; it does not cover the
  bundled `claude` CLI binary's internal wire protocol. If a future
  SDK release exposes per-agent Workflow progress through
  `query()`'s public `SDKMessage` union, `WorkflowBlock.tsx`'s
  "aggregate task only" comment (and this skip) should be revisited
  together.
