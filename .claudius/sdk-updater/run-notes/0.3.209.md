# SDK update 0.3.207 → 0.3.209

## Summary

`@anthropic-ai/claude-agent-sdk` moves 0.3.207 → 0.3.209. The 0.3.209
tag is a CLI-parity label bump only (v2.1.209, already handled by the
separate cc-parity pipeline in PR #145 — out of scope here); the
substantive content is 0.3.208's set of internal bug fixes (hook/abort
races, an `extraArgs`/`resume` CLI-flag-injection parsing bug, an
abort-listener leak, an MCP-timeout doc fix, a stdin-after-exit crash).
None of these add or change a public SDK type Claudius consumes — I
diffed `sdk.d.ts` between the two npm tarballs directly (beyond the
prose changelog) and the only shape-level diffs are an optional field
on a control-request type we never construct
(`SDKControlSetMaxThinkingTokensRequest.max_thinking_tokens`) and two
new/reordered fields on the CLI's own terminal `Settings` interface
(`vimInsertModeRemaps`, `preferredNotifChannel` reorder) that Claudius's
hand-rolled `ClaudeSettings` type doesn't mirror and has no browser
equivalent for. There are no model-identity items in this range at all
(no model names/aliases/context-window/pricing/capability-tier
changes), so that audit gate doesn't apply.

The one item that **did** need Claudius-side code: the `resume`
flag-injection fix. `POST /api/sessions` passes `body.resume` —
untrusted, unvalidated request JSON — straight through
`sessionManager.create()` into the SDK's `resume` option, which
becomes a literal CLI argument to the spawned Claude Code binary. The
0.3.208 SDK fix closes the parsing bug on the SDK's side, but this
route is a cross-origin-reachable sink for that value (Next.js route
handlers aren't CSRF-gated), so relying solely on the dependency to
close it felt thin. I added a UUID-shape validator at the route so a
crafted dash-leading `resume` (e.g. `--version`) is rejected with 400
before it ever reaches `sessionManager.create` / the SDK, as
defense-in-depth alongside the transitive SDK fix. Headline risk to
flag for review: this is a genuine (if narrow) hardening change to a
production route, not a pure version-bump no-op — worth a second look
at the 400 behavior for any legitimate caller that might send a
non-UUID `resume` (audited: none do — see "## Code changes").

## SDK changelog highlights

- 0.3.209 — "Updated to parity with Claude Code v2.1.209":
  `[skipped — parity work tracked separately]`. This label refers to
  the CLI-parity surface, which the cc-parity pipeline already shipped
  in `feat(cc-parity): claude-code 2.1.207 → 2.1.209 (#145)` (merged to
  `main` ahead of this branch). No additional SDK-level export changed
  for this label; confirmed via a full `sdk.d.ts` diff (0.3.207 tarball
  vs. installed 0.3.209), see Summary.
- 0.3.208 — "Fixed a caller abort during a pending SDK hook callback
  being converted into hook success, which let PreToolUse-gated tools
  execute after the abort": `[skipped — no Claudius call-site]`.
  Claudius registers its own `PreToolUse`/`PostToolUse`/`CwdChanged`
  hooks (`lib/server/session.ts` ~2242-2340); every one of them
  resolves synchronously without awaiting anything abortable, so the
  bug window (an abort racing a *pending* hook promise) never applied
  to our hooks. User-authored `settings.json` hooks benefit
  automatically from the SDK fix with no Claudius change needed.
- 0.3.208 — "Fixed a per-query resource leak in the SDK's process
  tracking when spawning the CLI fails": `[skipped — SDK-internal]`.
  Purely inside the SDK's own process-tracking table; no Claudius
  surface touches it.
- 0.3.208 — "Fixed an SDK `UserPromptSubmit` hook callback exceeding
  its timeout killing the entire query with an empty error; it now
  blocks the prompt with a clear timeout message and the session
  continues": `[skipped — no Claudius call-site, pure improvement for
  user hooks]`. Claudius doesn't register its own `UserPromptSubmit`
  hook (only lists it as an available hook type for user-authored
  `settings.json` hooks in `lib/shared/hook-events.ts`). Hook failures
  are already surfaced generically via the existing hook-stderr UI
  (cc-parity 2.1.199); the improved message just makes that existing
  channel's content clearer, no code change required to receive it.
- 0.3.208 — "Fixed `extraArgs` values that look like flags (e.g.
  `resume: '--version'`) being parsed as their own CLI flags":
  `[shipped]`. Claudius never sets `extraArgs`, but the *example* in
  this changelog entry is exactly our own `resume` call-site
  (`lib/server/session.ts` passes `this.resumeFrom` — sourced straight
  from `POST /api/sessions`' `body.resume` — as the SDK's `resume`
  option). The SDK fix is received transitively via the version bump,
  and on top of it I added a UUID-shape guard at the route (see "##
  Code changes") so a crafted dash-leading value never reaches the
  SDK's argv construction in the first place — defense-in-depth for a
  cross-origin-reachable POST body field, not something to leave
  solely to the dependency.
- 0.3.208 — "Fixed an abort-listener leak: streaming queries sharing
  one `AbortController` no longer accumulate `abort` listeners on its
  signal after each completed query": `[skipped — no Claudius
  call-site]`. Checked `lib/server/session.ts`'s `AbortController`
  usage (`this.abortController`, `this.recapAbort`): each is a fresh
  `AbortController` per session / per recap request (`recapAbort` is
  explicitly replaced with `new AbortController()` per call, aborting
  the prior one first), never a single controller reused across many
  independent streaming `query()` calls the way the bug requires. Pure
  upside from the bump, no code change needed.
- 0.3.208 — "Fixed `createSdkMcpServer` docs pointing at a nonexistent
  env var; the MCP tool-call timeout knob is `MCP_TOOL_TIMEOUT`":
  `[type-only]`. Doc-comment-only fix in the SDK's `.d.ts`; Claudius's
  own `createSdkMcpServer` usage (`lib/server/session.ts`
  `buildGoalMcpServer`) doesn't reference the env var name anywhere
  (grepped `MCP_TOOL_TIMEOUT|TOOL_TIMEOUT` — zero hits), so there was
  nothing stale to fix on our side.
- 0.3.208 — "Fixed an uncaught exception when writing to stdin after
  the Claude Code subprocess has exited": `[skipped — SDK-internal]`.
  Claudius never writes to the spawned CLI's stdin directly; all
  process lifecycle goes through the SDK's `query()` abstraction
  (grepped for direct `.stdin`/`child_process` usage in
  `lib/server/session.ts` and `session-manager.ts` — zero hits).

## Code changes

- `app/api/sessions/route.ts` — added a `UUID_RE` guard on
  `POST`'s `body.resume` that rejects a non-UUID-shaped value with
  `400 { error: "invalid resume id" }` before it reaches
  `sessionManager.create()`. This is the concrete fix tied to the
  `extraArgs`/`resume` flag-injection changelog item above: audited
  every caller of `resume:` on the client (`lib/client/use-session.ts`
  — always a session id from `crypto.randomUUID()` or a URL `?session=`
  param) and the other server-side resume path
  (`lib/server/session-resume.ts` `getOrResumeSession`, which only
  calls `sessionManager.create({ resume: id })` after confirming a
  matching on-disk session file exists for `id` — already implicitly
  gated), confirming no legitimate caller ever sends a non-UUID value,
  so the 400 can't regress a real flow.

## New UI surfaces

- No new UI surfaces this release. The only code change is a
  server-side input-validation guard on an existing API route; it has
  no client-visible affordance (a malformed `resume` value was never a
  supported/reachable UI flow to begin with — only a raw
  cross-origin/curl POST could trigger it).

## Tests

- vitest: 1 new spec
  (`tests/unit/sessions-create-route-resume-validation.test.ts`, 4
  tests) covering: a dash-leading `resume` (`--version`) → 400 and
  `sessionManager.create` never called; an arbitrary non-UUID string →
  400; a well-formed UUID `resume` → 200 and passed through verbatim;
  `resume` omitted entirely → 200 (fresh-session path unaffected).
- playwright: 0 new/updated specs. No user-visible behaviour changed —
  the guard only rejects a malformed field that no UI path produces —
  so there's nothing for an e2e spec to exercise beyond what the
  vitest unit test already covers at the route boundary.
- Not covered: the SDK's own internal argv-construction fix (the
  actual `--` binding change inside the CLI binary) isn't
  independently re-tested here — that's upstream SDK behaviour we
  trust via the version bump, not something Claudius's test suite can
  observe short of spawning the real CLI with a crafted resume value.

## Risks / follow-ups

- The 400 on a non-UUID `resume` is a genuine (if narrow) behaviour
  change to a production route. I audited every current caller (see
  "## Code changes") and found none that send anything but a
  `randomUUID()`-shaped id, but if some external/headless integration
  relies on a looser `resume` format, this would newly reject it.
  Flagging for reviewer sign-off rather than treating it as risk-free.
- `getOrResumeSession` (`lib/server/session-resume.ts`) was left
  unchanged — it's already effectively guarded (only forwards `resume`
  after confirming an on-disk session file matches the id), but it
  doesn't share the new explicit `UUID_RE` check. Low risk since the
  disk-lookup gate means an attacker-supplied non-existent id already
  short-circuits to `null` before reaching `sessionManager.create`, but
  a future refactor of that disk-lookup gate should preserve (or add)
  the same UUID validation rather than relying only on "file not
  found."
- None identified beyond the above.
