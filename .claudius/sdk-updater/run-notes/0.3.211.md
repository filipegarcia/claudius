## Summary

SDK 0.3.211 is a small release: a CLI-only replay-ordering fix, a new
`SDKAssistantMessage.timestamp` field on the live stream, a new `@alpha`
export of the CLI's canonical usage-limit prose prefixes
(`USAGE_LIMIT_ERROR_PREFIXES` and three siblings), and richer process-exit
error text. Three of the four items need no Claudius code change (audited
below — not hand-waved). The fourth is real: Claudius had **two**
hand-mirrored regexes (one in `lib/client/use-session.ts`, one duplicated
in `lib/server/session.ts`) that detected a "hard rate-limit hit" assistant
message by matching only the single "You've hit your … limit" phrasing —
missing the CLI's other eleven usage-limit templates (out of credits, org
out of usage, seat-type restrictions, admin-disabled, Fable-credits-
required, …). Both call sites now share one `lib/shared/rate-limit-prefixes.ts`
module built on the SDK's canonical list, with a vitest test that imports
the real SDK export and asserts exact equality so future drift fails loudly
instead of silently. Headline risk: the server-side detector also gates the
account-auto-rotate side effect, so this change widens *when* auto-rotate
fires (from 1 prefix to 12) — audited in Risks/follow-ups below; the
existing design is best-effort/opt-in so this is low-risk, but flagging for
review since it's a behavior change, not just a bugfix to display copy.

## SDK changelog highlights

- `[skipped — CLI-only flag, not in the SDK's Options surface]` Fixed
  `--replay-user-messages` with `--include-partial-messages` emitting the
  turn-start user replay after the first content block instead of before.
  Claudius sets `includePartialMessages: true` in `lib/server/session.ts`
  but never sets `replayUserMessages` — grepped `sdk.d.ts` and confirmed
  there is no `replayUserMessages` field on `Options` at all; this is a
  raw CLI flag (`--replay-user-messages`) with no SDK-level knob, so
  Claudius (which drives the SDK via `query()` options, never raw CLI
  flags) has no call site to fix.
- `[skipped — already handled generically, audited]` Added
  `SDKAssistantMessage.timestamp` (ISO-8601) to the live stream, matching
  `SDKUserMessage`. Audited both places Claudius reads message
  timestamps: (1) `lib/server/session.ts`'s historical-resume replay
  (~line 1753, "Carry-forward timestamp for assistants") and its
  `resyncFromDisk` twin (~line 5187) both already read
  `(m as { timestamp?: string }).timestamp` generically off *any* message
  — not role-restricted to user messages — and only fall back to
  carry-forward when the field is absent. They will pick up genuine
  assistant timestamps automatically the moment a resumed session's JSONL
  was recorded by an 0.3.211+ CLI; zero code change needed. (2) The live
  broadcast path (`consume()`'s `for await` loop → `broadcast()`)
  deliberately stamps `at` via `Date.now()` receive-time for every
  message type by design — the code comment says so explicitly ("live
  sites leave it unset and we default to 'now' here") — which is exactly
  the SDK's own documented fallback guidance ("consumers should fall
  back to receive time"). No behavior change needed there either.
- `[shipped]` Added rate-limit message prefix buckets
  (`USAGE_LIMIT_ERROR_PREFIXES` and siblings) as `@alpha` exports for
  classifying rate-limit messages without hand-mirrored lists. Of the
  four sibling consts:
  - `USAGE_LIMIT_ERROR_PREFIXES` — **shipped**. This is the list
    Claudius's existing (narrow, one-template) hand-rolled regex was
    trying to approximate. See "Code changes" below.
  - `USAGE_TRANSITION_PREFIXES` — `[skipped — no existing detector]`.
    Upstream docs it as "toast only; these never arrive as API errors."
    Grepped the codebase for any "now using credits" / usage-transition
    prose matcher — none exists, so there's no hand-mirrored logic to
    converge onto this list. Building a new transition-toast feature is
    out of scope for this SDK bump.
  - `USAGE_WARNING_PREFIXES` — `[skipped — no existing detector]`. Same
    reasoning: "footer/toast only", and Claudius has no "You've used …"
    prose matcher to fix.
  - `ORG_POLICY_LIMIT_PREFIXES` — `[skipped — no existing detector]`.
    Upstream docs route this to "org-disabled presentation" — Claudius
    has no such UI surface today (grepped for "disabled for your org" /
    "org policy" — no hits outside a comment). Adding one would be a new
    feature, not a fix to existing hand-mirrored logic, so out of scope.
- `[skipped — already handled generically, audited]` Improved
  process-exit errors to include the CLI's stderr output. Audited the
  one place Claudius catches a thrown query-iterator error
  (`consume()`'s `catch (err)` block, ~line 6947): it does
  `err instanceof Error ? err.message : String(err)` and broadcasts that
  string verbatim as a `session_error` — no truncation, no reformatting,
  no assumption about message shape. Since the richer text is baked into
  `err.message` by the SDK itself, it flows through to the client
  automatically with zero code change.

## Code changes

- `lib/shared/rate-limit-prefixes.ts` (new) — canonical
  `USAGE_LIMIT_ERROR_PREFIXES` copy (hand-copied literal, since the SDK's
  main entry is Node-only and would break the client bundle if imported
  as a value from browser code — see the file's doc comment) plus
  `matchesUsageLimitPrefix(text)`, a curly-quote-normalizing,
  case-insensitive prefix matcher against all twelve templates.
- `lib/client/use-session.ts` — `isRateLimitHitText` now delegates to
  `matchesUsageLimitPrefix` instead of the old `RATE_LIMIT_HIT_TEXT_RE`
  regex (which only matched "You've hit your … limit"). Same exported
  function name/signature, so no caller elsewhere needs to change.
- `lib/server/session.ts` — `isRateLimitHitSdkMessage` (now exported for
  unit testing) uses the same shared `matchesUsageLimitPrefix` instead of
  its own duplicated copy of the narrow regex. Updated the doc comments
  on `isRateLimitHitSdkMessage` and `noteAccountAutoRotateObservation` to
  describe the shared module and to call out, explicitly, that widening
  the prefix set also widens the account-auto-rotate trigger set (see
  Risks below).

## New UI surfaces

  account-auto-rotate side effect (server) fire — it does not add or

  unchanged; verified by reading `RateLimitHitPanel`'s call sites, which
  consume the same `isRateLimitHitText` boolean and `rateLimitTypeFromText`

## Risks / follow-ups

- **Auto-rotate trigger set widened.** `isRateLimitHitSdkMessage` gates
  `noteAccountAutoRotateObservation` (the account-switcher's
  auto-rotate-on-rate-limit side effect). Broadening detection from 1
  prefix to all 12 in `USAGE_LIMIT_ERROR_PREFIXES` means auto-rotate now
  also fires on: out-of-credits, org-wide billing issues (2 templates),
  seat-type restrictions (2 templates), admin-disabled allocation,
  $0 group limit, Fable-credits-required, and extra-usage exhaustion —
  previously only the generic "You've hit your … limit" phrasing
  triggered it. This is a deliberate, considered call, not an
  overlooked side effect: rotation here is strictly best-effort and
  opt-in (`autoRotateOnRateLimit` must be enabled, requires ≥2
  configured profiles, swallows all errors, never blocks or surfaces a
  failure). Even for an org-wide billing message, rotating to the next
  configured profile is harmless — that profile might belong to a
  different org/plan and genuinely unblock the user, and if not, it's a
  no-op modulo one wasted round-robin step. Flagging so a reviewer can
  confirm this reasoning; if it's wrong, the fix is to scope
  `noteAccountAutoRotateObservation`'s check to a subset of
  `USAGE_LIMIT_ERROR_PREFIXES` rather than the full list.
- **`USAGE_LIMIT_ERROR_PREFIXES` is `@alpha`.** If a future SDK release
  renames, removes, or reorders it, `tests/unit/rate-limit-prefixes-sdk-sync.test.ts`
  will fail on the next SDK-update run (by design — that's the point of
  the sync test), and `lib/shared/rate-limit-prefixes.ts` will need a
  matching update. Not a risk to this PR, but worth knowing why that
  test might go red on a later bump.
- Unrelated: while running the full e2e suite locally, several
  `docs/cc-parity/*.png` screenshots (committed by earlier, unrelated
  cc-parity PRs) were regenerated with pixel-level diffs as a side
  effect of re-running those specs. These are outside the scope of this
  SDK bump and were reverted before this PR's final commit so the diff
  stays scoped to the 0.3.211 migration.
