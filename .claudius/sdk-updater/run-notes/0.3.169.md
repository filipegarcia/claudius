## Summary

SDK 0.3.169 adds two features: (1) an experimental `usage_EXPERIMENTAL_MAY_CHANGE_DO_NOT_RELY_ON_THIS_API_YET()` method on `Query` returning structured session cost, claude.ai subscription type, and per-window plan rate-limit utilization; and (2) an `SSEOptions` alternative transport for the browser SDK's `BrowserQueryOptions`. Claudius ships the usage API — calling it after each successful turn to surface subscription type and rate-limit window utilization in the existing "Session cost & usage" overlay — and skips the browser-SDK SSE transport (Claudius does not import `@anthropic-ai/claude-agent-sdk/browser`). The headline risk is the API's explicit experimental label: the method name will change when stabilized, which will break the call site in `lib/server/session.ts` on a future SDK bump; mitigated by the try/catch wrapper and the `[skipped — experimental]` note in Risks.

## SDK changelog highlights

- `[shipped]` — `Query.usage_EXPERIMENTAL_MAY_CHANGE_DO_NOT_RELY_ON_THIS_API_YET()` returning `SDKControlGetUsageResponse` (session cost, `subscription_type`, `rate_limits_available`, per-window `rate_limits`). Wired server-side as a post-turn call in `lib/server/session.ts`; broadcast as a new `plan_usage` SSE event; consumed in `lib/client/use-session.ts`; displayed in the `CostOverlay`. The `session.*` cost fields overlap with what Claudius already accumulates from per-turn result events; only `subscription_type` and `rate_limits.*` are genuinely new. The `behaviors` field (local transcript attribution) is ignored — it's large, approximates local-only usage, and has no clear home in the current UI.
- `[skipped — browser SDK only]` — `SSEOptions` / `BrowserQueryOptions.sse` in `@anthropic-ai/claude-agent-sdk/browser`. Claudius uses the Node.js server-side SDK entry point exclusively; it has no imports of the `/browser` export. This transport is for browser-side consumers of the CCR streaming API and is not applicable here.

## Code changes

- `lib/shared/events.ts` — added `PlanUsageEvent` type (carries `subscriptionType`, `rateLimitsAvailable`, `rateLimits` windows) and added it to the `ServerEvent` union.
- `lib/client/types.ts` — added `PlanRateLimits` type (mirrors event payload); added `planUsage: PlanRateLimits | null` field to `ChatState`.
- `lib/server/session.ts` — imported `SDKControlGetUsageResponse`; after each `subtype === "success"` result event, calls `this.query.usage_EXPERIMENTAL_MAY_CHANGE_DO_NOT_RELY_ON_THIS_API_YET()` in a non-blocking, error-swallowing branch and broadcasts a `plan_usage` event when data arrives.
- `lib/client/use-session.ts` — added `planUsage: null` to initial state; handles `plan_usage` event in `applyEvent` to update `planUsage`.
- `components/overlays/CostOverlay.tsx` — accepts optional `planUsage: PlanRateLimits | null` prop; renders subscription type badge and per-window utilization bars when `rateLimitsAvailable` is true.
- `components/panels/BackgroundTasksPanel.tsx` — threads `planUsage` prop through to `CostOverlay`.
- `app/[workspaceId]/page.tsx` — passes `session.planUsage` to `CostOverlay`.

## Risks / follow-ups

- The SDK call site (`usage_EXPERIMENTAL_MAY_CHANGE_DO_NOT_RELY_ON_THIS_API_YET`) **will break** when the API is stabilized and the method is renamed in a future SDK bump. The try/catch wrapper ensures this is a silent no-op (not a runtime error) until then, but the Risks/follow-ups section for that future bump should grep for the long method name and update it.
- The `behaviors` field from `SDKControlGetUsageResponse` is silently dropped server-side. If the UI later wants per-skill/agent attribution it lives in that field — add it to the `PlanUsageEvent` shape at that time.
- `extra_usage` window (overage credit balance / enabled flag) is not surfaced in the overlay. Adding it is straightforward once the product wants to show it.
