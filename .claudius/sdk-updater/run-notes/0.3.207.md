## Summary

0.3.207 is a small release with two changelog items. (1) A `canUseTool`
contract fix in the SDK binary — `{behavior: "allow"}` without
`updatedInput` used to be mis-rejected as a raw-ZodError deny; now the
tool runs with the original input. Audited every `behavior: "allow"`
return site in `lib/server/session.ts` and every one already supplies
`updatedInput`, so Claudius never hit the buggy path — no code change
required there. (2) The Agent/Task tool's structured completion output
now has a real exported type (the changelog calls it
`AgentToolCompletedOutput`; the actual exported symbol in this
package's `.d.ts` is the `status: "completed"` arm of `AgentOutput` in
`sdk-tools.d.ts` — the changelog name doesn't match the export, see
Risks). Claudius already receives this payload — it rides the `user`
SSE event's `tool_use_result` field, forwarded verbatim by
`Session.broadcast()` — but never read it; subagent stats came only
from the separate `task_progress`/`task_notification` system-message
channel (tokens/tool-count/duration), and "Returned to parent" only
ever showed the flattened text. We shipped the enhancement the
changelog explicitly recommends ("render from it instead of parsing
the tool_result text"): `TaskBlock` now surfaces the resolved model,
a per-tool-kind usage breakdown, and worktree isolation info when the
completed payload carries them. Headline risk: this is optional
richness on an already-working display, not a required fix — the new
fields are best-effort and absent on older/replayed transcripts.

## SDK changelog highlights

- `canUseTool` allow-without-`updatedInput` no longer mis-rejected as
  a ZodError deny — `[skipped — audited]`. Every `behavior: "allow"`
  return in `lib/server/session.ts` (`resolveCanUseTool` at line 2480,
  the decision-resolution paths at 2621/2690/2750) already sets
  `updatedInput` alongside `behavior: "allow"`, so Claudius never
  triggered the bug either before or after this fix. `PermissionResult`
  in `sdk.d.ts` still types `updatedInput` as required on the allow
  variant, so no type change either.
- Agent tool structured result gets a published type
  (`AgentToolCompletedOutput` per changelog / `AgentOutput`'s
  `status: "completed"` arm in the actual `.d.ts`) — `[shipped]`.
  Wired the `tool_use_result` payload (already flowing over SSE,
  previously unread) into `TaskInfo` and rendered it in `TaskBlock`:
  resolved model badge, tool-kind usage breakdown
  (reads/searches/bash/edits +lines), and worktree branch/path chip
  for `isolation: "worktree"` subagents.

## Code changes

- `lib/shared/agent-tool-output.ts` (new) — pure helper
  `extractAgentToolCompletion()` that narrows an unknown
  `tool_use_result` payload to the completed `AgentOutput` shape and
  projects the fields Claudius renders (`resolvedModel`, `toolStats`,
  `worktreePath`, `worktreeBranch`). Kept out of `lib/server` /
  `lib/client` because both sides need it: the client reads it live
  off the SSE stream, and it's plain data-shape logic with no I/O.
- `lib/client/types.ts` — `TaskInfo` gains four optional fields:
  `resolvedModel`, `toolStats`, `worktreePath`, `worktreeBranch`.
- `lib/client/use-session.ts` — in the `msg.type === "user"` handler,
  read `msg.tool_use_result` alongside the existing text-based
  `extractToolResult()` call and, when it parses as a completed Agent
  output, merge the projected fields onto the matching `TaskInfo` row
  via a new `mergeAgentCompletionIntoTask()` reconciler (same
  "reconcile off the tool_result" pattern already used for
  `reconcileTasksOnToolResult`, TaskCreate re-keying, etc. right next
  to it).
- `components/chat/TaskBlock.tsx` — renders the resolved-model pill,
  the tool-kind stat breakdown, and a worktree branch chip when
  `TaskInfo` carries them; all additive and hidden when absent (older
  transcripts / non-worktree runs), so existing renders are unchanged.

## Risks / follow-ups

- The changelog names the type `AgentToolCompletedOutput`; the actual
  exported symbol in `node_modules/@anthropic-ai/claude-agent-sdk/sdk-tools.d.ts`
  is `AgentOutput` (a union; we narrow to its `status: "completed"`
  arm). If a future SDK release renames/splits this export, the
  narrowing in `lib/shared/agent-tool-output.ts` is the one place to
  update.
- The new TaskBlock fields are best-effort: `tool_use_result` is only
  present on live/replayed transcripts produced by an SDK new enough
  to emit it, and older persisted sessions won't have it. The block
  already degrades gracefully (fields simply don't render), but there's
  no backfill for historical rows.
- `resolvedModel`/`toolStats`/worktree info aren't surfaced anywhere
  outside `TaskBlock` (e.g. no session-level rollup of subagent
  worktree usage) — flagged as a possible follow-up, not attempted
  here since it's out of scope for this changelog item.
