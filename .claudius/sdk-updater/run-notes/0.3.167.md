## Summary

SDK 0.3.167 (which subsumes 0.3.166) brings parity with Claude Code v2.1.167.
The TypeScript API surface changes all landed in 0.3.166: a new
`SDKControlRegisterRepoRootRequest` control-request type, two new optional
fields on `SDKResultSuccess` (`ttft_stream_ms`) and `Settings`
(`fallbackModel?: string[]`), and two new optional fields formalising
`WorkflowOutput.taskType`/`workflowName` that Claudius was already reading from
raw message payloads. 0.3.166 also includes a runtime binary fix for MCP
resource tools not being injected when servers are added via `mcp_set_servers`
at runtime — Claudius's `mcp-dynamic` route benefits automatically. 0.3.167
itself is a binary-only rebuild. None of the type-surface changes required any
Claudius code updates: the new control request has no Claudius UI surface, the
timing field is an optional add-on Claudius does not surface, the Settings
`fallbackModel` is in the settings-file interface (not the `QueryOptions`
interface Claudius uses), and `WorkflowOutput.taskType`/`workflowName` were
already handled via raw message reads. All four gates are green with no code
changes.

## Risks / follow-ups

- **`SDKControlRegisterRepoRootRequest`**: The new control request is not wired
  up in Claudius. If a future workspace feature needs to add a subdirectory root
  mid-session (e.g., opening a nested repo inside a monorepo), this control
  request is the mechanism to use. No action needed now; noted for future
  multi-root workspace work.
- **`Settings.fallbackModel?: string[]`**: The SDK's settings-file interface now
  supports a `fallbackModel` array. Claudius exposes fallback model only as a
  per-session `QueryOptions` parameter; there is no settings-file UI for it. If
  we later want to persist a default fallback model to `~/.claude/settings.json`,
  we should update `ClaudeSettings` in `lib/server/settings.ts` to include
  `fallbackModel?: string[]` and wire it through the session-defaults path.
- **Opaque binary changes**: Both changelog entries are brief
  ("parity with Claude Code v2.1.x") with no specifics. Any behavioral changes
  in the bundled binary (altered tool-result shapes, changed session lifecycle,
  modified streaming) would only surface at runtime, not in the typecheck or
  unit-test gate.
