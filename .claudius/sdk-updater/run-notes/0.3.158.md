## Summary

The 0.3.157 and 0.3.158 releases ("parity with Claude Code v2.1.157/158") carry
two additive type-surface changes. The upstream changelog is vacuous; the ground
truth is a diff of all `.d.ts` files against the packed 0.3.156 tarball — only
`sdk.d.ts` and `sdk-tools.d.ts` changed; `assistant.d.ts`, `bridge.d.ts`,
`browser-sdk.d.ts`, `agentSdkTypes.d.ts`, and `extractFromBunfs.d.ts` are
unchanged. One change — `workflowKeywordTriggerEnabled` on `Settings` — is
absorbed by the `[key: string]: unknown` catch-all in Claudius's `ClaudeSettings`
type and requires no code. The other — `response?: string` on
`AskUserQuestionOutput` — has a clear home: when a single-question form is
answered with the "Other" free-text path (user typed instead of selecting a
structured option), Claudius now forwards that text in the top-level `response`
field of `updatedInput`, exactly matching the SDK's new semantics. The
`updatedInput` assembly was also extracted into a pure, tested helper function
`buildAskUpdatedInput` (lib/shared/events.ts) to improve testability. A
pre-existing React anti-pattern in `CustomizationsDrawer` (side effect inside a
`setState` updater) was also fixed — it caused intermittent e2e flakiness
unrelated to the SDK bump.

## Code changes

- `lib/shared/events.ts` — new exported pure function `buildAskUpdatedInput(questions, answers)`
  that assembles the `updatedInput` object for AskUserQuestion resolution.
  Includes the SDK 0.3.158 `response` field logic (single-question / "Other"
  path only). Extracted here (rather than left inline in session.ts) so it
  can be unit-tested without instantiating a Session.

- `lib/server/session.ts` — `submitAskAnswer`: replaced inline
  `answersMap` / `annotations` / `response` assembly with a call to
  `buildAskUpdatedInput`. JSDoc updated to document the new `response` field.
  Import of `buildAskUpdatedInput` added.

- `components/nav/CustomizationsDrawer.tsx` — fixed a pre-existing React
  anti-pattern: `onOpen()` was called inside a `setState` updater function,
  triggering a "Cannot update a component while rendering a different
  component" warning in concurrent mode. Moved the `onOpen()` call outside
  the updater (reading `open` directly from the closure, which is safe in a
  synchronous click handler). This resolved intermittent e2e flakiness in
  `customizations-drawer.spec.ts`.

## New UI surfaces

- No new UI surfaces this release. Both SDK changes are either absorbed
  invisibly by the catch-all type or are a backend-only wire-through of data
  the user already provides in the existing AskUserQuestion modal.

## Tests

- vitest: 1 new spec file (`tests/unit/ask-answer-response.test.ts`), 3 cases
  covering the single-question "Other" path (response populated), the
  single-question structured-option path (response absent), and the
  multi-question path (response absent). All 748 specs pass.
- playwright: 0 new specs — the `response` wiring is a server-side
  `updatedInput` mutation inside the SDK control channel; there is no
  browser-observable behaviour difference at the Playwright layer.
  103 chromium specs pass (5 skipped).
- Not covered: multi-question forms with multiple "Other" answers (the
  `response` field is intentionally left unset for those — see Risks).

## Risks / follow-ups

- Multi-question forms with multiple "Other" answers do not populate `response`
  (the field is a single string; ambiguous which question it would map to).
  If the SDK ever clarifies multi-question `response` semantics, this should be
  revisited.
- `workflowKeywordTriggerEnabled` is silently accepted via the catch-all but
  not surfaced in the Claudius settings UI. If a user sets it in their
  `.claude/settings.json` directly it will be respected by the SDK subprocess,
  but there is no Claudius toggle. Schedule for the next settings-surface pass.
