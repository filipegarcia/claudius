## Summary

Claude Code 2.1.206 is a small release: most of its substance is
terminal-UX polish (`/cd` path autocomplete, agents-view TUI fixes),
CLI-distribution mechanics (background-agent self-upgrade), and
auth/gateway infrastructure that Claudius inherits automatically
through the Agent SDK. The one product-side feature Claudius
reimplements this release is `/doctor`'s new "checked-in CLAUDE.md is
too big" check: `app/doctor` gains a per-project-workspace warning when
a checked-in `CLAUDE.md`/`.claude/CLAUDE.md` crosses a line-count
threshold, with a link into the existing per-workspace Memory editor
(`/[workspaceId]/memory`) rather than a stub. An adversarial review
pass (a `code-reviewer` agent instructed to refute the UX fit, not just
check correctness) caught and fixed two real issues before this was
called done: an off-by-one in the line-counter that over-flagged any
file with a trailing newline, and `customization`-kind workspaces
(Claudius's own dogfood workspaces) being counted alongside real user
projects. The headline risk to flag for review: the size threshold
(300 lines) is a Claudius-invented heuristic, not something scraped
from upstream (upstream does in-session, model-driven trim analysis —
the running model reads the file and proposes cuts — which this check,
a fast, syntax-only doctor probe with no active session, has no way to
replicate); a reviewer should confirm 300 lines is the right knob, and
whether one warn row
per oversized workspace (vs. an aggregated single row) is the right
cardinality once a user has many large workspaces.

## Changelog classification

- `[C — CLI/terminal only] Added directory path suggestions to /cd, matching /add-dir behavior` — confirmed via `code.claude.com/docs/en/commands`: `/cd` relocates a terminal session's active working directory (and its on-disk resume storage) — a concept with no Claudius analog (Claudius sessions are pinned to a workspace's `rootPath`, not free-navigable like a shell `cd`). This entry is *only* about adding fuzzy tab-completion to the path argument of an existing terminal command — readline-style autocomplete is a terminal-input affordance Claudius's plain-text `/add-dir` field doesn't need (users paste/type full paths, or browse via the existing Files page), and `/cd` itself was never adopted as a Claudius feature in any prior cc-parity run (checked `.claudius/cc-parity/run-notes/*.md` — no prior classification exists). Building a filesystem-autocomplete widget off this one line would be scope invention past what the entry asks for.
- `[B — reimplement in Claudius] Added a /doctor check that proposes trimming checked-in CLAUDE.md files by cutting content Claude could derive from the codebase` — a product-side diagnostic: `/doctor` is a settings/health surface Claudius already reimplements independently of the SDK (`app/doctor/page.tsx` + `app/api/doctor/route.ts`), so a new check it gains upstream is something Claudius's own doctor implementation needs to gain too, not something that flows through automatically. Per the changelog's own wording, upstream does this via in-session, model-driven trim analysis (the running model reads the file and proposes specific cuts); every other Claudius doctor check is a fast, deterministic, session-less GET probe with no active model call, so replicating a model-driven analysis here would be a first (and a cost/latency surprise on a page that currently loads instantly). The conservative reimplementation substitutes a fixed line-count heuristic that flags oversized files and links to the existing Memory editor for a human (or a chat turn) to do the actual trim — see Risks for why 300 lines specifically, and why this is a Claudius-invented number rather than one derived from upstream. Shipped as `[extend: /doctor]` below.
- `[skip — ambiguous, see Risks] /commit-push-pr now auto-allows git push to the repo's configured push remote (remote.pushDefault, or the sole remote when only one is configured) in addition to origin` — `/commit-push-pr` is not documented in the public Claude Code docs (`code.claude.com/docs/en/commands` omits it entirely) and there is no `v2.1.206` tag on `anthropics/claude-code` yet to diff a commit from, so its full behavior can't be pinned down beyond the changelog line. Claudius has no `/commit-push-pr` command (it ships its own git integration instead: the `/[workspaceId]/git` page + the `commit-draft` skill), so there is nothing to extend even under the most conservative reading — a Bash-tool permission auto-allow rule for a command Claudius doesn't run.
- `[A — already via SDK updater] Gateway: /login now supports Anthropic-operated public gateway endpoints` — confirmed via `code.claude.com/docs/en/third-party-integrations`: LLM-gateway routing is configured through `ANTHROPIC_BASE_URL`/`ANTHROPIC_*_BASE_URL` env vars and resolved inside the CLI/SDK's own auth+networking layer, which Claudius invokes via `@anthropic-ai/claude-agent-sdk`'s `query()` — Claudius's own OAuth flow (`lib/server/account-oauth.ts`) is a separate concern (Claudius's own sign-in), not the gateway-endpoint resolution this entry changes.
- `[A — already via SDK updater] EnterWorktree now asks for confirmation before entering a git worktree outside the project's .claude/worktrees/ directory` — `EnterWorktree` is a harness-level SDK tool (see the doc comment at `lib/server/session.ts:127-131`: "Harness-level `EnterWorktree` moves the session into a git worktree WITHOUT firing..." — already a known, named SDK tool call site). The confirmation behavior lives inside that tool's own implementation in the SDK/CLI; Claudius already renders any tool call (including its confirmation prompt) through the generic permission-request pipeline, so this arrives automatically whenever the SDK exposes the updated tool — no Claudius-side code to write. (Not "already shipped via the SDK migration on this branch" — the 0.3.205→0.3.206 SDK bump was a confirmed no-op per `.claudius/sdk-updater/run-notes/0.3.206.md`; this is "arrives whenever a *future* SDK bump exposes the updated tool," which is the ordinary bucket-A story, not something already landed on this branch.)
- `[C — CLI/terminal only] Background agents now upgrade to a new version in the background right after a Claude Code update, instead of paying a slow stale-session upgrade when you attach` — about the CC CLI binary's own self-update + background-daemon (`claude --bg` / `claude agents`) process-upgrade mechanics. Claudius has no local CLI binary to self-update and no background daemon architecture analogous to CC's `--bg` jobs; Claudius's own background tasks are the SDK's in-process background Bash tool, unaffected by this.
- `[skip — not a feature] Improved /code-review findings quality on claude-opus-4-8 across all effort levels` — an upstream model/prompt-quality improvement to a bundled skill, not a discrete capability with a call site to reimplement or a UI surface to add. Claudius's own `code-review` skill already calls through to whichever model the session uses; nothing to change.
- `[C — CLI/terminal only] Improved agents view: status column now uses full terminal width instead of truncating at 64 characters` — CC's `claude agents` TUI table layout. Claudius's session list is an HTML/CSS layout with no terminal-column-width constraint to work around.
- `[C — CLI/terminal only] Changed agents view: Ctrl+X now permanently removes a completed session, and sessions no longer render twice; deleted background jobs stay deleted` — same TUI (`claude agents`) — a key-chord + a rendering bug fix in a terminal list Claudius doesn't have (Claudius session deletion is already a single non-duplicated action in the Sessions UI).

Pure bug-fix entries (`Fixed …`) elided per the classification instructions — none of them describe a capability Claudius doesn't already have correct.

## Implemented (bucket B)

- `[extend: /doctor] "Checked-in CLAUDE.md is too big" check` — `app/api/doctor/route.ts` gains a `claudeMdSizeChecks()` pass over every `kind !== "customization"` workspace (`listWorkspaces()`), reading the `project` and `project-claude` CLAUDE.md scopes via `lib/server/claudemd.ts#readScope` and warning (with a line/KB count, using a trailing-newline-safe line counter) when the combined checked-in content crosses 300 lines. `app/doctor/page.tsx` renders the new `Check.link` field as a button that opens that workspace's `/[workspaceId]/memory` page — reusing Claudius's existing per-workspace memory editor instead of building new UI for the "go trim it" action. Files: `app/api/doctor/route.ts`, `app/doctor/page.tsx`. Rejected alternative: routing the "Fix" action to an LLM call inside the API route that actually proposes trims (mirroring upstream's model-driven analysis) — rejected because every existing `/doctor` check in Claudius is a fast, deterministic, session-less probe (no model call), and adding one that spins up an agent turn from a GET-triggered diagnostic page would be a first (and a cost/latency surprise); sending the user into the existing chat/Memory surface to do the actual (necessarily judgment-based) trimming fits Claudius's established pattern better.

  sub-list UI or picking one workspace arbitrarily, and (b) in practice
  most Claudius installs manage a handful of workspaces, so the
  amber-row count is expected to stay small. Rejected alternative:
  a single aggregated "M of N workspaces have oversized CLAUDE.md"
  row linking to a new small listing view — deferred as unnecessary
  complexity until real usage shows the per-workspace list gets noisy.
