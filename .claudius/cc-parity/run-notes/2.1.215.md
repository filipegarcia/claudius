## Summary

This combined-mode run follows the 0.3.214→0.3.215 SDK bump on the
same branch, which was a **zero-code no-op** (verified byte-for-byte
in `.claudius/sdk-updater/run-notes/0.3.215.md` — every `.d.ts` the
SDK ships is identical, the only real delta is a native-binary
rebuild). That means no changelog entry here was "already shipped" by
that migration; every engine-side item below is a plain
`[A — already via SDK updater]`. Claude Code 2.1.212–2.1.215's
substantive product-side additions are almost entirely engine/CLI/
terminal internals (permission-check hardening, OTel plumbing,
statusline payloads, background-daemon lifecycle fixes). Two genuine
bucket-B gaps survived the audit and are implemented this release:
(1) three engine-side session limits (`CLAUDE_CODE_MAX_WEB_SEARCHES_PER_SESSION`,
`CLAUDE_CODE_MAX_SUBAGENTS_PER_SESSION`, `CLAUDE_CODE_MCP_AUTO_BACKGROUND_MS`)
now get a Claudius Settings UI that maps to real env vars the spawned
SDK subprocess reads (confirmed via `bridge.mjs`'s env-var schema
registry, not dead code); (2) a "needs input" session status,
generalizing CC's sandbox/MCP-input/managed-settings distinction to
Claudius's own `pendingPermissions`/`pendingAskQuestions`/`pendingPlans`
state, closing a real gap where a background session blocked on a
permission prompt was visually indistinguishable from one actively
running. The headline risk to flag for review: the needs-input status
is a UX-idiom translation (CC's terminal `claude agents --json` status
label → a browser tab status dot), not a literal port, so the reviewer
should sanity-check the dot's precedence rules (turn-in-flight vs.
pending-permission) match user expectations. Two changelog items were
deliberately *not* built despite initially looking bucket-B
(`claude auto-mode reset`, `/resume` picker over deleted sessions) —
see "## Risks / follow-ups" for why.

## Risks / follow-ups

- **`claude auto-mode reset` — considered and rejected as bucket B.**
  CC's command resets an *accumulating* auto-approve configuration.
  Claudius represents permission behavior as a live mode selector plus
  a single `disableAutoMode: "disable"` escape-hatch flag — there is no
  accumulating config to reset, so a bespoke "Reset to default +
  confirm" button around a one-field clear would be exactly the
  pasted-in-CLI affordance the adversarial UX check exists to catch.
  Maximal alternative considered: add an explicit "Reset" button next
  to `disableAutoMode` in Settings. Rejected because clearing the field
  already *is* the reset action today with no extra plumbing needed.
- **`/fork` → background session / `/subtask`.** Conservative reading:
  Claudius's `/fork` already matches the new CC default (produces a
  full session, not a subagent), so nothing to change. Maximal
  alternative: add a `/subtask` native slash command that explicitly
  launches an in-session subagent on typed command. Rejected — in
  Claudius, subagent invocation is assistant-initiated (the `Task`
  tool), not something a user types as a command; inventing a
  user-typed subagent-launch affordance isn't grounded in an existing
  Claudius interaction pattern.
- **`/resume` picker over deleted sessions — deferred, not built.**
  Requires a soft-delete/trash model Claudius doesn't have
  (`sessions-store.ts` `remove()` is permanent). Building that is a
  materially larger feature than this single changelog line
  justifies; flagging for a dedicated follow-up rather than bolting a
  half-built trash concept onto this release.
- **Memory file frontmatter `modified` timestamp — marked ambiguous.**
  Could not confirm from the changelog text or a quick look at
  Claudius's `auto-memory.ts` whether this maps to anything Claudius
  owns (Claudius's memory system is a from-scratch reimplementation,
  not a consumer of CC's built-in memory tool). Did not `gh api`/
  `WebFetch` further given the low likely payoff; a human with more
  context on whether Claudius's auto-memory should also gain a
  `modified` field can confirm.
- Session limits settings: conservative shape is a flat numeric field
  per limit with no unit conversion; the maximal alternative (a
  toggle + human-friendly duration picker for the MCP-auto-background
  threshold, e.g. "2 minutes" instead of raw milliseconds) was
  rejected as unnecessary polish for an advanced/rarely-touched
  settings group — matches how Claudius's existing numeric
  `SDK_SETTINGS_CATALOG` entries are presented (plain number inputs).
- Needs-input status: precedence between "turn in flight" and
  "pending permission" needed a judgment call — a permission prompt
  can only exist while the SDK is mid-turn, so needs-input is checked
  and takes precedence over the generic running state wherever a
  pending prompt exists. Reviewer should confirm this matches
  expectations for the (rare) case of a permission prompt outliving
  its turn.
