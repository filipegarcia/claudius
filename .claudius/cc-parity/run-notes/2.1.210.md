## Summary

2.1.210 is a small release: most of its substance is engine/CLI-internal
(Agent-tool hardening, auto mode's permission classifier, the bundled
dataviz skill, and — per this run's combined-mode note — the
Bash/PowerShell auto-background timeout message, which the sibling SDK
0.3.208→0.3.210 bump already shipped on this branch) or terminal-only
(the `claude agents` view's footer "waiting on input" hint and its `←`
list-navigation selection state). Four items are genuine product-surface
gaps Claudius
reimplements this release: (1) the chat transcript's collapsed tool-call
row has no live elapsed-time indicator, unlike the Activity rail's
Task/Bash rows which already tick at 1Hz; (2) the Permissions page lets
users type `Write(path)` / `NotebookEdit(path)` / `Glob(path)` rules that
Claude Code itself now flags as unsupported path-scoped forms, with zero
in-app feedback; (3) Claudius's own `MEMORY.md` index writer
(`lib/server/auto-memory.ts` — confirmed independent of the SDK, which
ships no memory-tool code) appends forever with no size guard, so a
degenerate write today fails silently rather than erroring; (4) the
existing Shift+Tab permission-mode cycle has no assistive-tech
announcement when the mode changes. All four are minimal, additive
changes to screens Claudius already has. The headline risk to flag for
review is #4's shape: Claude Code gates this behind an opt-in
"screen reader mode" setting, and this run chose to always announce via
an `aria-live` region instead (see Risks) — a human should confirm that
default-on read is the right call for Claudius rather than porting the
opt-in toggle literally.

## Risks / follow-ups

- **Screen-reader announcement shape (item 4).** Chose an always-on
  `aria-live` region over porting Claude Code's opt-in "screen reader
  mode" toggle. Reasoning: ARIA live regions are inert for sighted users
  and browsers/AT already let users control verbosity, so gating this
  behind a Claudius-specific setting only adds a discovery step for the
  assistive-tech users who need it. A human should confirm this
  default-on choice rather than the more literal toggle-gated port.
- **Agents footer "waiting on input" count (reclassified C mid-review).**
  This was first drafted as `[skip — ambiguous]` with a floated
  alternative (a badge on the "Agents" SideNav item mirroring the
  existing unread-notifications badge pattern, fed by a new aggregate
  query over live sessions' permission-pending state). Reclassified to
  `[C — CLI/terminal only]` for consistency with the immediately
  preceding `←` list-navigation item — both are `claude agents` TUI
  footer/chrome, not separable features, and the underlying need
  ("tell me when a session is waiting on me") is already served in
  Claudius by the notifications system and the 2.1.207
  blocked-session-peek work. Recording the originally-floated SideNav
  badge here in case a future release makes the case for it more
  concretely (e.g. if Claude Code's `claude agents --json` `waitingFor`
  field shape gets a closer Claudius analogue).
- **Working branch.** This run's harness prompt describes a
  "combined-mode" setup where the SDK migration and this cc-parity pass
  share one branch, but the branch actually checked out in this working
  tree is `sdk-update/0.3.210` (the SDK updater's branch), not a
  `cc-parity/2.1.210` branch — no such branch exists locally or on
  `origin` (`git ls-remote origin cc-parity/2.1.210` is empty). Per the
  prompt's own combined-mode framing ("SDK migration ... has just been
  completed on this same branch"), this run committed its work directly
  onto `sdk-update/0.3.210` rather than creating a separate
  `cc-parity/2.1.210` branch. A human should rename/re-target the
  branch before merge if the orchestrator expects the latter name.

  still running — low CPU, likely blocked on `gh` API calls per this
