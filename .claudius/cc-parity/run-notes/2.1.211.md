## Summary

2.1.211 is a bug-fix-heavy release (Vertex/Bedrock/Windows/Claude-in-Chrome
crashes, credential/daemon respawn races, terminal rendering) with very
little new product surface, and almost none of what remains is something
Claudius reimplements independently: most substantive entries are either
already covered by the SDK the moment it exposes them (subagent model
overrides, background-task result reporting, prompt-caching billing,
rules-loading scope, permission-rule persistence, env-var integer parsing)
or are CLI/terminal/Chrome-extension-only (stream-json's
`--forward-subagent-text`, terminal rendering perf, Vim-mode key bindings,
the VSCode Remote Control banner, Claude in Chrome file-upload hardening).
The one bucket-B item this release is narrow but real: CC's fix to "measure
only loaded content, excluding frontmatter and HTML comments" when warning
that a memory index is over its read limit is a direct evolution of
Claudius's own `MEMORY_INDEX_READ_LIMIT_BYTES` guard in
`lib/server/auto-memory.ts` — itself a from-scratch Claudius
reimplementation built for 2.1.210 parity, since the SDK ships no
memory-tool logic at all. Headline risk to flag for review: this is a
backend-only measurement refinement with no new UI, so the "is this the
right shape" question is really "is the conservative frontmatter/HTML-comment
stripping regex the right proxy for 'loaded content'" — see Risks below.

## Risks / follow-ups

- **Frontmatter/HTML-comment stripping regex is a conservative proxy.**
  `measureMemoryIndexLoadedBytes()` strips only a *leading* `---\n…\n---\n`
  block (not frontmatter anywhere mid-file) and `<!-- … -->` comments
  anywhere. The conservative reading was: MEMORY.md's own doc comment says
  it's "a lightweight one-line-per-entry pointer index" with no frontmatter
  of its own, so a leading-block-only strip covers the realistic case (a
  user hand-prepending a note block) without trying to parse the pointer
  lines themselves for embedded HTML comments in a way that could
  misfire on a legitimate `<!--` inside a description. The maximal
  alternative — treating any `---`-delimited block anywhere in the file as
  frontmatter — was rejected because MEMORY.md's own generated lines start
  with `- [`, never `---`, so a mid-file `---` is far more likely to be
  literal content (e.g. a description containing three dashes) than a
  frontmatter block, and stripping it would risk under-counting real
  content.
- **`[A — already via SDK updater]` calls for "background agent result
  reporting" and "always allow permission rules" are the two softest
  calls in the classification.** Both rest on "Claudius has no hand-rolled
  version of this today, so there's nothing to fix" rather than positive
  confirmation of exactly how the SDK will expose the upstream fix. If a
  future SDK bump's run-notes show these landing with a different shape
  than expected (e.g. always-allow rules requiring an explicit Claudius-side
  UI to surface, not just pass-through), that's new information, not a
  contradiction of this call — flagging so the next SDK-parity run checks
  for it explicitly.
- **No new UI this release** means there's no adversarial UX-fit
  verification to speak of — the only design call was the measurement
  regex above.
