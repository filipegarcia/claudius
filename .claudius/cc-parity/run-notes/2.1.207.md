## Summary

Claude Code 2.1.207 is a mostly bug-fix release (terminal streaming
freezes, worktree config leaks, background-session naming, Remote
Control reliability, Bedrock credential churn) — none of that
reimplements in Claudius since it either isn't a feature Claudius
has (Remote Control, agent-teams mailbox) or is already SDK-side
behavior that lands automatically via the sibling SDK-update
pipeline. Two items are genuine product-surface additions: (1)
`disableAutoMode`, a new settings escape hatch for the "Auto"
permission mode (which Claudius already ships end-to-end as a
first-class `PermissionMode`) now that upstream auto-enables Auto
mode on Bedrock/Vertex/Foundry without an opt-in; and (2) an "agent
view" polish where a still-pending blocked-session notification now
leads with the actual question/tool text instead of a generic label,
plus a worded "waiting Xm" staleness clock. Both are implemented this
release (see below). The headline risk to flag for review: item (2)
is a UX call about how prominently to surface pending permission
requests in the cross-workspace NotificationsDrawer — the alternative
of leaving the generic-title-first layout untouched (and only adding
the clock wording) was considered and rejected; see Risks. Getting a
clean full-suite `test:e2e` gate also required an incidental one-line
fix to a pre-existing 2.1.204 spec that was leaking a fake session id
into shared test state (see Risks); a separate, unrelated pre-existing
flake in the notifications test files was investigated, confirmed via
`git stash` to have zero relation to this release, and left alone.

## Implemented (bucket B)

- `[extend: Settings (Permissions section) + StatusLine's ModeSelector]` **`disableAutoMode` settings escape hatch.** Added `disableAutoMode?: "disable"` to `ClaudeSettings` (`lib/server/settings.ts`), matching the SDK's own single-literal schema exactly rather than approximating it as a boolean. Enforced server-side in `Session.setPermissionMode` (`lib/server/session.ts`) — a client requesting `"auto"` is coerced back to `"default"` when the user-scope setting is set, regardless of what the client sent. Surfaced client-side via a new `useDisableAutoMode` hook (`lib/client/useDisableAutoMode.ts`) that hides "Auto" from the `ModeSelector` dropdown and the Shift+Tab cycle (`components/chat/ModeSelector.tsx`, `components/chat/ChatSurface.tsx`, `components/chat/StatusLine.tsx`); the pure cycling math moved to `lib/shared/permission-modes.ts` so it's vitest-testable. Persisted through the existing generic "SDK settings catalog" mechanism on `/settings` (new `Permissions` section entry, `app/settings/page.tsx`) rather than a bespoke toggle row — this is the same mechanism already used for `skipDangerousModePermissionPrompt` and a dozen other direct SDK-settings passthroughs. Rejected alternative: a bespoke boolean `ToggleRow` (like `sessionRecapEnabled`) — rejected because the SDK's own schema stores this as a single-literal flag (`"disable"`, not `true`/`false`), and using the existing enum-typed catalog row keeps the on-disk representation byte-compatible with what a real Claude Code CLI sharing the same `~/.claude/settings.json` would write.
- `[extend: NotificationsDrawer (right-rail Activity panel)]` **Blocked-session peek leads with the question.** For notification rows whose kind the agent is actually blocked on (`permission_request` / `ask_user_question` / `plan_approval_request`, via the existing `isActionableKind` helper) and that are still unread, `NotificationRowItem` (`components/nav/NotificationsDrawer.tsx`) now promotes the tool/question preview text to the bold primary line, demotes the generic kind label ("Claude needs permission") to the secondary line, and replaces the plain "Xm ago" timestamp with a "waiting Xm" staleness clock (new `formatWaiting` helper). Once the row is read it reverts to the ordinary title-first layout like every other notification kind. Rejected alternative: changing the underlying `title`/`body` fields produced by `mapEventToKind` in `lib/server/notification-bus.ts` — rejected because those strings are also used for the OS-level `Notification(title, {body})` popup, where the generic label-first phrasing ("Claude needs permission" / detail below) is the more conventional OS-notification shape; the swap belongs only in the drawer's own row rendering, not the persisted/broadcast data.

## Risks / follow-ups

- **`disableAutoMode` UX shape** — considered gating "Auto" via a bespoke boolean toggle in the existing "Chat" settings section (next to `promptSuggestionEnabled`/`sessionRecapEnabled`) instead of the generic SDK-settings catalog's "Permissions" section. Went with the catalog because the SDK's schema stores this as a single-literal flag (`"disable"`), which the catalog's `enum` field type represents byte-for-byte; a boolean toggle would have forced picking an arbitrary `true`/`false` on-disk representation that a real Claude Code CLI sharing the same settings file wouldn't recognize. Reviewer should sanity-check that "Permissions" (not "Chat") is the right home for a mode-picker-facing setting.
- **Server-side enforcement is the authority, not just UI hiding** — `Session.setPermissionMode` coerces a requested `"auto"` back to `"default"` whenever `disableAutoMode` is set, independent of the client. This means if a user has an existing session already sitting in "auto" mode when an admin flips the setting, that session is NOT force-exited retroactively — it simply can't be re-entered once left. The more aggressive alternative (force-exiting live sessions the moment the setting changes) was considered and rejected as more invasive than the conservative "gate future entry" reading of the changelog line, which says nothing about retroactive session state.
- **Blocked-session peek UX** — the alternative of leaving `title`/`body` semantics untouched and only adding the "waiting Xm" clock (skipping the question-first promotion) was considered and rejected: the upstream changelog explicitly calls out "lead with the question" as the change, and Claudius's existing rows already carry the question text in `body` — promoting it is a presentation-only change with no new data plumbing required. Reviewer should confirm the swapped visual hierarchy (question bold/primary, generic kind label secondary) reads well against the OS-notification convention, which intentionally keeps the opposite ordering.
- `[skip — ambiguous, see Risks]` **agent-teams mailbox crash loop** — couldn't confirm from the changelog line alone (and didn't find an "agent teams" / "mailbox" concept anywhere under `lib/server/`) whether this maps to any Claudius internal; treated as out of scope rather than guessing at a reimplementation.
- **Incidental fix: `cc-parity-2.1.204-token-expiring-notice.spec.ts` was polluting shared test state.** That spec mounts a fake `SESSION_ID` via `page.goto("/?session=cc204-...")` but never mocked `PUT /api/sessions/open-tabs`, so the fake id got written for real into the workspace's persisted per-cwd `.claudius.db` tab list. In a full-suite run, whichever spec ran next inherited that fake tab as the "last active" one, and its own `page.goto("/") + waitForURL(session-uuid-regex)` would then hang against the non-UUID fake id until timeout. This broke exactly the three new `cc-parity-2.1.207-*` tests (alphabetically right after `2.1.204`) when run as part of the full suite, even though each passed cleanly in isolation. Fixed by stubbing that PUT route in the 2.1.204 spec's `beforeEach` (one-line addition, same pattern as its other route mocks). Pre-existing bug, unrelated to any 2.1.207 changelog entry — fixing it was necessary to get a clean full-suite gate.
- **Pre-existing flake, confirmed unrelated to this release: `notifications.spec.ts`'s "R1" test and 3 tests in `notifications-multi-tab.spec.ts` fail intermittently in full-suite runs** with notification-row counts that don't converge (e.g. expecting 3 unread rows, observing 9) or a `waitForURL`/open-tabs poll timing out. Verified this is NOT caused by anything in this release: `git stash`-ing every change in this diff (leaving only the new untracked files, which weren't part of the targeted run) and re-running just these two spec files reproduced the same failures. Re-running the identical command again (still with everything stashed) sometimes passed and sometimes didn't — confirming it's a genuine timing race in the shared-fixture cross-test notification convergence, not a deterministic bug, and not something introduced here. `git log` shows a prior dedicated fix for this exact flake class (`fac8819 test(e2e): fix pre-existing flake in notifications-multi-tab.spec.ts`), so this is a known-recurring issue in that test file, not new. Did not attempt a further fix — it's orthogonal to every 2.1.207 changelog entry and deserves its own investigation rather than a bolted-on timing patch here.
- Pre-existing environment note (not a cc-parity issue): this machine's `better-sqlite3` native binding needed `node scripts/native-abi.mjs ensure node` to run once before `vitest run` would pass (`bun run test` includes this step and is green; a bare `bunx vitest run` is not). Not a code issue — flagging only in case a fresh CI runner hits the same first-run gap.
