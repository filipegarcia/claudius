## Summary

2.1.216 is a mostly-maintenance release: one settings.json addition
(`sandbox.filesystem.disabled`) and ~50 bug fixes, almost all of them
tied to CLI-only surfaces (terminal rendering, daemon/background-agent
lifecycle, key chords, PowerShell/Windows quirks) that have no Claudius
analog. Two changelog lines described behavior Claudius was *not*
actually at parity on despite already having the surrounding feature
(`/context` exceeding the window, a failed `/compact`), so this release
also closes those two small gaps. Bucket B this release: (1) a new
"Skip filesystem isolation" checkbox nested under the existing Sandbox
toggle in `WorkspaceForm.tsx`, (2) an explicit "Context window exceeded"
banner state (shown unconditionally, even past the user's "Never nudge
me" threshold pref), and (3) a visible "Compaction failed: …" error
when a `/compact` turn ends without producing a `compact_boundary`.
Headline risk to flag for review: for (3), Claudius infers "the compact
failed" from the *absence* of a `compact_boundary` on the very next
result rather than from an SDK-native "this was a compaction turn and it
failed" signal — the SDK doesn't expose the latter, so this is a
best-effort heuristic (see Risks) that could theoretically misfire if a
`/compact` command is ever queued behind another turn in a way that
breaks the "one turn in flight at a time" assumption it leans on.

## Risks / follow-ups

- **`[B] sandbox.filesystem.disabled`**: alternative considered — exposing it as a raw `SDK_SETTINGS_CATALOG` entry (the generic settings.json passthrough pattern used for flatter booleans like `promptSuggestionEnabled`) instead of a bespoke workspace-form checkbox. Rejected because Claudius's existing sandbox toggle (`sandboxEnabled`) is *already* a bespoke per-workspace default, not a catalog entry — nesting the new leaf under the catalog would have split one logical feature (sandbox config) across two different settings UIs (the Workspace form and the generic Settings page), which would be more confusing than consistent with the precedent that already exists.
- **`[B] Context window exceeded`**: the ≥100% override in `shouldShowContextWarning` intentionally overrides the user's "Never" (threshold=100) preference. This is a deliberate product call (an "exceeded" state is qualitatively different from "approaching" and arguably always worth surfacing), but it does mean a user who explicitly opted out of *all* context nudges will still see this one — flagging for reviewer sign-off in case the "Never" preference was meant to be absolute.
- **`[B] Compaction failed`**: as noted in the Summary, the failure signal is inferred from "no `compact_boundary` landed before the next `result`", not from an SDK-native "this turn was a compaction and it failed" field (no such field exists in the installed SDK's `SDKResultError`/`SDKResultSuccess` types). This heuristic relies on Claudius's existing invariant that only one turn is ever in flight at a time (enforced server-side via `turnInFlight`). If that invariant is ever relaxed (e.g. a future feature allows a `/compact` to be explicitly queued behind other input in a way that reorders results), this detection could misattribute a failure. Worth a follow-up SDK-message-handler review if `Options`/result shapes ever add an explicit compaction-outcome field.
- **`[skip — ambiguous] Spend limit adjustment prompt`**: disambiguation attempt was reading Claudius's own cost/budget code (`components/cost/`, `LimitsPanel.tsx`) — found only local per-workspace/session USD caps (`maxBudgetUsd`), no org/admin spend-limit-policy concept at all. The changelog line implies a server-enforced org policy rejecting a user's requested limit increase, which has no Claudius equivalent to extend. Flagging in case a future release clarifies this is meant to apply to Claudius's own budget caps (e.g. a workspace cap increase getting rejected by some policy) — nothing like that exists today, so there was nothing to extend.
- **`[C] Background sessions "needs input" in agent view`**: the maximal alternative considered was building a persistent, cross-session "needs input" badge on the sessions list (`app/[workspaceId]/sessions/page.tsx`), modeled on the existing `McpNeedsAuthNoticeEvent` → `use-session.ts` → `systemEntries` pipeline but promoted to a sessions-list-level indicator. Rejected as out of scope for this release: Claudius's session lifecycle reaps a session shortly after the last subscriber disconnects (no indefinitely-running "detached" sessions the way Claude Code's daemon model has), so the underlying "a background agent needs your attention while you're not looking" scenario this CC feature addresses doesn't really arise in Claudius's architecture today. Worth reconsidering if Claudius ever grows a true long-lived detached-session concept.
- **Pre-existing gap, out of scope**: `disableBundledSkills` (from an earlier CC release, tracked in `lib/server/claude-code-changelog.ts:43`) is still not exposed anywhere in Claudius (confirmed absent from `SDK_SETTINGS_CATALOG`). Not part of this release's changelog, not touched here, but noted since this release's dataviz-skill-update item brushes the same "bundled skills" area.
- **`.claude` symlink-write safety**: the workflow/scheduled-task symlink fix (elided above as CLI-only/not-applicable) is a reminder that any *future* Claudius feature writing user-named files under a `.claude`-style directory from request-controlled input should follow the existing `CLAUDE.md` path-safety conventions (`path.resolve` + inline `startsWith` check at the sink) — no such feature exists today, so no action was needed this release.
