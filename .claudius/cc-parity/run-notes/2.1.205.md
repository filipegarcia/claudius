## Summary

2.1.205 is a mostly bug-fix / CLI-internals release. Of the substantive
(non-bug-fix) entries, two are auto-mode safety heuristics and one is
notification copy — all engine-composed and delivered to Claudius
automatically once the SDK updater bumps `@anthropic-ai/claude-agent-sdk`
(bucket A). Three entries (`claude attach`, `claude mcp
add-from-claude-desktop`, the auto-updater's streaming download) are
CLI/binary-only and don't translate to a browser app (bucket C).
Two entries ("agent view" PR linking; colored state word + classifier
headline + peek) describe `claude agents`, which upstream docs confirm
is an **interactive terminal UI screen** ("opens the agent view") for
monitoring background CLI sessions — also bucket C, not a rail widget.
That leaves exactly **one** bucket-B item this release: `/doctor` is
now "a full setup checkup that can diagnose and fix issues" with a
`/checkup` alias. Claudius already ships an analogous `/doctor` slash
command and `/doctor` page — this release extends both with the
`/checkup` alias and real auto-fix actions for the two checks that are
actually safely fixable from the running Next.js process (creating
`~/.claude` and `~/.claude/projects` when missing). Headline risk to
flag for review: whether a single-item bucket-B release with a narrow
Doctor extension is the right call, versus treating the `claude
agents` entries as a UI feature worth prototyping anyway — see Risks.

## Changelog classification

- `[A — already via SDK updater] Added an auto mode rule that blocks tampering with session transcript files` — this is an engine-side auto-mode safety heuristic. Claudius has zero bash/file-safety heuristics of its own (confirmed by grep over `lib/server`); it forwards `permissionMode: "auto"` straight to `query()` and the SDK/CLI engine decides what to allow before `canUseTool` is even invoked. Arrives automatically with the SDK bump.
- `[A — already via SDK updater] Improved auto mode to ask before running rm -rf on a variable it can't resolve from context` — same reasoning: an engine-side auto-mode heuristic, not something Claudius implements or could implement without duplicating the CLI's own command-safety classifier.
- `[A — already via SDK updater] Background task notifications now explicitly state that no human input has occurred, preventing fabricated in-transcript approvals from being acted on` — the notification text is composed by the engine and delivered as a `task_notification`/transcript message; Claudius only relays and renders whatever text arrives (confirmed in `lib/server/session.ts`'s `captureTaskState`). No Claudius-side copy to change.
- `[C — CLI/terminal only] Improved agent view: sessions that edit, merge, comment on, or push to an existing PR now link it in claude agents` — `claude agents` is an interactive **terminal UI screen** ("opens the agent view to monitor and dispatch parallel background sessions", per `code.claude.com/docs/en/cli-reference`), not a data feed the SDK exposes (the SDK's public types have no PR-link field — checked `node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts`, only an unrelated `prUrlTemplate`). Confirmed there is no Claudius analog to bolt this onto: Claudius's own "Agents" nav item (`app/agents`, `app/[workspaceId]/agents/page.tsx`) is a static **editor for agent-definition files** (`.claude/agents/*.md` / `~/.claude/agents/*.md` — name/model/tools/frontmatter), not a live session monitor, and it shows no per-row status of any kind. There is no cross-session "fleet" monitor anywhere in Claudius today — "FleetView" only exists as an unimplemented idea name in unrelated SDK-updater run-notes, not a real screen. `claude agents`' PR-linking is dashboard chrome for *whole background CLI sessions*, a domain Claudius has no surface for at all (not even an unstyled one), so there is nothing to extend here.
- `[C — CLI/terminal only] Improved agent view: rows now show a colored state word and a classifier-written headline instead of raw tool call text, and the peek opens with full status including the exact ask for blocked sessions` — same `claude agents` TUI screen. Verified Claudius's only comparable surface, `BackgroundTasksPanel`'s Tasks section, is scoped to Task-tool **subagents inside one active session** (driven by `task_started`/`task_notification`/`background_tasks_changed`), not sibling background *sessions* — a different, narrower domain than `claude agents`. Where that in-session surface does apply, the substance described is already present: it shows a colored-tone row (`TASK_TONES`) and a classifier-written live summary (`t.summary`, from SDK `agentProgressSummaries`) in place of raw tool text. The "peek with the exact ask for blocked sessions" part needs a `needs_input`/`blocked` state Claudius's `TaskStatus` doesn't model (`pending | running | completed | failed | killed | stopped`) and there's no cross-session registry to attach one to; inventing both a new status and a new session-monitor screen just to backfill one peek modal would be building a feature the changelog entry doesn't actually ask Claudius for. See Risks.
- `[C — CLI/terminal only] Fixed claude attach erroring when a background agent was mid-upgrade restart instead of waiting for it to come back` — `claude attach` reattaches a terminal to a background CLI session; no terminal in a browser app.
- `[C — CLI/terminal only] Fixed claude mcp add-from-claude-desktop getting stuck when a server name contains unsupported characters` — a CLI import command from Claude Desktop's local config; Claudius has its own MCP UI (`/mcp`) with its own add flow, unaffected.
- `[C — CLI/terminal only] Auto-update binary downloads now stream to disk instead of buffering in memory` — the `claude` CLI binary's self-updater. Claudius has its own separate update/build pipeline (out of scope of this parity pass per the hard constraints on `scripts/sdk-update/` / `scripts/cc-parity/`); nothing to port.
- `[skip — ambiguous, see Risks] Reserved the "Claude Browser" MCP server name (alongside "Claude Preview") ahead of the Claude Desktop pane rename; user-configured MCP servers can no longer register under either name` — this is Anthropic namespacing its own upcoming Desktop feature inside the CLI's *own* MCP registry, not a capability change a user would ask Claudius for. Claudius has its own separate MCP server registry (`/mcp`, `lib/server/mcp.ts`) with no "Claude Browser"/"Claude Preview" collision risk today. No docs page describes user-facing behavior to mirror. See Risks for the maximal reading considered.
- `[B — reimplement in Claudius] /doctor is now a full setup checkup that can diagnose and fix issues; /checkup is its alias` — Claudius already reimplements `/doctor` as a native slash command (`lib/shared/slash-commands.ts`) plus a full `/doctor` page (`app/doctor/page.tsx`, `app/api/doctor/route.ts`) with pass/warn/fail checks. This release: (1) adds the `/checkup` alias to the existing command registry, and (2) adds real "fix" actions for the checks Claudius can safely auto-remediate from the running server process (creating `~/.claude` and `~/.claude/projects` when missing/unwritable) — matching upstream's "diagnose **and fix**" framing rather than diagnose-only.
- Bug fixes elided per the prompt's instructions: `--json-schema` schema validation, mid-turn message loss at `--max-turns`, Windows worktree/NTFS-junction deletion, background-agent-list staleness after `SendMessage` resume, "needs input" flipping back to "working", PR-linking missing output >30K, plugin LSP server blocking another plugin's LSP, Windows crash on a deleted/locked/unmounted launch dir, crash on file-watcher-closed-mid-scan, project-verify-skills rewritten every session, agent-view one-line clipping, stale "Running" status in the hosted web/mobile Remote Control panels (Anthropic's own hosted service, not Claudius), Cowork VM-mode login failure.

## Risks / follow-ups

- **Doctor fix scope is the conservative reading.** Upstream's "can diagnose and fix issues" is unspecified in scope — the maximal interpretation would auto-fix everything up to and including running `npm install` for a missing SDK package or shelling out to `git`/auth flows. We deliberately fixed only the two checks that are pure, safe, local filesystem operations against fixed, non-user-controlled paths (`homedir()/.claude`, `homedir()/.claude/projects`). Auth/package-install fixes are left as manual guidance text (unchanged) since auto-running installs or writing credentials from a "Fix" button in a web app is a materially different trust boundary than a local CLI prompt.
- **The two `claude agents` entries were classified C, not B**, after confirming via `code.claude.com/docs/en/cli-reference` that `claude agents` "opens the **interactive agent view**" — a terminal UI screen for background *CLI sessions*, not a Claudius data surface. We explicitly checked for a Claudius analog before ruling this out (this was re-verified with a dedicated read-only audit after an initial pass): Claudius's "Agents" nav item is an agent-*definition* file editor (`app/[workspaceId]/agents/page.tsx`), not a live monitor, and "FleetView" — the one name that sounded like a candidate — turned out to be an unimplemented idea mentioned only in unrelated SDK-updater notes, not a shipped screen. The rejected alternative: bolt PR-link detection and a "peek" modal onto `BackgroundTasksPanel`'s Tasks section (which tracks in-session Task-tool subagents, a different domain — sibling background *sessions* have no rail representation in Claudius at all today, definition-editor included). We did not build this because (a) the PR-linking signal isn't in the SDK's public types, so Claudius would have to invent its own Bash-output PR-URL scraper with no upstream contract to match against, and (b) the "peek for blocked sessions" half requires a `needs_input`/`blocked` `TaskStatus` value Claudius doesn't have, plus a cross-session registry that doesn't exist at all — adding both to backfill a single peek modal would be scope invention well beyond what this changelog entry supports. If a future release wants this, the right target is a new session-level (not task-level, not definition-editor) surface, since "sessions ... push to an existing PR" is about whole background sessions — a genuinely new domain, not an extension of either existing Agents-adjacent screen.
- **`[skip — ambiguous]` MCP name reservation** — no docs page or `gh api` commit lookup surfaced a user-facing behavior change for Claudius to mirror (searched `code.claude.com/docs/en/cli-reference`, `code.claude.com/docs/en/background-tasks`, `code.claude.com/docs/en/claude-code-on-the-web`; `gh api repos/anthropics/claude-code/commits` 404s on this path/sha combination for this checkout). The maximal reading — reserving "Claude Browser"/"Claude Preview" as forbidden names in Claudius's own `/mcp` server registry — was rejected because Claudius has no matching first-party feature under either name; reserving the strings would just be defensive naming with no concrete collision to prevent.
- **Pre-existing e2e flake, unrelated to this release:** `tests/e2e/session-management.spec.ts:30` ("Chat resumes the last-active session, refresh keeps it, naming persists") failed once under a full-suite run (`getByTestId('recap-title-input')` not visible within 15s) but passed cleanly when re-run in isolation immediately after. This diff touches only `lib/shared/slash-commands.ts`, `app/api/doctor/**`, and `app/doctor/page.tsx` — nothing in session naming/recap/SideNav — so this is environmental flakiness under full-suite resource contention, not a regression introduced here. Not fixed as part of this PR per the hard-constraint guidance to document rather than edit unrelated tests; flagging so a flaky re-run at the orchestrator's gate isn't mistaken for a real break.
