## Summary

Claude Code 2.1.191 ships one user-visible product feature, six SDK-automatic
improvements, nine CLI/terminal-only fixes, and one internal performance
optimisation that does not transfer to Claudius's architecture. The only
Bucket-B item is **"/rewind after /clear"**: when the user runs `/clear` (or
clicks the header Clear button), Claudius now records the cleared-from session
ID in `sessionStorage` and surfaces a yellow **ClearedFromBanner** at the top
of the fresh empty session. Clicking its Rewind button (or typing `/rewind` in
the slash bar) navigates back to the previous conversation. The banner auto-hides
once the user sends their first message in the new session. The implementation
avoids any new migrations (no DB column added) and is fully tab-scoped so it
survives page refresh within the same browser tab.

The CPU-coalescing optimisation (100 ms delta batching) was considered for
Bucket B but deferred — see Risks / follow-ups.

## Implemented (bucket B)

- **[extend: chat session] `/rewind` after `/clear`** — `ClearedFromBanner`
  shown at the top of a freshly-cleared empty session so users can navigate
  back to the conversation they just cleared.
  - Files touched: `components/chat/ClearedFromBanner.tsx` (new),
    `lib/client/types.ts` (`createNewSession` return type widened to
    `Promise<string | null>`), `lib/client/use-session.ts` (`createNewSession`
    now returns the new session id), `app/[workspaceId]/page.tsx` (state,
    render-time latch, `/clear` slash handler, header `onClear`, `/rewind`
    handler, banner render).
  - Delivery shape: a yellow persistent banner (`border-amber-500/30
    bg-amber-500/10`) with a "Rewind ↺" button and an × dismiss button.
    Banner auto-suppresses the moment the session gains a user-role message.
  - Persistence: `sessionStorage["cleared:<newId>"] = oldId` — survives
    page refresh within the same tab; does not cross to new tabs.
  - Rejected alternative: rely on the existing sessions list + `/resume <id>`;
    rejected because the whole point of this feature is discoverability — the
    user who just did `/clear` doesn't know the old session id.

## Risks / follow-ups

- **CPU coalescing (100 ms delta batching) — deferred.** CC 2.1.191 reduced
  CPU usage ~37 % by batching `content_block_delta` events. Claudius calls
  `setMessages` on every delta today (no coalescing). This is a real
  optimisation opportunity, but it was excluded from this parity run because:
  (a) it has no product-visible surface (no screenshot, no command, no
  setting) and therefore cannot satisfy the run-notes "New UI surfaces"
  requirement, (b) the `message_stop` final-flush and `streaming:false`
  interaction requires careful testing that cannot be covered by the
  existing e2e suite, and (c) `use-session.ts` is the most critical file in
  the codebase — a subtle race there would be worse than leaving it
  un-optimised. Track separately; verify with a dedicated manual streaming
  test before landing.
- **`sessionStorage` scope.** The "cleared from" pointer is tab-scoped only.
  Opening the new session in a new tab will not show the banner. A future
  improvement could store `clearedFromSessionId` in the sessions DB via
  `mergeSessionState` and load it from a dedicated `GET /api/sessions/:id/state`
  endpoint on session bind. For this release, the tab-scoped behaviour is
  acceptable.
- **`/rewind` in a session that already has messages.** The banner and the
  `/rewind` handler both gate on "no user-role messages yet". Once the user
  sends their first message in the new session, the banner hides and `/rewind`
  reverts to the existing "use ↺ buttons" hint toast. This is intentional —
  rewinding within an ongoing conversation is handled by the existing per-bubble
  rewind controls, not by this session-level affordance.
