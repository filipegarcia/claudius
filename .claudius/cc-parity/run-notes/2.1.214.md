## Summary

Claude Code 2.1.212–2.1.214 is mostly bug fixes and engine-side hardening (Bash/permission-rule fixes, background-daemon lifecycle fixes, Windows/PowerShell fixes) that arrive automatically via the SDK updater. Two features are genuinely product-facing and don't flow through the SDK, so Claudius reimplements them this release: **session-wide WebSearch-call and subagent-spawn caps** (a runaway-loop safety net, extending the existing Cost → Limits panel with two new fields and a server-side `canUseTool` gate) and an **ISO "modified" timestamp on Claudius's own auto-memory frontmatter** (surfaced as a "Last modified" label in the Memory screen). Three other entries (`/fork`'s paired `/subtask` command, `claude auto-mode reset`, and the agent-view "Needs input" status) were judged ambiguous relative to Claudius's existing conventions and deliberately skipped rather than guessed at — see Risks. The headline risk to flag for review is the **default-disabled shape of the new tool-call caps**: they follow Claudius's existing "0/undefined disables" convention rather than upstream's default-on-at-200, which is the more conservative choice but diverges from Claude Code's out-of-the-box behavior — worth a second opinion on whether that's the right default for Claudius.

## Implemented (bucket B)

- `[extend: Limits panel (Cost screen)] Session-wide WebSearch / subagent-spawn caps` — added `maxWebSearches`/`maxSubagents` to `Limits` (`lib/server/limits-store.ts`), validated in `app/api/limits/route.ts`'s `PUT` handler, and surfaced as two new numeric fields in `components/cost/LimitsPanel.tsx` alongside the existing USD caps. Enforcement is server-side: `Session.canUseTool` (`lib/server/session.ts`) gates `WebSearch`/`Task` tool calls against a per-session counter and the configured cap, via a pure, unit-tested helper (`lib/shared/tool-budget.ts`). Rejected alternative: a brand-new "Session budgets" screen — overkill for two numeric fields that are conceptually the same "runaway-loop safety net" the existing USD caps already provide; extending `LimitsPanel` keeps all session-protection settings in one place.
- `[extend: Memory screen] Auto-memory frontmatter "modified" timestamp` — `lib/server/auto-memory.ts`'s `writeMemoryFile`/`patchMemoryFile` now stamp/refresh an ISO `modified:` line in the frontmatter on every write; `parseMemoryFrontmatter` reads it back. `app/[workspaceId]/memory/page.tsx` surfaces it as a "Last modified" label in both the auto-memory file list (from `fs.stat`'s `modifiedMs`, already computed but previously unused in the UI) and the edit form (from frontmatter, proving the stamp actually landed on disk). Rejected alternative: reusing this for the CLAUDE.md scope editor (`ClaudeMdSection` in the same page) too — that editor has no frontmatter concept at all (plain markdown), so injecting one would be a much larger, more surprising behavior change for a cosmetic timestamp; the auto-memory panel is the direct analogue of upstream's memory-file frontmatter and was the conservative, well-scoped target.

## Risks / follow-ups

- **WebSearch/subagent caps: opt-in (0/undefined disables) vs. upstream's default-on-at-200.** I chose the conservative, Claudius-consistent shape — matching the existing `projectDailyUsd`/`sessionUsd` "0/undefined disables" convention in `lib/server/limits-store.ts` — over silently enabling a cap of 200 for every existing session the first time this ships. The maximal/upstream-matching alternative (default-on at 200) would more closely mirror Claude Code's out-of-the-box safety net, but risks surprising an existing Claudius user with tool calls suddenly being denied after an unrelated deploy. Worth a follow-up discussion on whether Claudius should ship with the cap enabled by default.
- **The tool-budget gate only fires when the SDK actually invokes `canUseTool`.** Permission modes that skip the check for the tool in question (most notably `bypassPermissions`) also skip the cap. This is documented in the `LimitsPanel` copy and in code comments, but it means the cap is not a true hard limit in every mode — same caveat that already applies to nothing else in Claudius today, since this is the first server-enforced (not just client-soft) cap in the Limits feature.
- **`/subtask` (explicit in-session subagent spawn) — skipped, classification "ambiguous".** Upstream 2.1.212 split `/fork`'s old behavior into a new `/subtask` command. Claudius's existing `/fork` already matches the *new* `/fork` semantics (creates a session copy), but there's no Claudius equivalent of "manually invoke a subagent inline via a slash command" — today subagent spawning is entirely model-initiated (the `Task` tool). Building `/subtask` would mean designing a new manual-dispatch UX with no existing Claudius precedent to match against, which felt like scope creep for a single changelog line; flagging for a future dedicated review rather than guessing at the right shape here. See changelog entry: "`/fork` now copies your conversation into a new background session… the in-session subagent it used to launch is now `/subtask`".
- **`claude auto-mode reset` — skipped, classification "ambiguous".** Claudius's `disableAutoMode` is a single boolean escape hatch (`lib/server/settings.ts`), not the richer per-tool auto-mode allow-list config upstream's `auto-mode reset` resets. Mapping a "reset" action onto a boolean is either a no-op or requires inventing config Claudius doesn't have; deferred rather than building a speculative feature.
- **"Needs input" tab status — classified C (CLI/terminal only) rather than extended.** Claudius's `SessionTabs` status model (`running`/`idle`/`starting`/`error`/`background`) already signals roughly the same thing background sessions need — a case could be made for adding a dedicated "needs input" status when a tab is blocked on a permission/plan/ask-question prompt. Conservative call was that the existing pending-permission badge already covers this need in Claudius's UI idiom, so a literal port of the CLI's status label would be redundant chrome. Flagging in case a reviewer disagrees.
- No secrets, migrations, or destructive operations touched by this release's bucket-B work.
