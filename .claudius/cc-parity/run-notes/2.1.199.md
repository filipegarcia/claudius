## Summary

Claude Code 2.1.199 is overwhelmingly a bug-fix release: of the ~24
changelog entries, all but five are pure reliability/regression fixes to
the agent engine, the background-agent daemon, or CLI-only chrome. Four of
the non-bug-fix entries — stacked slash-skill loading (up to 5), automatic
retry-with-backoff for transient 429s, a raised default on the
`CLAUDE_CODE_RETRY_WATCHDOG`/`CLAUDE_CODE_MAX_RETRIES` env-var pair, and a
`claude agents` PR-link display tweak — are engine-side behavior exposed
via the SDK or bare CLI/env-var chrome with no Claudius surface. One entry
that reads like a pure bug fix turned out, on inspection of the actual SDK
type (`SDKHookResponseMessage` in the already-bumped
`@anthropic-ai/claude-agent-sdk@0.3.199`), to be a genuine Claudius gap:
"`SessionStart`/`Setup`/`SubagentStart` hooks silently hiding stderr on
exit code 2" — the SDK's `hook_response` system message has always carried
a `stderr` field, but Claudius's client reducer only ever read
`hook_name`/`exit_code`/`outcome` off it and silently dropped `stderr` on
the floor. **This is the release's one bucket-B item**, shipped this run:
failed hook pills now render an error tone and an expandable stderr body.
Three further entries read as bug fixes and touch a UI affordance closely
enough to warrant an explicit call rather than silent elision: idle
subagents collapsing into a summary row, a notice when `/model`/`/fast` is
typed while viewing a subagent, and `/color` surviving session
backgrounding. All three were checked against the actual Claudius
components (`components/chat/TaskBlock.tsx`,
`components/panels/widgets/ModelPicker.tsx`,
`components/chat/ChatSurface.tsx`/`PromptInput.tsx`) and confirmed to have
no matching gap: Claudius has no consolidated live "agent panel" with an
idle subagent state to collapse (each `Task` call is already its own
independent, collapsed-by-default transcript row, statuses limited to
`pending | running | completed | failed | killed | stopped` — no `idle`),
no "viewing a subagent" navigational mode for `/model` to misfire in, and
no tab/session list that renders the per-session `/color` accent outside
the active composer border. The headline risk to flag for review: the
hook-stderr fix was caught only by reading the installed SDK's `.d.ts`
directly rather than trusting the changelog's "Fixed ... silently hiding
stderr" phrasing (which reads CLI-only at a skim) — a reviewer should
double-check the shape of the fix (inline expandable pill vs. a dedicated
hooks log) against Claudius's conventions, and sanity-check the three
remaining "checked, no gap" calls below.

## Changelog classification

- `[A — already via SDK updater] Stacked slash-skill invocations like /skill-a /skill-b do XYZ now load all leading skills (up to 5), not just the first` — verified directly: every `category: "skill"` entry in `lib/shared/slash-commands.ts` (`batch`, `simplify`, `review`, `security-review`, `ultraplan`, `ultrareview`, `autofix-pr`, …) is `handler: "sdk"`, and `ChatSurface.tsx`'s `handleSend` only peels off the leading `/word` token to look up a command for the *native*/toast paths — it never expands or limits multiple leading skill invocations itself. The full prompt text (including any stacked `/skill-a /skill-b …`) is forwarded to the SDK's `query()` verbatim; skill discovery/loading/stacking is entirely the SDK's own parsing, so the 5-skill cap raise lands the moment the SDK updater bumps `@anthropic-ai/claude-agent-sdk`. No Claudius-side parser exists to touch.
- `[A — already via SDK updater] Transient server rate-limit errors (429s unrelated to your usage limit) are now retried automatically with backoff for subscribers instead of failing the turn` — pure engine retry-loop behavior; ties into the same `SDKAPIRetryMessage`/`system`/`api_retry` message Claudius already renders in the "Claude is working…" spinner (shipped for 2.1.198, `lib/client/api-retry.ts` + `SpinnerTip.tsx`). No new UI needed — more retries just means more/longer-lived spinner-tip messages through the existing pipe.
- `[A — already via SDK updater, env knob is C] CLAUDE_CODE_RETRY_WATCHDOG now raises the default retry count for non-capacity transient errors to 300 and lifts the cap of 15 on CLAUDE_CODE_MAX_RETRIES` — the *behavior change* (more retry attempts before giving up) is engine-side and arrives with the SDK bump; Claudius never caps or reads these two values itself (confirmed: `maxRetries` in `lib/client/api-retry.ts`/`use-session.ts` is a pure display field sourced from the SDK's `max_retries` on the retry message, not a Claudius setting). The env vars themselves are bare process-level CLI knobs — no Claudius settings UI is warranted for a watchdog escape hatch; a power user can already set the env var on the server process if needed.
- `[C — CLI/terminal only] claude agents session rows now show pull-request links as bare #N without the redundant "PR" label` — cosmetic tweak to the `claude agents` background-dispatch TUI. Per the 2.1.198 precedent, Claudius sessions are directly chat-driven, not dispatched into a detached background queue the user reattaches to later — there is no Claudius screen listing session rows with PR links to begin with.
- `[C — no Claudius analog, checked against actual components] Fixed idle subagents vanishing from the agent panel while other subagents were still working; surplus idle agents now collapse into an expandable summary row` — this fixes the CLI's own live multi-subagent TUI panel, which tracks a per-subagent `idle` status. Claudius has no equivalent: `components/chat/TaskBlock.tsx` renders each `Task` tool call as its own independent, collapsed-by-default transcript row (`pending | running | completed | failed | killed | stopped` — no `idle` state exists), and `components/chat/StatusLine.tsx`'s working/idle badge is session-level (the main thread), not per-subagent. Claudius already sidesteps the clutter problem the CLI is patching here (surplus rows) via a different, pre-existing mechanism (default-collapsed independent blocks), so there's no bug to reproduce or gap to fill.
- `[C — no Claudius analog] Fixed typing /model or /fast while viewing a subagent silently opening the lead's model picker; a notice now explains the command applies to the lead` — Claudius's `ModelPicker` (`components/panels/widgets/ModelPicker.tsx`) only ever operates on the current session's model; there is no "viewing a subagent" navigational view to switch into in the first place (subagent model is only set statically per-agent-definition frontmatter in `app/[workspaceId]/agents/page.tsx`, never live-picked). Nothing in Claudius can misfire this way.
- `[C — no Claudius analog] Fixed backgrounding a session with ← or /background dropping its /color from the agent view row` — Claudius does have `/color` (`lib/shared/slash-commands.ts` id `color`, `handler: "native"`), but it is rendered only as the active tab's composer-border accent (`components/chat/PromptInput.tsx`) — no tab strip, session list, or "agent view row" in Claudius currently surfaces that color for a backgrounded/inactive session, so there is nothing for a background action to drop. Building that display first, purely to backport a fix for a bug that would only exist once it's built, would be manufacturing scope the changelog entry doesn't ask for — see Risks.
- `[B — reimplement in Claudius] Fixed SessionStart, Setup, and SubagentStart hooks silently hiding stderr when exiting with code 2 — the error is now shown in the transcript` — this reads as a pure CLI bug fix ("shown in the transcript" = the terminal's own transcript), but `lib/server/hooks.ts` turned out to be settings CRUD only (list/add/remove hook groups in `settings.json`) — it does not execute hooks or capture output, so it can't be "already forwarding" anything as the first-pass draft assumed. Hook execution is entirely the SDK's; Claudius consumes it as a `hook_response` system message. Reading the installed SDK's type directly (`node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts:3657`, `SDKHookResponseMessage`) confirms the message has always carried a `stderr: string` field, and Claudius's client reducer (`lib/client/use-session.ts`, the `subtype === "hook_response"` branch) only ever destructured `hook_name`/`exit_code`/`outcome` off it — `stderr` was silently dropped on the floor regardless of what CC 2.1.199 changed upstream. That's a genuine Claudius-side gap (a field the SDK provides that Claudius's UI never rendered), not something a `@anthropic-ai/claude-agent-sdk` version bump fixes by itself, since the SDK type/message shape doesn't change here — only Claudius's *handling* of a field it already had access to needed to change. Implemented this run — see "## Implemented (bucket B)".
- Pure bug-fix entries elided (all engine/daemon/CLI-chrome, no product-surface gap): SSL certificate error guidance; streaming responses retained after a mid-stream overloaded error; subagents returning partial work after a rate limit/server error; subagents no longer reporting API errors as success; Linux background-agent daemon self-kill loop; SSH cold-start failure on macOS (2.1.196 regression); `claude stop` racing a background-agent respawn; background job progress-indicator stalls; low-memory background-session messaging; remote-session Working/Idle flapping; `claude --dangerously-skip-permissions daemon <subcommand>` CLI arg parsing; `SendMessage` re-spawned-agent-name misrouting detection (agent-orchestration internals); sessions with no new messages no longer growing the transcript file on open/resume (SDK/session-storage internals); corrupted config file now backed up before the startup recovery dialog resets it (CLI-only recovery flow); Claude in Chrome reconnect-page loop (separate browser-extension product, not Claude Code/the SDK); plan mode now correctly auto-allowing read-only `browser_batch` calls while still prompting for state-changing ones (permission-engine logic for the Chrome tool, exposed via the SDK's permission system).

## Risks / follow-ups

- **Hook-stderr pill shape (the one bucket-B item shipped)** — considered
  and rejected a dedicated "Hooks" log surface (new panel/tab under the
  right rail, listing every hook invocation with timing + exit code +
  stdout/stderr) instead of extending the existing inline
  `hook_started`/`hook_response` pills. Rejected because today's hook
  pills are low-frequency, low-noise, and already inline in the
  transcript where the failure is contextually relevant (right after the
  turn that triggered the hook) — a separate panel would split that
  context across two places for a single new field. If hook usage grows
  (more `PreToolUse`/`PostToolUse` power users, longer stdout/stderr
  bodies routinely truncated by the `max-h-60 overflow-auto` clamp),
  revisit a dedicated panel then.
- **Scope conservatism on the stderr surfacing condition** — the pill
  only shows `stderr` when `outcome` is `"error"` or `"cancelled"`. The
  changelog entry specifically calls out exit-code-2 (a blocking
  failure); the SDK's `hook_response` type doesn't expose a bare `exit_code
  === 2` check independent of `outcome` (both fields are present but
  `outcome` is the SDK's own success/fail classification), so gating on
  `outcome` is the closest available proxy and, in practice, exit-2 hooks
  should always set `outcome: "error"`. Maximal alternative: also surface
  `stdout` (not just `stderr`) whenever it's non-empty, even on success —
  rejected as out of scope, since the changelog entry is specifically
  about stderr being *hidden*, not about stdout visibility in general.
- **Three "checked, no gap" C classifications** — idle-subagent collapse,
  `/model`/`/fast`-while-viewing-a-subagent notice, and `/color` surviving
  backgrounding — were each verified this run by grepping the actual
  component (`components/chat/TaskBlock.tsx`'s status union has no
  `idle` state; `promptColor`/`activePromptColor` usage in
  `ChatSurface.tsx`/`PromptInput.tsx` is confined to the active
  composer's border with no tab-strip/session-list consumer). A reviewer
  who disagrees that "the underlying Claudius concept doesn't exist yet"
  is the right read (vs. "build the missing concept now, then land the
  fix") should treat this as the maximal alternative deferred, not
  rejected outright — none of the three are hard blockers on anything
  else in this release.
- **Combined-mode branch note** — this run executed on `sdk-update/0.3.199`
  (the SDK-migration branch for the same version pair), per the
  "Combined-mode run" instruction at the top of the prompt, not on a
  fresh `cc-parity/2.1.199` branch off `origin/main` as the generic
  Step-0 boilerplate describes. Confirmed via `git log
  origin/main..HEAD` (only the SDK-bump + SDK-notes commits were ahead)
  and precedent (`f00f0c0` merged `sdk-update/0.3.198` with the 2.1.198
  cc-parity commits stacked on top of the same branch) before proceeding
  — flagging here so a reviewer isn't surprised the PR isn't off a
  `cc-parity/2.1.199` branch.
- None identified beyond the above — no changelog entry was left in
