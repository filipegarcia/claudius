## Summary

Claude Code 2.1.208 is a 45-bullet release: 4 new settings/env-var
affordances plus a long tail of CLI/TUI bug fixes and perf work. On this
combined-mode branch, the SDK bump (`0.3.207` → `0.3.208`) already landed
(commits `273d779`, `c7aa676`) and its run-notes
(`.claudius/sdk-updater/run-notes/0.3.208.md`) concluded "zero Claudius code
changes required" for the public API-surface diff — that's a call-site-
breakage verdict, not a product-surface one, so this pass did its own
audit rather than inheriting that conclusion. One genuine bucket-B item
came out of it: the CLI's `/usage` command now shows last-known bars with
an "as of" note when the usage endpoint is rate-limited instead of an
error screen. Claudius's analog — the "Plan" section of the "Session cost
& usage" overlay, fed by the experimental
`usage_EXPERIMENTAL_MAY_CHANGE_DO_NOT_RELY_ON_THIS_API_YET()` SDK call —
already degraded gracefully (a failed/rate-limited fetch silently keeps
the last-known state, no error screen), but had no freshness cue, so
stale utilization bars could read as live. This release adds a new
`plan_usage_unavailable` event (broadcast from the same server-side catch
branch that previously swallowed the error silently) that explicitly
flags the last-known `plan_usage` data as stale, plus an "as of &lt;time&gt;"
note and dimmed bars in `CostOverlay.tsx` once that flag is set. The first
implementation cut inferred staleness from elapsed time instead (a
`fetchedAt` timestamp aged past a 5-minute threshold); an adversarial
UX-fit review caught that this misfires on healthy long-running agentic
turns (routine in Claude Code, easily &gt;5 minutes) and introduced a
client/server clock-skew bug besides, so it was replaced with the explicit
event-driven signal described above before landing. Everything else in
the changelog is either CLI/TUI-only (screen-reader mode, vim insert-mode
remaps, mouse-click menu support) or transparently inherited through the
typed SDK/CLI surface with no Claudius call site. Headline risk to flag
for review: staleness can only be detected at per-turn boundaries (there's
no on-demand refetch the way the CLI's own `/usage` re-queries on open) —
see Risks below.

## Implemented (bucket B)

- `[extend: Session cost & usage overlay]` Added an "as of &lt;time&gt;"
  staleness note (with dimmed utilization bars) to the "Plan" section of
  `CostOverlay.tsx`, mirroring the CLI's `/usage` graceful-degradation
  behavior. Files touched:
  - `lib/shared/events.ts` — `PlanUsageEvent.fetchedAt` (epoch ms, stamped
    on every successful fetch) and a new `PlanUsageUnavailableEvent`
    (broadcast when a fetch attempt fails), added to the `ServerEvent`
    union.
  - `lib/server/session.ts` (~line 6816-6875) — stamp `fetchedAt: Date.now()`
    into the success-path event; broadcast `PlanUsageUnavailableEvent` from
    the `catch` branch that previously returned silently.
  - `lib/client/types.ts` — `PlanRateLimits.fetchedAt` and `.stale`.
  - `lib/client/use-session.ts` (~line 1745-1762) — set `stale: false` on a
    successful `plan_usage`; on `plan_usage_unavailable`, flip `stale: true`
    on whatever `planUsage` state already exists (no-op if none yet).
  - `components/overlays/CostOverlay.tsx` — `isStale = !!planUsage?.stale`
    (no client-side time math), "as of" note, `opacity-60` on the bars
    container when stale (matching `TodosBanner.tsx`'s existing stale-dim
    convention).
  Rejected alternative #1: a new dedicated `/usage`-style screen/route —
  overkill, this is a freshness cue on an existing overlay section, not a
  new domain; extending `CostOverlay.tsx` (which already renders this exact
  data) is the obviously right shape. Rejected alternative #2 (first
  implementation cut, reverted after review): infer staleness client-side
  from `Date.now() - fetchedAt > 5 minutes` instead of an explicit server
  event — simpler, but wrong: adversarial review found it false-flags any
  session with a turn that simply runs longer than 5 minutes (routine for
  Claude Code), and it compared a server-stamped timestamp against the
  client's own clock with no skew handling. The explicit-event design has
  no equivalent failure mode — staleness is a plain boolean, true only when
  a fetch actually failed.

## Risks / follow-ups

- **Bucket-B design call — resolved via adversarial review, documented for
  visibility:** the first cut inferred staleness client-side from elapsed
  time since the last successful `fetchedAt` (a 5-minute threshold). An
  adversarial UX-fit pass caught two real bugs in that design: (1) it
  false-flags any session where a single turn simply runs past 5
  minutes — normal for Claude Code's multi-step agentic turns — since
  `fetchedAt` only advances at turn end and opening the overlay doesn't
  trigger a refetch the way the CLI's own `/usage` does; (2) it compared a
  server-stamped epoch against the client's own `Date.now()` with no skew
  handling, which breaks on Claudius's remote-browser deployment mode.
  Replaced with an explicit `plan_usage_unavailable` event broadcast from
  the server's own catch branch, so staleness is a plain boolean fact from
  the one place that actually knows the fetch failed, not a client-side
  guess. Documented here because the "infer from elapsed time" shape is a
  plausible first instinct a future reviewer might reach for again — it
  doesn't work for this data source.
- **Known limitation, not fixed:** staleness can only be detected at the
  next turn's fetch attempt — Claudius has no equivalent of the CLI's
  on-demand `/usage` re-query when the overlay opens. A user who opens the
  overlay mid-turn sees whatever the last attempt reported, stale or not,
  until the current turn ends. Considered adding an on-open refetch;
  rejected as scope creep beyond what the changelog entry asked for (the
  CLI's own fix is about `/usage`'s *display* behavior on a rate-limited
  response, not about a new refetch trigger), and there's no low-risk way
  to add ad-hoc calls to an experimental, undocumented SDK method
  (`usage_EXPERIMENTAL_MAY_CHANGE_DO_NOT_RELY_ON_THIS_API_YET`) outside its
  one existing per-turn call site without auditing rate-limit implications
  on that endpoint itself.
- **Screen-reader mode (classified C):** the audit found a real,
  pre-existing accessibility gap unrelated to this specific changelog item
  — the chat transcript (`MessageList.tsx`) has no `aria-live`/`role="log"`,
  so streaming assistant text isn't announced to screen-reader users as it
  arrives. This wasn't built here (porting the CLI's "opt-in plain-text
  mode" into a browser app would be the wrong shape — that failure mode is
  a terminal-rendering problem the DOM doesn't have), but it's worth a
  dedicated accessibility follow-up independent of CC parity.
- **`vimInsertModeRemaps`/`editorMode` (classified C):** conservative
  reading is "no Claudius composer analog, skip entirely" (what was done).
  The maximal reading would be exposing `editorMode`/`vimInsertModeRemaps`
  in the settings catalog's generic "Other" JSON editor for round-trip
  parity with the CLI's own config file, even though it does nothing in
  Claudius's browser composer — rejected as pure clutter (a setting with
  provably zero effect shouldn't get catalog billing), consistent with how
  the SDK-updater run-notes already reasoned about `editorMode` for the
  same release.
- **`CLAUDE_CODE_PROCESS_WRAPPER` classification:** tagged
  `[A — already via SDK updater]` rather than `[C]` because it's genuinely
  transparent engine/env-passthrough behavior, not a CLI-terminal-only
  affordance — but reasonable minds could tag this `[skip — ambiguous]`
  instead since "already via SDK updater" implies the SDK bump did
  something, when really *nothing* needed to happen on either side. Noted
  here so a reviewer can re-tag if the distinction matters for their
  purposes.
- No other bucket-B candidates were identified after auditing the full
  45-bullet changelog against Claudius's existing surfaces (composer, cost
  overlay, MCP page, settings catalog, transcript rendering) — the escape
  hatch ("no bucket-B items") was seriously considered and rejected only
  because the `/usage` staleness item cleared the bar on inspection, not
  assumed from the SDK run-notes' "zero code changes" conclusion (a
  different question, see Summary).
