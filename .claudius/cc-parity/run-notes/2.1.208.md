## Summary

Claude Code 2.1.208 is a 45-bullet release: 4 new settings/env-var
affordances plus a long tail of CLI/TUI bug fixes and perf work. On this
combined-mode branch, the SDK bump (`0.3.207` → `0.3.208`) already landed
(commits `273d779`, `c7aa676`) and its run-notes
(`.claudius/sdk-updater/run-notes/0.3.208.md`) concluded "zero Claudius code
changes required" for the public API-surface diff — that's a call-site-
breakage verdict, not a product-surface one, so this pass did its own
audit rather than inheriting that conclusion. One genuine bucket-B item
came out of it: the CLI's `/usage` command now shows last-known bars with
an "as of" note when the usage endpoint is rate-limited instead of an
error screen. Claudius's analog — the "Plan" section of the "Session cost
& usage" overlay, fed by the experimental
`usage_EXPERIMENTAL_MAY_CHANGE_DO_NOT_RELY_ON_THIS_API_YET()` SDK call —
already degraded gracefully (a failed/rate-limited fetch silently keeps
the last-known state, no error screen), but had no freshness cue, so
stale utilization bars could read as live. This release adds a
`fetchedAt` timestamp to the `plan_usage` event and an "as of &lt;time&gt;"
note in `CostOverlay.tsx` once that timestamp ages past 5 minutes.
Everything else in the changelog is either CLI/TUI-only (screen-reader
mode, vim insert-mode remaps, mouse-click menu support) or transparently
inherited through the typed SDK/CLI surface with no Claudius call site.
Headline risk to flag for review: the 5-minute staleness threshold is a
judgment call (there's no explicit "this fetch was rate-limited" signal
to key off, only the growing age of the last successful fetch) — see
Risks below.

## Risks / follow-ups

- **Bucket-B design call:** the 5-minute staleness threshold
  (`PLAN_USAGE_STALE_AFTER_MS` in `CostOverlay.tsx`) is a judgment call, not
  a value derived from the CLI. Claudius has no explicit "this fetch was
  rate-limited" signal — the server-side catch swallows the error
  indiscriminately (network hiccup, actual rate-limit, or the experimental
  API changing shape) — so staleness is inferred purely from how old the
  last successful `fetchedAt` has grown. Alternative considered and
  rejected: always show "as of &lt;time&gt;" regardless of age. Rejected
  because it adds visual noise to the common case (data refreshes every
  successful turn) for no benefit — the note is only useful when it
  signals something has gone stale, which is what CC's own "when rate-
  limited" gating conveys. If a reviewer wants a different threshold or an
  explicit "degraded" flag instead of an inferred one, that's a one-line
  change in `CostOverlay.tsx` / a small addition to the `session.ts` catch
  branch.
- **Screen-reader mode (classified C):** the audit found a real,
  pre-existing accessibility gap unrelated to this specific changelog item
  — the chat transcript (`MessageList.tsx`) has no `aria-live`/`role="log"`,
  so streaming assistant text isn't announced to screen-reader users as it
  arrives. This wasn't built here (porting the CLI's "opt-in plain-text
  mode" into a browser app would be the wrong shape — that failure mode is
  a terminal-rendering problem the DOM doesn't have), but it's worth a
  dedicated accessibility follow-up independent of CC parity.
- **`vimInsertModeRemaps`/`editorMode` (classified C):** conservative
  reading is "no Claudius composer analog, skip entirely" (what was done).
  The maximal reading would be exposing `editorMode`/`vimInsertModeRemaps`
  in the settings catalog's generic "Other" JSON editor for round-trip
  parity with the CLI's own config file, even though it does nothing in
  Claudius's browser composer — rejected as pure clutter (a setting with
  provably zero effect shouldn't get catalog billing), consistent with how
  the SDK-updater run-notes already reasoned about `editorMode` for the
  same release.
- **`CLAUDE_CODE_PROCESS_WRAPPER` classification:** tagged
  `[A — already via SDK updater]` rather than `[C]` because it's genuinely
  transparent engine/env-passthrough behavior, not a CLI-terminal-only
  affordance — but reasonable minds could tag this `[skip — ambiguous]`
  instead since "already via SDK updater" implies the SDK bump did
  something, when really *nothing* needed to happen on either side. Noted
  here so a reviewer can re-tag if the distinction matters for their
  purposes.
- No other bucket-B candidates were identified after auditing the full
  45-bullet changelog against Claudius's existing surfaces (composer, cost
  overlay, MCP page, settings catalog, transcript rendering) — the escape
  hatch ("no bucket-B items") was seriously considered and rejected only
  because the `/usage` staleness item cleared the bar on inspection, not
  assumed from the SDK run-notes' "zero code changes" conclusion (a
  different question, see Summary).
