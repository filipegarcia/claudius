## Summary

Claude Code 2.1.201–2.1.204 is mostly a bug-fix cycle for the CLI's
background-agent daemon and `claude agents` TUI (worktree isolation, PATH/env
inheritance, crash-loop and stale-token recovery) plus a handful of
engine-side behavior tweaks that land automatically once the SDK updater
bumps `@anthropic-ai/claude-agent-sdk` (MCP `roots/list` for additional
working directories, workflow OpenTelemetry attributes, reduced subagent
re-delegation, no mid-conversation system role for Sonnet 5). Exactly one
feature is genuinely product-surface and outside the SDK boundary: 2.1.203's
proactive "your login is about to expire" warning, so background sessions
don't get silently interrupted by a lapsed credential. Claudius reimplements
it end-to-end for its own account-switcher (`lib/server/accounts-store.ts` /
`lib/server/session.ts` / a new `TokenExpiringPanel` banner). The manual
permission-mode visibility fix (2.1.203's grey ⏸ badge) and the "claude
agents" empty-state section fix are both classified bucket B but ship no new
code — Claudius's `ModeSelector` and `BackgroundTasksPanel` already satisfy
the underlying UX intent (see classification below). 2.1.202's "Dynamic
workflow size" `/config` setting is classified `[skip — no
Claudius-reachable surface]`: its intent is clear (advisory
small/medium/large fan-out guidance), but that guidance lives entirely
inside the Workflow tool's own SDK-provided description text — confirmed
by grepping the repo for that prompt's distinctive phrasing and finding
zero hits — and `sdk.d.ts` @ 0.3.204 exposes no `Options` field to
parameterize it, so there is no Claudius-owned surface to attach a
setting to today (see classification + Risks for the reasoning and the
follow-up path). Headline risk for review: confirm the login-expiry warning's 24-hour
threshold and its practical reach — Claudius's own OAuth flow deliberately
mints long-lived tokens that rarely report an `expiresAt` at all, so the
banner is correct-but-narrow (see Risks).

## Implemented (bucket B)

- `[extend: chat banner surface] Proactive "your login is about to expire"
  warning` — mirrors the existing `AuthFailedPanel` (reactive 401 banner)
  with a new, amber-toned sibling for the *proactive* case.
  `lib/server/accounts-store.ts` now persists an optional
  `AccountProfile.expiresAt` (previously dropped even though the OAuth
  exchange already returns it); `app/api/accounts/oauth/route.ts` passes it
  through on profile creation. `lib/server/token-expiry.ts` is a new pure
  helper (`shouldWarnTokenExpiring`, `TOKEN_EXPIRY_WARNING_WINDOW_MS = 24h`).
  `lib/server/session.ts` captures the active profile's `expiresAt` at
  construction and fires a one-shot `token_expiring_required` SSE event on
  the first live `system:init` (same timing/fire-once shape as
  `noteMcpNeedsAuthAtStartup`). `lib/shared/events.ts` adds the event type;
  `lib/client/use-session.ts` / `lib/client/types.ts` add the state +
  dismiss handler; the new `components/chat/TokenExpiringPanel.tsx` renders
  the dismissible banner (CTA → `/usage#accounts`, reusing
  `AuthFailedPanel`'s exported `ACCOUNTS_URL`); wired into
  `components/chat/ChatSurface.tsx` next to `AuthFailedPanel`. Rejected
  alternative: a passive `kind: "info"` transcript pill (the shape used for
  the 2.1.193 MCP needs-auth notice) — rejected because this nudge needs a
  one-click remediation CTA and a clearly higher urgency tone than an
  informational pill, which the existing banner family (`AuthFailedPanel`,
  `FastModeNoticePanel`) already expresses via color + dismiss + link.

## Risks / follow-ups

- **Narrow practical reach:** Claudius's own OAuth flow
  (`lib/server/account-oauth.ts`) deliberately requests the `inferenceOnly`
  scope specifically to mint a long-lived, `setup-token`-style credential —
  by design, most profiles will never populate `expiresAt` at all, so the
  new banner may rarely fire in practice. It's still the correct shape
  (some exchange responses, org policies, or future token types may report
  a real expiry, and the reactive `AuthFailedPanel` sibling already proves
  Claudius doesn't auto-refresh), but a reviewer should confirm this
  narrowness is acceptable rather than expecting parity with however often
  the CLI's own warning fires.
- **24-hour threshold is a documented assumption, not a confirmed CC
  value:** neither the changelog nor the docs state the CLI's own warning
  window. `TOKEN_EXPIRY_WARNING_WINDOW_MS` in `lib/server/token-expiry.ts`
  picks 24h (comfortably covers an overnight-idle background session
  without nagging at every session start for a token that's merely "not
  indefinite"). A reviewer with access to the actual CC source/behavior
  should confirm or adjust.
- **Rejected alternative for the banner shape:** a passive transcript info
  pill (2.1.193's MCP needs-auth notice shape) was considered and rejected
  in favor of a dedicated dismissible banner with a CTA — see "Implemented"
  above for the reasoning.
- **Rejected reclassification — "/workflows agent list layout" (2.1.202):**
  considered promoting this from `[C]` to `[B — extend:
  BackgroundTasksPanel]` because Claudius's "Tasks" list already has an
  analogous title/time/tool-count row shape and upstream's change is
  literally the inverse of what Claudius shows today (drop the tool-call
  count). Rejected because the upstream entry is column-width/label tuning
  for a fixed-width terminal grid — a constraint a responsive CSS list
  doesn't share — so importing it verbatim ("shorter model names", "no
  tool-call counts") would be copying a CLI-flavoured cosmetic tweak rather
  than delivering a browser-idiomatic affordance. Also note upstream's
  `/workflows` view shows one row per *individual agent* inside a workflow
  run; Claudius's `BackgroundTasksPanel` "Tasks" list is one row per
  workflow *task* overall (`components/chat/WorkflowBlock.tsx` explicitly
  aggregates because "the SDK does not expose the per-agent breakdown"),
  so even a faithful port would need a data source Claudius doesn't have
  yet. A reviewer who wants deeper per-agent workflow visibility should
  treat that as its own future feature, not a port of this cosmetic
  changelog entry.
- **`[skip — no Claudius-reachable surface]` item:** "Dynamic workflow
  size" `/config` setting (2.1.202). On reflection this is not actually
  ambiguous in *intent* — it's an advisory small/medium/large nudge to how
  large the model tends to make a dynamic workflow's agent fan-out. The
  blocker is structural: that guidance lives entirely inside the Workflow
  tool's own SDK-provided description text (confirmed via
  `grep -rl "DEFAULT TO pipeline\|Cap fan-out\|a handful of agents per
  phase"` across the repo — zero hits, meaning Claudius doesn't author or
  template any part of that prompt), and `sdk.d.ts` @ 0.3.204 exposes no
  `Options` field to parameterize it either. Claudius has a
  `systemPromptAppend` hook for the *general* agent system prompt
  (`lib/server/session.ts`) but nothing equivalent scoped to an individual
  built-in tool's description. **If a future SDK version adds an
  `Options.workflowSize` (or similar) passthrough**, this flips to bucket
  A and arrives automatically via the SDK updater. **If it stays
  CLI-config-only with no SDK passthrough**, the only way to build it in
  Claudius would be prompt-injecting size guidance ourselves via
  `systemPromptAppend` (imprecise — it steers the general system prompt,
  not the Workflow tool description specifically, so it would compete
  with the SDK's own "few vs. comprehensive" guidance rather than cleanly
  override it) — a reviewer who wants this sooner should treat that as
  its own follow-up, not a faithful port of the CC setting.
- **Manual-mode badge and empty-agents-view items ship no code:** both are
  classified bucket B (real product-surface intent) but audited as already
  satisfied by Claudius's existing `ModeSelector` and `BackgroundTasksPanel`
  respectively — see classification above for the specifics a reviewer
  should double-check.
